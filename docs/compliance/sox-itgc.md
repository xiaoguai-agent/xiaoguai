# SOX § 404 IT General Controls Mapping — Xiaoguai Wave-3

Last updated: 2026-05-25
Scope: Xiaoguai platform as of wave-3 (v1.3.x-prep / main @ 9970aa0)
Basis: Sarbanes-Oxley Act § 404; COBIT 2019 framework; AICPA Trust Services Criteria (TSP 2017).
Status legend: ✅ shipped · 🚧 partial · 🛣 not yet done (gap)

---

## Executive Summary

**Xiaoguai is not itself in scope for a SOX § 404 audit.** SOX § 404 scoping is an organisational
decision made by the public company's external auditors. However, when an operator deploys the
**finance-fpa pack** (or any pack that touches financial-reporting workflows — period-close data,
variance analysis, board-deck drafting), Xiaoguai becomes a component of the company's **IT
environment supporting financial reporting** and therefore subject to IT General Controls (ITGC)
review.

**What activates SOX relevance:**
- `packs/finance-fpa/` processes live period-close data from NetSuite / Anaplan and produces
  board-ready variance narratives routed to the CFO.
- `packs/ar-collections/` automates dunning and DSO drift monitoring over accounts-receivable data.
- Any skill pack writing to or reading from ERP-sourced financial tables falls within the ITGC
  perimeter once the external auditor determines the system is relied upon for financial reporting.

**What Xiaoguai technically provides across the five COBIT ITGC categories:**
- Tamper-detectable audit log (HMAC chain) recording every actor action with timestamp, tenant, and result.
- HotL policy enforcer enforcing human-in-the-loop gates and segregation of preparer vs. approver.
- Migration-tracked schema changes with named, sequenced SQL files.
- GitHub CI/CD pipeline (`release.yml`, `audit.yml`, `deny.yml`, `snyk.yml`) as change-deployment controls.
- Anomaly detector + Grafana dashboards for ongoing operational monitoring.
- SAST (`cargo audit`, `cargo deny`, `snyk.yml`) + SBOM attestation (cosign) for SDLC controls.

**Honest gaps (5, SOX-specific):**
1. No separation-of-duties enforcement at the *human workflow* layer — HotL gates AI-agent actions,
   not human-to-human financial-approval workflows. Finance-fpa approval requires a named human
   approver tier; operators must configure this for effective SoD.
2. No automated change-ticket integration — ADRs and migration files serve as change records, but
   no system links them to a change-management tool (ServiceNow, Jira).
3. `installed_skill_packs` captures pack install but not pack *uninstall* or *version downgrade*;
   audit trail for rollbacks is incomplete.
4. No privileged-access review (PAR) report generated automatically — role assignments via Casbin
   can be queried but no quarterly PAR report is generated by the platform.
5. Audit log archival strategy undefined (gap G-011 in `docs/compliance/compliance-gaps.md`) —
   operators must configure WAL-based export to long-term storage to meet seven-year SOX retention.

This document maps controls; it does not constitute legal or accounting advice. All scoping
decisions remain with the operator's external auditors.

## ITGC Category 1 — Change Management (CC1)

*COBIT domain: BAI06 (Managed IT Changes). AICPA TSP: CC8.1.*
*Controls over moving code, schema, and configuration from development to production.*

| ITGC Ref | Description | Xiaoguai feature | Status | Audit-evidence path |
|----------|-------------|-----------------|:------:|---------------------|
| CC1.1 | Formal change request and approval before promoting to production | GitHub PR review gate; branch-protection rules require at least one reviewer approval before merge to `main` | ✅ | GitHub PR history; branch protection settings in repo settings; `git log --merges` |
| CC1.2 | Change authorisation documented prior to deployment | ADRs (`docs/architecture/adr/`) provide design-level approval records; PR description links to ADR for significant changes | ✅ | `docs/architecture/adr/` file listing; GitHub PR descriptions |
| CC1.3 | Separation of dev, test, and production environments | GitHub Actions `release.yml` deploys only from tagged releases; Helm chart values-ha.yaml and compose files are env-specific | ✅ | `.github/workflows/release.yml`; `deploy/helm/xiaoguai/values-ha.yaml` |
| CC1.4 | Schema / database changes controlled and tracked | Migrations numbered sequentially (`0001_initial.sql` through `0015_skill_packs.sql`) with named files; applied atomically at startup | ✅ | `crates/xiaoguai-storage/migrations/` directory; migration apply log in application startup output |
| CC1.5 | Config-as-code prevents uncontrolled configuration drift | Helm chart values files; Kustomize overlays; `deploy/config.example.yaml` are version-controlled; no manual config changes outside SCM | ✅ | `deploy/helm/xiaoguai/`; git log on config files |
| CC1.6 | Skill-pack installs recorded as application-layer change events | `installed_skill_packs` table records `pack_slug`, `version`, `installed_at`, `tenant_id`; UNIQUE constraint prevents silent re-install | ✅ | `SELECT * FROM installed_skill_packs ORDER BY installed_at;` — provides change log for pack deployments |
| CC1.7 | Emergency change controls (hotfix path) | Tagged hotfix releases via `release.yml` with mandatory review; no unreviewed push to main (branch protection) | 🚧 | GitHub branch protection rules; auditor can verify no force-push events in repository audit log |
| CC1.8 | Rollback capability for changes | Migration rollback is manual (no down-migration files); Helm rollback via `helm rollback`; pack rollback via re-install of prior version | 🚧 | Gap — `installed_skill_packs` records only the current version; prior version history not captured in table |

**SOX-specific note (finance-fpa):** When `finance-fpa` v1.0.0 is installed, the install event is
recorded in `installed_skill_packs` with `pack_slug = 'finance-fpa'`, `version = '1.0.0'`, and
`installed_at` timestamp. For an auditor examining whether a change to the FP&A processing system
was authorised, this row — combined with the GitHub release tag and PR review history — constitutes
the change-approval evidence chain.

## ITGC Category 2 — Logical Access (CC2)

*COBIT domain: DSS05 (Managed Security Services), APO01.08. AICPA TSP: CC6.1–CC6.7.*
*Controls over authentication, authorisation, and segregation of duties.*

| ITGC Ref | Description | Xiaoguai feature | Status | Audit-evidence path |
|----------|-------------|-----------------|:------:|---------------------|
| CC2.1 | Unique user identification for all access | OIDC token-based authentication; every API call carries a bearer token bound to a principal; `xiaoguai-auth` crate validates tokens before any action | ✅ | Bearer token audit trail in `audit_log` (`actor` column); OIDC provider user directory |
| CC2.2 | Password / credential controls | Delegated to OIDC provider; Xiaoguai never stores raw credentials; API keys stored hashed | ✅ | OIDC provider credential policy; application does not accept plaintext passwords |
| CC2.3 | Role-based access control (RBAC) | Casbin RBAC enforcer in `xiaoguai-auth`; roles: viewer / operator / admin per tenant; `PUT /v1/tenants/:id/roles` admin API provisions roles; all role changes audit-logged | ✅ | `SELECT * FROM casbin_rules WHERE tenant_id = ?`; audit log entries for role-change events |
| CC2.4 | Segregation of duties — AI-agent actions | HotL policy (`hotl_policies`) requires a named human approver for privileged actions; `hotl_usage_log` records both the initiating agent and the approving actor separately | ✅ | `SELECT * FROM hotl_usage_log WHERE tenant_id = ? ORDER BY occurred_at;` — preparer (agent) vs. approver (human) rows visible |
| CC2.5 | Escalation / approval hierarchy | Per-tier HotL escalation: `escalate_to` field routes to named IM channel; multi-tier approval configurable via policy scope nesting | ✅ | `hotl_policies.escalate_to` column; IM gateway escalation audit trail in `audit_log` |
| CC2.6 | Tenant isolation — no cross-tenant data access | Postgres RLS enforces `tenant_id` predicate on every query; isolation enforced at DB layer, not application layer | ✅ | RLS policy DDL (`ALTER TABLE ... ENABLE ROW LEVEL SECURITY`); penetration test result (if performed by operator) |
| CC2.7 | Session termination and revocation | `DELETE /v1/sessions/:id` revokes active sessions; OIDC token expiry enforced; Casbin role revocation via admin API | 🚧 | Revocation endpoint documented; no automated session-expiry report generated |
| CC2.8 | Privileged access review (PAR) | Casbin rules queryable via admin API; no automated PAR report generated by platform | 🛣 | `SELECT * FROM casbin_rules WHERE role = 'admin';` — manual query; auditor must request this from operator quarterly |
| CC2.9 | No shared / generic accounts | OIDC provider enforces individual accounts; API service accounts are per-integration, not shared | ✅ | OIDC provider user list; `audit_log.actor` uniqueness analysis |

**SOX-specific note (finance-fpa SoD):** The finance-fpa `variance-analyzer` agent writes draft
narratives to `fpa_commentary_log`; the HotL policy requires a named human approver (CFO or
delegate) to approve before any narrative is published. The `hotl_usage_log` row for the approval
event captures the approver's `actor` ID separately from the agent's initiating event — creating a
machine-readable preparer / approver segregation record satisfying AICPA CC6.3 (segregation of
duties over financial reporting).

## Application Controls — Finance-FPA Pack

When `packs/finance-fpa/` is deployed, the following application-level controls activate on top
of the ITGC layer above. Application controls are distinct from ITGCs: they operate on financial
data directly and are often tested separately by auditors.

| App Control | Description | Mechanism | Status | Evidence |
|-------------|-------------|-----------|:------:|----------|
| AC-1 — Variance materiality gate | No variance narrative is published for immaterial variances; `variance-threshold.yaml` watch fires only when adverse variance exceeds 10% or $50 k (whichever is larger) | `packs/finance-fpa/watches/variance-threshold.yaml`; `fpa_variance_flags.severity` column | ✅ | `SELECT * FROM fpa_variance_flags WHERE period_id = ? AND severity = 'critical';` |
| AC-2 — Human approval before publication | All variance narratives are drafted by `variance-analyzer` agent and held in `fpa_commentary_log` with `status = 'draft'`; no content is published without an explicit HotL approval event | `hotl_policies` scope `fpa_narrative_publish`; `fpa_commentary_log.status` transitions | ✅ | `SELECT * FROM fpa_commentary_log WHERE period_id = ? ORDER BY created_at;`; `hotl_usage_log` join |
| AC-3 — Preparer / approver segregation | The agent that drafts the narrative (`variance-analyzer`) and the human that approves it are recorded as separate actors in `hotl_usage_log`; agent cannot self-approve | HotL enforcer requires `require_human_approval = true`; `hotl_usage_log.actor` ≠ `agent_name` constraint | ✅ | `hotl_usage_log` rows — distinct `actor` values for draft and approve events for same `fpa_narrative` |
| AC-4 — Period-close trigger integrity | Processing only begins after `fpa.period_closed` event, which requires `fpa_periods.is_closed = true` and `closed_at IS NOT NULL`; prevents processing open periods | `packs/finance-fpa/watches/period-close-trigger.yaml`; `fpa_periods` DDL | ✅ | `SELECT period_label, is_closed, closed_at FROM fpa_periods;` |
| AC-5 — Outcome telemetry for financial impact attribution | `agent_outcomes` table records `revenue_usd`, `cost_saved_usd` per session; finance-fpa agents post outcome records attributing each narrative cycle to measurable business value | Migration 0012 (`agent_outcomes`); `outcome-telemetry` feature flag in `pack.yaml` | ✅ | `SELECT * FROM agent_outcomes WHERE agent_name LIKE 'fpa_%' ORDER BY attributed_at;` |
| AC-6 — Anomaly-flagged revenue shortfalls escalated before close | `anomalies/revenue-shortfall.yaml` fires during the period if rolling-window revenue deviates beyond EWMA threshold; escalates to FP&A Slack channel before close, not after | `packs/finance-fpa/anomalies/revenue-shortfall.yaml`; IM gateway | ✅ | IM channel history; `anomaly_events` records (when available) |

**AR-collections pack** (also potentially in scope for SOX): `packs/ar-collections/` monitors
DSO drift via `anomalies/dso-drift.yaml` and automates dunning workflows. If accounts-receivable
balances are material to financial statements, auditors may extend ITGC review to this pack.
Evidence path: `ar_aging` table (migration `packs/ar-collections/migrations/0001_ar_aging.sql`);
dunning agent HotL logs.

## Audit Walkthrough — What a Big 4 Auditor Would Request

The following simulation mirrors a standard ITGC walkthrough request list (PBC — Provided By
Client) for a SOX § 404 review covering the Xiaoguai platform supporting the finance-fpa pack.

### PBC Item 1 — Change Management Evidence

| Evidence item | How to gather | Location |
|---------------|--------------|----------|
| List of all production deployments in the audit period | GitHub Releases page; `git tag --list 'v*'` | GitHub release history |
| Change approvals for each release | GitHub PR merge events linked to each release tag | GitHub PR history (`git log --merges`) |
| Schema changes applied in the audit period | Migration files added to `crates/xiaoguai-storage/migrations/` between audit-period start and end tags | `git log --diff-filter=A -- 'crates/xiaoguai-storage/migrations/*.sql'` |
| Finance-fpa pack install / update events | `installed_skill_packs` table | `SELECT pack_slug, version, installed_at FROM installed_skill_packs WHERE pack_slug = 'finance-fpa';` |
| Evidence that no changes bypassed review | GitHub branch-protection audit log (admin → Settings → Branches → audit log) | GitHub organisation audit log |

### PBC Item 2 — Logical Access Evidence
