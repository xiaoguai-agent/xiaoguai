# v0.6.2 — LlmRouter wired + JwtValidator boot + Casbin per-route

Status: **Done** — 2026-05-23.

## Goal

Close the three remaining strands carved off the original v0.6.1 plan
when v0.6.1 was scoped to RLS-only:

1. **LlmRouter** actually replaces the `MockBackend` default in
   `xiaoguai-core serve`, reading provider rows from PG on boot.
2. **JwtValidator** is wired into `AppState.auth` behind a
   `XIAOGUAI_AUTH__REQUIRED=true` settings knob.
3. **Casbin** per-route middleware enforces the policy file that was
   already shipped with v0.5.1.

These three together are what turns "a server that boots" into "a
server you could plausibly run in a production demo with real users".

## What landed

### `xiaoguai-llm`

- `impl LlmBackend for LlmRouter` so `Arc<LlmRouter>` slots into
  `AppState.backend: Arc<dyn LlmBackend>` and through the agent loop
  unchanged. The trait impl forwards to `chat_stream(ResolveCtx::default(),
  req)`; tenant-aware per-request routing remains a follow-up (would
  require a wider trait or a context-carrying adapter).
- New `build` module exposing:
  - `build_router(rows: &[LlmProvider], env: &dyn EnvResolver) ->
    (LlmRouter, BuildReport)` — pure function over a row list.
  - `EnvResolver` trait with `OsEnvResolver` for production and any
    `Fn(&str) -> Option<String>` for tests.
- 6 new unit tests covering empty input, fallback ordering, system
  defaults, missing API-key warnings, Ollama-kind ignoring api_key_env,
  and the closure-based resolver path.

### `xiaoguai-config`

- `AuthSettings.required: bool` (default `false`), overridable via
  `XIAOGUAI_AUTH__REQUIRED=true`.

### `xiaoguai-api`

- New module `rbac` with:
  - `path_to_resource(path)` — strips `/v1`, normalises single-segment
    "collection" paths with a trailing `/` so Casbin's `keyMatch`
    against `/sessions/*` succeeds.
  - `method_to_action(m)` — GET/HEAD/OPTIONS → `read`, DELETE →
    `delete`, everything else → `write`.
  - `require_authorized(authz, req, next)` axum middleware. Reads
    `Claims` from extensions (set by `require_bearer`), iterates roles,
    allows on the first match, returns 403 on no match.
- `AppState` gains `authz: Option<Arc<Authz>>`. When `None`, the route
  layer is not wired and the middleware is bypassed entirely.
- `router()` mounts the rbac layer *inside* the bearer-auth layer so
  the rbac layer always sees `Claims`. Public `/healthz` is unaffected.
- 8 new unit tests + 5 new integration tests covering allow / deny /
  empty-roles / system_admin wildcard / healthz-bypass.

### `xiaoguai-core` (the binary)

- `run_serve` now:
  1. Reads `llm_providers` via `PgLlmProviderRepository::list_global()`.
  2. If empty, logs a warning and keeps `MockBackend` so a fresh
     deployment still boots and serves a deterministic response.
  3. Otherwise calls `build_router(&rows, &OsEnvResolver)`, logs any
     warnings, picks a sensible default agent model from the rows, and
     uses `Arc::new(router)` as `AppState.backend`.
- Two helpers `build_auth` / `build_authz` gate JWT + Casbin wiring on
  `settings.auth.required`. When `false`, both return `None` and the
  router behaves as before.

### In-memory test stubs

- Updated every `AppState { ... }` construction site (api / im-gateway
  / cli tests) to include `authz: None` so they continue to compile and
  exercise the dev-mode path.

## Deferred to v0.6.3 / later

- **Per-request tenant routing in LlmRouter via the trait surface.**
  The trait impl uses `ResolveCtx::default()`; threading
  `tenant_id` (and possibly `user_id`) through the agent loop into a
  ResolveCtx-aware path is a separate slice.
- **Rate-limit middleware per tenant.**
- **WebSocket fallback `/v1/sessions/:id/ws`.**
- **OpenAPI spec generation via `utoipa`.**
- **`/v1/admin/tenants` + `/v1/admin/audit` endpoints** to back the
  admin-ui placeholder panes.
- **`GET /v1/sessions?user_id=...` listing** for the chat-ui sidebar.
- **`provider remove --tenant` / `mcp remove --tenant`** scoped delete
  flags.
- **`UserRepository` migration to the shared `begin_tenant_tx` helper**
  (cleanup, not a correctness gap).
- **Dynamic policy reload** for Casbin — currently the policy is
  compiled into the binary; runtime grants via `Authz::grant_role`
  exist but no API surfaces them yet.
