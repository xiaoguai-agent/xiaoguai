# v0.10.1 — reactive triggers

**Status:** done.

## What shipped

Second P1 slice from the v0.9–v0.12 roadmap. The scheduler grew the
*reactive* half of the `passive → reactive → proactive` ladder
(roadmap §2.3). Cron / Interval jobs already worked in v0.10.0; with
this tag a job can also be driven by an external event — file
changes, webhook deliveries, git pushes, database rows — all routed
through one `tokio::select!` inside `JobRunner::run_loop`.

The differentiation principle (roadmap §3) holds: platforms with
strong triggers are weak on agents and vice versa. v0.10.1 unifies
both behind one runtime, same `JobExecutor` / `AuditAppender` /
`PushSink` traits as v0.10.0. Reactive sources don't fork the
machinery; they just produce `TriggerEvent`s that go through the
same fire → audit → push pipeline.

## Pieces

### `Trigger` enum — four new reactive variants

`trigger.rs` adds:

- `Trigger::FileWatch { path }` — driven by the `notify` crate.
- `Trigger::Webhook { route_id }` — driven by xiaoguai-api (HTTP
  route lands with operator wiring; in v0.10.1 the producer is
  in-process).
- `Trigger::GitPush { repo_url, branch }` — data variant only.
  Concrete polling/webhook source ships in v0.10.1.x; storing the
  variant now means persisted job rows are forward-compatible.
- `Trigger::DbPoll { query }` — data variant only. Concrete poller
  pairs with the PG scheduler repos in v0.12.0.

Each variant has an eager constructor (`Trigger::file_watch(path)`,
…) that rejects empty inputs. Reactive triggers return `None` from
`next_fire_after` and `false` from `is_scheduled` — they're invisible
to the wall-clock scan in `JobRepository::list_due` and only fire
when a matching `TriggerEvent` arrives.

The new `Trigger::is_scheduled()` / `Trigger::is_reactive()` helpers
let the repository layer filter reactive jobs out of the
`list_due` result without parsing each variant.

### `trigger_source` module

New `TriggerEvent` (`job_id` + `fired_at` + opaque `detail` JSON) +
`TriggerSource` async trait + `event_channel()` helper (returns a
`tokio::sync::mpsc::{Sender, Receiver}` pair with a capacity-256
default). The shared channel pattern means a single runner can fan
in from many sources.

Source contract:

* `id(&self) -> &str` — diagnostic label.
* `async fn start(&self, tx: EventSender) -> Result<(), SourceError>`
  — spawn whatever background work the source needs; sources own the
  cloned `EventSender` and exit cleanly when the receiver drops.

### `sources/file_watch.rs` — real `notify` watcher

`FileWatchSource` wraps `notify::RecommendedWatcher` (fsevent on
macOS, inotify on Linux, ReadDirectoryChangesW on Windows). Routes
are `(job_id, path)` pairs; on each filesystem event the source
emits one `TriggerEvent` per route whose `path` is an ancestor of
(or equal to) the changed path.

`notify` events go through a `std::sync::mpsc` ↔ `tokio::sync::mpsc`
bridge run on `tokio::task::spawn_blocking` — keeps the runner's
event channel single-impl and stops `notify`'s thread when the
runner drops its receiver (`blocking_send` returns `SendError`, the
bridge task bails).

Event filter accepts create / data-modify / name-modify / file-or-
folder remove kinds. Pure access (`inotify` `IN_ACCESS`) is dropped
— it fires on every read and would saturate the runner.

### `sources/webhook.rs` — in-process push handle

`WebhookSource::add_route(WebhookRoute::new(route_id, job_id))`
registers an HTTP-route → job binding. `WebhookSource::push(
route_id, detail)` fans the call out to every bound job. The HTTP
plumbing (axum handler at `/v1/scheduler/webhooks/:route_id`) lands
together with the operator wiring deferred from v0.10.0.

### `JobRunner` upgrades

- `fire` is now private, takes an optional `extra_audit_detail:
  serde_json::Value` and merges it into every audit row under the
  `trigger` key — that's how reactive sources surface "fired because
  file X changed at T" in the audit-first console (v0.11.1).
- New `fire_event(TriggerEvent)` — fetches the job, skips silently
  if it's missing or disabled (deletes racing with events are
  normal), then calls `fire` with the event's detail.
- New `fire_now(job_id)` — operator helper for admin-ui's "Run now"
  button.
- New `run_loop(events, tick_interval)` — `tokio::select!` between
  the event channel and an optional `tokio::time::interval`. Uses
  `MissedTickBehavior::Skip` so a long fire sequence doesn't queue
  a burst when control returns. Loop exits cleanly when both arms
  are exhausted (event channel closed AND `tick_interval is None`);
  otherwise the caller cancels via task abort or wraps the future
  in its own `tokio::select!`.

### `repository::InMemoryJobRepository::list_due` filter

Filters reactive triggers out of the scheduled scan
(`trigger.is_scheduled()`). Reactive jobs still appear in `get` /
`upsert` / `record_fire` so the bookkeeping bumps `last_fire_at`
after a reactive fire (next_fire_at stays None — there's no
schedule to advance).

### Workspace plumbing

- `notify = "7"` added to `[workspace.dependencies]`.
- `tempfile = "3"` added (dev-dep for the integration test).
- `xiaoguai-scheduler/Cargo.toml` enables `tokio` features `sync` +
  `macros` + `rt` (`run_loop` needs `select!`); dev-deps gain
  `tempfile`.

## Deferred

- **`GitPushSource` + `DbPollSource`.** The trigger data variants
  ship so persisted job rows are forward-compatible, but no concrete
  source. Pairs naturally with the runtime extraction in v0.12.0 (PG
  repos are a prerequisite for `DbPoll`, and `GitPush` is best
  triggered from the webhook source once xiaoguai-api fronts it).
- **xiaoguai-api webhook route.** `/v1/scheduler/webhooks/:route_id`
  → `WebhookSource::push(...)`. Cross-crate; bundled with operator
  wiring (next bullet) so the live path lands together.
- **Operator wiring in `xiaoguai-core`.** Still deferred from
  v0.10.0. `JobRunner::run_loop(rx, Some(30s))` in a `tokio::spawn`
  inside `serve`; configurable `scheduler.tick_interval_secs` in
  `xiaoguai-config::Settings`; an admin-ui "Run now" button calling
  `JobRunner::fire_now`. Held back to land alongside the PG repos in
  v0.12.0 so the same slice introduces durability + the production
  binary entry point.
- **PG repository impls.** Still deferred to v0.12.0 alongside the
  `xiaoguai-runtime` extraction. v0.10.1 expanded the in-memory
  `JobRepository` filter to skip reactive triggers — the PG version
  will need the same predicate (`WHERE jsonb_extract_path_text(
  trigger, 'type') IN ('cron', 'interval') AND ...`).
- **`notify-debouncer-full` integration.** Today the file-watch
  source emits one event per `notify` event. A user editing a
  10 MB file in vim can yield several events in a few ms. If
  feedback shows we need stronger coalescing we'll layer
  `notify-debouncer-full` in v0.10.1.x; the source-side API stays
  the same.
- **Cancellation token in `run_loop`.** Today callers cancel by
  aborting the spawned task. A built-in `CancellationToken` knob
  would be cleaner but ties us to `tokio_util` and adds zero
  expressive power over the abort pattern. Revisit if a caller
  actually wants graceful drain semantics.
- **Proactive triggers** (LLM-initiated + per-user budget + reason
  field) — v0.10.2. The push payload schema will gain `reason:
  String` then; adding the field later is non-breaking.

## Design decisions worth flagging

- **One unified event channel** instead of per-source channels. The
  runner only needs one `mpsc::Receiver` to drain; sources clone the
  `EventSender` they receive in `start`. Keeps `run_loop` a single
  `tokio::select!` no matter how many sources are wired.
- **`TriggerEvent.detail` is opaque JSON.** The runner doesn't
  parse it; it just merges into the audit row under `details.trigger`.
  That's enough for the audit-first console (v0.11.1) to render
  "file X changed" without the runner growing a per-source type
  switch.
- **`run_loop` with an active ticker only exits via cancellation**,
  not via channel close. A deployment without reactive sources
  still ticks; the operator binary owns the shutdown signal and
  wraps the future in its own `tokio::select!`.
- **Filter at the repository layer, not at the runner.** Reactive
  jobs never make it into `list_due`'s result, so `tick` doesn't
  have to know about reactive variants. Keeps the PG repo's
  equivalent SQL filter simple too (a single `WHERE` predicate, not
  a join).
- **`fire_event` swallows missing/disabled-job races.** Reactive
  sources outlive the jobs they fire; a user disabling a job mid-
  stream shouldn't crash the loop. We log at `debug` and continue.
- **GitPush + DbPoll variants ship without sources.** Persisting
  the data shape now means a user who creates a `db_poll` job
  before the source lands gets a stable row (won't migrate when the
  source is wired). Reactive-only jobs simply don't fire until
  there's a source to push events.
