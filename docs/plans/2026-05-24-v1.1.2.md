# v1.1.2 — conversation fork

Branch a session from any prior message so the user can explore an
alternative continuation without losing the original thread.

## Why

Single-conversation chat has a sharp edge: once a user wants to try a
different prompt or continuation, the only options today are "edit and
hope" or "start over from scratch and re-type 10 turns of context."
Notion AI, Claude.ai, ChatGPT all ship some flavour of "branch from
here." This is the simplest one that's still useful — clone the prefix
into a new session, leave the parent untouched, surface the lineage on
the new row.

## Scope

- DB schema: parent pointer + fork message pointer on `sessions`.
- Repo: copy `messages` with `created_at <= cutoff_msg.created_at` from
  parent → child. Ordering by `created_at` (NOT id; some seed ids are
  UUIDs and don't sort meaningfully).
- API: `POST /v1/sessions/:id/fork` body `{from_message_id, title?}` →
  201 with the new `SessionResponse`.
- Frontend: hover-reveal "Branch from here" button on every assistant
  bubble. Click → call the API → `window.open` the new session URL.
- Wire types in `@xiaoguai/shared`.
- Production wiring in `xiaoguai-core/src/main.rs::run_serve` via a new
  `xiaoguai-core/src/sessions_bridge.rs` module.

## Out of scope

- Forking from a *user* message (only assistant bubbles get the button
  — forking after the model's reply is the meaningful unit; forking
  right before your own prompt is just "create a new session").
- A "lineage tree" UI in the sidebar — the new session simply shows up
  in the regular list, ordered by `updated_at`. Tree visualisation
  defers to a later milestone.
- Cross-tenant forks. The endpoint enforces same-tenant lineage and
  surfaces 404 if the caller can't see the parent.
- Streaming the fork (it's a one-shot DB copy; the response is the new
  session row).

## Implementation plan

1. Migration `0008_session_parent.sql` — two nullable columns + partial
   index. Verify: `\d sessions` shows the columns.
2. Extend `xiaoguai_types::Session` with both fields defaulting to
   `None`. Verify: `cargo build -p xiaoguai-types` clean.
3. Add `RepoError::Unsupported(String)` so the default trait impl
   doesn't need every mock to override. Verify: `cargo build
   -p xiaoguai-storage` clean.
4. Add `SessionRepository::fork` trait method with a default-error
   body; override only in `PgSessionRepository`. Verify: workspace
   build clean, no mock needs an explicit override.
5. New shim trait `SessionForker` in `xiaoguai-api/src/sessions_ext.rs`
   + `SessionForkError`. Re-exported from `lib.rs`.
6. New `POST /v1/sessions/:id/fork` handler in `routes/sessions.rs`.
   Add to `routes/mod.rs` at the end of the v1 chain.
7. `AppState.session_forker: Option<Arc<dyn SessionForker>>`. Every
   existing AppState construction site gets `session_forker: None,`
   appended via a sed script. Verify: `cargo build --workspace`.
8. Production impl `PgSessionForker` in
   `xiaoguai-core/src/sessions_bridge.rs`. Wire in `run_serve`.
9. Unit + integration tests for the copy semantics, tenant isolation,
   404 on unknown `from_message_id`.
10. Frontend: extend `@xiaoguai/shared` with `ForkSessionRequest` and
    the two new `Session` fields, add `forkSession()` to the client,
    add the hover button in `ChatPage.tsx`, navigate via `window.open`.
11. Plan doc (this file).
12. Tag `v1.1.2` locally. Don't push.

## Deferred

- "Branch from here" button on user bubbles — see scope note above.
- Visualising the fork tree in the sidebar.
- A "merge branches" operation. Branches stay distinct; this isn't git.
- Showing the parent/cutoff in the new session header. The columns
  ship on the wire so a later UI pass can render the breadcrumb without
  another DB change.
- Bulk fork from a search hit (operator-facing "regression case from
  here" tooling).
