# v0.7.4 — IM gateway PG-history default + persisted tool turns + replay cap

**Status:** done.

## What shipped

Closes three v0.7.4 backlog items:

1. **PG-backed IM history by default in `xiaoguai-core`.** v0.7.3 built
   the `PgImHistoryStore` trait + impl but the production binary never
   instantiated it (the in-process `ConversationHistory` was the
   default). v0.7.4 flips the default: when the PG pool is up and the
   Feishu signing key is provided, `PgImHistoryStore` is the store. New
   env-var escape hatch `XIAOGUAI_IM__USE_IN_PROCESS_HISTORY=true`
   falls back to the in-process store for single-replica deploys that
   don't want PG history.
2. **`Role::Tool` turns persist in the IM transcript.** `llm_to_domain`
   no longer drops tool turns on write; the IM transcript now survives
   a restart with full tool context. `domain_to_llm` still skips tool
   turns on replay (matches the v0.7.3 reader semantics — the agent
   only sees text, the audit log sees everything).
3. **Per-conversation replay cap.** New `[im].max_messages_per_conversation`
   knob (default `50`). The PG history store's `snapshot` reads at most
   this many trailing turns; older messages remain in the DB for audit
   but aren't replayed to the agent.

Also lands as a side-effect (necessary for #2 to work in
`run_agent_and_reply`): the gateway now persists the entire newly-
produced sequence (user turn + any assistant `tool_calls` + tool
results + final assistant text), not just the `(user, assistant)` pair.
Existing in-memory store tests stay green because the simple path
collapses to the same two-message extend.

The Feishu webhook is now actually mounted by `xiaoguai-core` when
`XIAOGUAI_IM_FEISHU__VERIFICATION_TOKEN` is set; previously the
gateway crate existed but nothing in core wired it. App-id / app-secret
env vars switch the reply sink from stub to the real OpenAPI client.

## Pieces

### Config

* New `[im]` section in `xiaoguai-config::Settings`:
  - `use_in_process_history: bool` (default `false`).
  - `max_messages_per_conversation: usize` (default `50`).
  Defaults preserved across config-file and env-var loaders.

### IM gateway

* `llm_to_domain` returns `DomainMessage` directly (was
  `Option<DomainMessage>`); the v0.7.3 `Role::Tool → None` filter is
  removed. `domain_to_llm` still returns `None` for tool turns so the
  replay window stays text-only.
* `run_agent_and_reply` (`router.rs`):
  - Calls `history.resolve_tenant` to wire v0.6.5 tenant routing.
  - Persists `outcome.messages[idx_of_inbound..]` to the store —
    catches the whole new turn including any `tool_call` /
    `tool_result` messages. Falls back to `[inbound, assistant_reply]`
    when the slide-window dropped the inbound (very long threads).

### Core binary

* New `build_feishu_gateway(settings, &pool, &state, &default_model)`
  helper builds the right history store, the right Feishu provider
  (API sink when `XIAOGUAI_IM_FEISHU__APP_ID/APP_SECRET` are set, stub
  otherwise), and returns the axum router fragment.
* New `serve_with_state_and_extras(addr, state, extra)` helper merges
  the IM gateway fragment with the main API router (the IM mount is
  per-operator wiring, not part of the stable HTTP contract — keeps
  `xiaoguai-api::serve_with_state` clean).
* Required deps added: `xiaoguai-im-gateway`, `xiaoguai-im-feishu`,
  `axum` (for the merge helper).

## Deferred

* **DingTalk + WeCom adapters.** Same `ImProvider` shape, separate
  webhook signature schemes. Lift directly when the v0.7.x backlog
  needs to ship a second IM channel.
* **Interactive cards (rich messages w/ tool-call viz).** Feishu /
  DingTalk card schemas are vendor-specific; the gateway returns plain
  text today.
* **Long-task handoff** (reply > Feishu hard size cap). Need a split-
  and-followup pattern.
* **Encrypted-payload Feishu apps.** The current code only handles the
  plaintext-payload variant; encrypted-payload variant needs the AES
  key plumbed.
* **Per-conversation token cap (not just message cap).** The current
  knob is message-count; a token-budget knob lands when the tokenizer
  ships in `xiaoguai-llm`.
* **Audit emission from the IM path.** v0.7.4 persists transcripts;
  audit-log rows for IM-triggered actions wait for the v0.6.5 deferral
  on per-handler `AuditAppender` wiring.
