## 1. Positioning

> **xiaoguai is a Rust-implemented, audit-first, scheduler-native local
> agent platform.**
>
> We do not compete on prompt-magic, UI polish, or agent marketplace size.
> We compete on engineering seriousness: every tool call has an HMAC-chain
> audit row, every scheduled job has retry + replay + transcript, every
> model interaction has a regression-eval safety net.
>
> Audience: technical individuals + small teams + scenarios with a
> compliance / traceability constraint.

This is the consequence of the CLAUDE.md preamble — *"models are
unreliable, but systems can be reliable"* — taken to a product.

## 2. Operating principles (locked 2026-05-23)

1. **RAG: wrap a mature layer, don't build our own.** R2R (HTTP) is the
   primary backend; an embedded Qdrant + `fastembed-rs` path exists for
   single-binary use cases. We do not implement chunking / embedding /
   re-ranking from scratch.
2. **Connectors: reuse what exists.** MCP is the connector abstraction.
   We do not invent a parallel trait. We only write our own connector
   when no MCP server covers it (macOS native sources are the obvious
   gap).
3. **Scheduler: design for `passive → reactive → proactive`.** Cron is
   table stakes; event-driven triggers (file change, webhook, git push)
   match n8n; LLM-initiated proactive runs with a reason field and a
   per-user budget are the differentiator nobody has shipped well.
4. **Web console: personal-use UX, not enterprise admin.** Settings
   ocean is the enemy. Presets and good defaults; advanced controls live
   behind a fold. Audit view is the *default* home page, not chat —
   inverts the OpenWebUI / LobeChat / LibreChat convention.

## 4. Non-goals

Listing these is half the point of the doc — saying no early saves
hundreds of hours.

- ❌ **Letta-style memory abstraction (3-tier / Core+Recall+Archival).**
  Letta has a moat; we won't out-Letta Letta. Users who need that
  abstraction should use Letta and point its API at our LLM router.
- ❌ **Visual DAG / flow builder.** n8n / Flowise / Dify own this. We
  stay code-first.
- ❌ **Agent marketplace UI.** LobeHub's MCP marketplace has ~10k
  entries; we can't catch up. We curate, we don't host.
- ❌ **Knowledge-graph RAG.** R2R + RAGFlow have heavy investment;
  marginal value for our audience.
- ❌ **Single-binary + SQLite mode.** Deferred. The PG path is the
  spine; revisiting this is a separate desktop-mode decision.
- ❌ **Visual prompt IDE / chain debugger.** Out of scope; transcripts
  + eval suite is our analog.

## 5. Design decisions to lock now

These cross many tags and benefit from being decided once.

### 5.1 `ContentBlock::Citation` is a first-class variant

Today `ContentBlock = Text | ToolCall | ToolResult`. Citations are
common enough (every RAG turn produces them) that hiding them inside
`ToolResult` payloads breaks chat-ui rendering and the soon-to-arrive
`xiaoguai-eval` graders.

Add:

```rust
ContentBlock::Citation {
    source_uri: String,     // file:// or http(s):// or a custom scheme
    span: (u32, u32),       // [start_byte, end_byte] or [line_start, line_end]
                            // — decide once and stick to it; we choose lines
    preview: String,        // chunk text shown in the hover card
    score: f32,             // retrieval score; tie-break in UI ordering
    collection_id: String,  // for back-tracing which RAG collection produced it
}
```

Touches: `xiaoguai-types`, every repo's `MessageRow → Message`
deserialization, chat-ui rendering. One-time pain; everything downstream
benefits.

### 5.2 MCP server publish reuses the existing `Toolbox`

`xiaoguai-api` exposes `/v1/mcp/serve` (SSE) and `/v1/mcp/stream`
(streamable-HTTP). The MCP tool list is the `Toolbox`'s tool list. No
duplication: agents internal to xiaoguai see the same surface external
agents see.

This is the path Dify v1.6 took. Reference their schema; don't invent.

### 5.3 Scheduler `JobRun` is audit-tracked by default

Every job run writes `audit_log` rows with `actor = "scheduler:<job_id>"`.
Cross-reference from `JobRun.audit_session_id` → `sessions.id` →
`audit_log.session_id`. This is how the audit-first console gets a
unified view across chat / IM / scheduled.

### 5.4 Eval substrate is the existing sessions + audit tables

No new DB. An eval case is `(input_messages, expected_assertions)`.
Assertions run against:

- **Outcome graders** — final state of the in-memory repo, final
  message content, tool invocation count.
- **Transcript graders** — sequence of `AgentEvent` records, sequence
  of tool calls (name + argument hash).

`MockBackend::with_script(...)` already produces deterministic traces;
eval cases combine a script and an assertion list. The CLI
(`xiaoguai eval run --suite regression`) walks the cases, prints
pass/fail, writes a JSON report. No new infra.

### 5.5 Proactive budget defaults are conservative

Per user, per day: **3 proactive pushes** by default. Configurable.
The push payload schema *forces* a `reason: String` field that the
sink renders first. Sinks may refuse to deliver if `reason.is_empty()`.

This is non-negotiable for keeping the feature from devolving into
spam — community signal is unanimous.

### 5.6 RAG citation schema is the contract

Every backend (R2R, Qdrant, future ones) must populate the same
citation fields. If R2R doesn't return line spans, the adapter computes
them from chunk offsets at ingest time. Adapters that can't produce a
citation MUST NOT silently emit unsourced text — they emit a
`ToolResult` with `is_error: false` but no citation block, so the
console can flag it.

### 5.7 No memory abstraction beyond `sessions.messages`

Conversation history = `messages` rows. Cross-session context, if ever
needed, is the user's RAG collection (their notes). We do not invent
"working memory" / "episodic memory" / "archival memory" layers. Refer
users wanting that to Letta.
