# v0.11.2 — eval pane (run suites + convert session to case)

**Status:** done.

## What shipped

Third P1 console slice. Closes the audit ↔ eval loop the roadmap §3
v0.11.2 row sketches: operators can run a `.eval.yaml` suite from the
browser and one-click "convert prod run to regression case" against any
session id the audit-first console surfaced. The v0.11.0 substrate
(`xiaoguai-eval`) and v0.11.1 console layout (sidebar + pane +
auto-refresh) both land in production traffic shapes here for the first
time.

Roadmap §5.4 still holds: **no new DB tables.** A session-to-case
translation reads `sessions` + `messages` + `audit_log` rows that
already exist; eval reports stay JSON-on-disk per the v0.11.0 plan; the
console caches the last verdict per suite in `localStorage`.

## Pieces

### Backend — `xiaoguai-api::eval` module

* `EvalService` struct holds the runner + suites directory + a
  pluggable `CaseFromSessionSource`. `AppState.eval: Option<Arc<EvalService>>`
  follows the same opt-in pattern as `audit` / `today` — `None` → 503.
* Three new admin routes (all behind the existing bearer / Casbin /
  rate-limit stack):
  - `GET  /v1/admin/eval/suites` — enumerate suites discoverable on
    disk. Directories holding `*.eval.yaml` files count as multi-case
    suites; loose top-level `*.eval.yaml` files become single-file
    suites named after the file stem.
  - `POST /v1/admin/eval/run` — `{ suite_name, cases_dir? }`. Loads the
    suite via `EvalSuite::load_from_dir`, executes synchronously with
    the same `EvalRunner` + `DefaultEvalAgentBuilder` the CLI uses, and
    returns the full `EvalReport`. Caps: `MAX_CASES_PER_RUN = 100`
    (returns 413) and `MAX_RUN_DURATION = 60s` (returns 504).
  - `POST /v1/admin/eval/case-from-session` — `{ session_id }`. Reads
    the session's messages + matching `tool.invoke` audit rows, projects
    them into a ready-to-edit `EvalCase`, and returns the YAML string +
    a suggested filename. The endpoint does **not** write to disk — the
    operator reviews and pastes.
* `EvalServiceError` maps cleanly: `NotFound → 400` (the route is
  cross-tenant admin; `not_found` is reserved for sessions that exist
  but the caller can't access), `InvalidArgument → 400`,
  `SuiteTooLarge → 413`, `SuiteTimedOut → 504`, `Backend → 500`. New
  `ApiError::PayloadTooLarge` + `ApiError::GatewayTimeout` variants
  carry the 413 / 504 mappings.
* `StaticCaseFromSessionSource` ships for route tests, mirroring how
  `StaticAuditReader` / `StaticTodayReader` exercise the trait without
  Postgres.

### Backend — `xiaoguai-core::eval_bridge`

Same layering choice as `audit_bridge.rs` / `today_bridge.rs`: the api
crate stays sqlx-free, the SQL lives in the binary.
`PgCaseFromSessionSource`:

* Looks up `sessions.tenant_id` for existence check; returns `Ok(None)`
  for an unknown id so the api layer can render `404 → 400` cleanly.
* `SELECT role, content, created_at FROM messages WHERE session_id = $1
  ORDER BY created_at ASC` — translates `ContentBlock::Text` blocks into
  `Message::content`, drops tool blocks (the eval substrate replays them
  via the mock script), tracks the last assistant text as the
  "expected final reply".
* `SELECT details FROM audit_log WHERE action = 'tool.invoke' AND
  details->>'session_id' = $1 ORDER BY ts ASC` — pulls the canonical
  pattern documented in `xiaoguai-eval/tests/regression_from_audit.rs`.
  Bypasses `MessageRepository` (which is RLS-scoped); the admin path is
  cross-tenant.

### Backend — `xiaoguai-config`

New `[eval]` section:

```yaml
eval:
  suites_dir: "./eval-suites"    # default
  max_iterations: 0              # 0 = use xiaoguai-agent default
```

`EvalSettings` is `Option<...>`-friendly so old config.yaml files still
load. The directory not existing isn't an error — `list_suites` just
returns `[]`.

### Frontend — `Eval` pane

New top-level admin route below `Today`:

* Left column: clickable list of suites with case-count subtitle.
* Right column tabs:
  - **Run suite** — primary "Run suite" button calls
    `POST /v1/admin/eval/run` synchronously, shows an inline spinner
    while waiting (no SSE yet — deferred), then renders a pass-rate
    card + per-case table. Failed rows expand to show every
    `reasons[]` entry. Verdict cached in
    `localStorage[xiaoguai.eval.lastRun.<suite>]` so a refresh
    doesn't lose it.
  - **Convert from session** — input + button → `POST /v1/admin/eval/
    case-from-session`. Result rendered as a dark `<pre>` block with
    a top-right `CopyButton` that copies the YAML to the clipboard.

Pass-rate card flips green when `passed === total`. Fail rows tint red
+ surface the assertion reasons in-table. Spinner is a small CSS
keyframes spinner (no SVG library added).

### Frontend — `CopyButton` (admin-ui local copy)

The v0.8.3 chat-ui `CopyButton`
(`frontend/chat-ui/src/codeblock.tsx`) was a natural fit for the
session-to-case YAML output. **We shipped an admin-ui local copy at
`frontend/admin-ui/src/components/CopyButton.tsx`** rather than reusing
the chat-ui one or lifting it into `@xiaoguai/shared`, because:

1. admin-ui doesn't depend on chat-ui — dragging that dep across is a
   bigger commitment than the 30-line duplication.
2. `@xiaoguai/shared` is types-only today (no React in its `peerDependencies`).
   Adding React would pollute every consumer.

Both files behave identically (1.2 s "Copied!" state, falls through on
clipboard rejection). If either grows behaviour, keep them in sync.

## Deferred

* **SSE / streaming progress.** Synchronous POST + spinner is the
  v0.11.2 contract; suites cap at 100 cases / 60 s so the longest
  in-pane run is bounded. Per-case streaming via
  `tokio::sync::mpsc<EvalResult>` was sketched in the v0.11.0 plan's
  deferred list — pairs naturally with v0.12.0's
  `xiaoguai-runtime` extraction.
* **PG eval history.** Reports are JSON on disk; the console caches
  the latest in `localStorage`. A `pg_eval_reports` table is the
  v0.12.0 candidate per the v0.11.0 plan. Operators wanting an audit
  trail can write reports to disk via `xiaoguai eval run --out
  report.json` in CI and curl them.
* **In-browser YAML editing.** The pane is read-only. Operators edit
  on disk and re-run; pairs with a v0.12.x "case editor" only if real
  feedback says authoring in YAML is a friction point.
* **Real-MCP `EvalAgentBuilder`.** v0.11.0 deferred this; v0.11.2
  uses the same empty-Toolbox `DefaultEvalAgentBuilder`. The trait is
  `Arc<dyn EvalAgentBuilder>` so the slot is ready when v0.11.x ships
  a `McpEvalAgentBuilder` against the v0.9.4 marketplace.
* **Per-tenant eval scoping.** All three routes live under
  `/v1/admin/*` and bypass tenant scoping; the case-from-session
  source reads cross-tenant on the assumption that admin callers see
  every session. Per-tenant scoping pairs with the same
  `/v1/admin/today/tenant/:id` rebuild deferred in v0.11.1.
* **`PgCaseFromSessionSource` testcontainer e2e.** Static unit tests
  cover the projection function + the PG bridge's helpers
  (`flatten_text` / `map_role`). A real-PG run-through pairs with
  v0.11.1's deferred adapter e2e.
* **MCP supervisor live-pickup for eval-time MCP servers.** Out of
  scope — pairs with v0.9.4.1 already shipped.

## Design decisions worth flagging

* **`EvalService` over duplicating the trait constellation.** The
  service struct holds runner + suites_dir + case_source so handlers
  pull one `Arc` from `AppState` instead of three. Mirrors how
  `audit_bridge.rs` collapses reader + verifier into one `PgAuditSink`-
  backed adapter — keeps the surface narrow for the next person
  reading the wiring.
* **`CaseFromSessionSource` trait, PG impl in core crate.** Same
  layering rule the v0.6.4 / v0.6.5 / v0.11.1 plan docs all enforce:
  `xiaoguai-api` doesn't depend on sqlx. The bridge module owns the
  query; the trait owns the wire contract.
* **Suite-too-large + timeout caps return HTTP-shaped errors.** The
  console can render a clear "your suite is too big for the pane"
  message — agents reading the response don't have to parse free-form
  text. 413 + 504 are the HTTP-native answers; we added the two
  `ApiError` variants rather than reaching for 500.
* **`POST` for `case-from-session` even though it's read-only.** The
  body holds the only required parameter (`session_id`), and treating
  this as a write keeps the route distinguishable from idempotent
  reads in the rate-limit / audit middleware (`POST` consumes a
  bucket token; `GET` /v1/admin/today does not). Future-proofs the
  endpoint if we ever record "operator generated case from session X
  at time Y" as an audit row.
* **Per-case YAML returned in full, not paginated.** Cases stay
  small (one session = one case). Streaming the YAML would force the
  console to handle SSE for a single-string response — over-engineered.
* **Default-suite selection on first load.** When the user lands on
  `/eval` with no `selected`, we pick the first suite alphabetically.
  Mirrors the v0.11.1 Today pane's "show the most recent first" — an
  operator hitting the pane sees actionable content immediately, not a
  blank canvas.
* **`localStorage` cache vs. server-side last-run.** Per the v0.11.0
  plan's "reports are artifacts, not first-class data" — caching
  client-side keeps the substrate honest. Multi-operator collaboration
  on a shared "last run" is the v0.12.0 PG history question, not a
  v0.11.2 problem.
* **CopyButton duplication over `@xiaoguai/shared` move.** Documented
  in the Pieces section above. Two 30-line files beat dragging React
  into the types package.
