# v0.9.3 — `ContentBlock::Citation` + chat-ui hover/click

**Status:** done.

## What shipped

The roadmap-locked citation contract gets a first-class home in the
domain model. Before this tag, RAG hits could only land in the chat
transcript as either plain `Text` or a `ToolResult` JSON payload —
both lose the structure the UI needs to render a click-to-source
chip. v0.9.3 adds a fourth `ContentBlock` variant:

```rust
ContentBlock::Citation {
    source_uri: String,    // file://, https://, custom scheme
    span: (u32, u32),      // 1-indexed [start, end] lines; (0,0) = no anchor
    score: f32,            // [0, 1]
    preview: String,       // ~200-400 char chunk text
    collection_id: String, // for "find more from this source"
}
```

This is the **load-bearing schema** for every RAG-aware UI surface
and every future eval grader. From this tag forward, the
type-checker enforces what the v0.9.2 citation contract previously
just asserted in prose.

## Pieces

### Domain types (`xiaoguai-types`)

* New `ContentBlock::Citation` variant. Same `#[serde(tag = "type")]`
  envelope as the other blocks → JSONB round-trips for free; the
  v0.5.1 `messages.content` schema needs no migration.
* Three new unit tests:
  1. Full ser-de round-trip with non-zero span.
  2. Unknown `"type"` tag deserialisation fails (defends against
     silent schema drift).
  3. Zero-span citation serialises as `"span":[0,0]` (the no-anchor
     contract).

### Conversion layer (`xiaoguai-api::convert`)

`domain_to_llm` now has an explicit branch for `Citation`: it
*skips* them. Citations are visible to the UI but invisible to the
LLM on the next turn. Reasoning: the retrieved content already
appeared in a `Text` block (the assistant turn that used it), and
re-injecting `[1] file:///x.md` as text would only confuse the
model. The branch is documented inline so future maintainers don't
"fix" it.

### Frontend wire (`@xiaoguai/shared`)

`ContentBlock` TS union gains the same shape. `span` is `[number,
number]` not a tuple type — TS spreads work without the tuple
narrowing.

### Frontend renderer (`chat-ui`)

* New `src/citations.tsx` — `CitationStrip` + `CitationChip` with
  hover card. Chip text is `[1]`, `[2]`, … in score-descending
  order. Opacity scales with score so weaker hits visually recede.
* `ChatPage.tsx` — `DisplayBubble` gains optional `citations: CitationBlock[]`.
  `messageToBubbles` harvests every Citation block in a message and
  attaches them to the *last assistant text bubble* produced. The
  strip renders inside the same `.bubble.assistant` container.
* Anchored links: when `span !== (0, 0)` we append `#L<start>-L<end>`
  to the source URI. GitHub-style. `file://` ignores the fragment
  (no harm); GitHub blob URLs jump straight to the lines.
* `styles.css` — `.citation-strip` / `.citation-chip` /
  `.citation-card` token-driven so dark mode works without extra
  rules.

## Deferred (v0.9.3.x)

- **Click-jump fidelity for `file://`.** Browsers ignore the
  `#L<n>-L<m>` fragment for `file://` URIs; an IDE handoff (`vscode://
  file/<path>:<line>`, `jetbrains://...`) would be a real
  improvement. Behind a setting because schemes vary by editor.
- **Inline anchor superscripts in the text body.** Today the strip
  hangs below the assistant turn; OpenWebUI's `[1]` inline marker
  inside the text is what the community wants long-term. Needs a
  markdown-rendering hook to interpolate `[n]` after the relevant
  sentence, which is non-trivial for streamed text.
- **"Find more from this collection"** quick action on the hover
  card. Trivial once the v0.11.0 `/library` route lands.
- **Cited-but-empty contract enforcement.** Today `Citation` blocks
  emitted alone (no `Text` partner in the same turn) render fine; we
  should treat that as a model-side bug and either suppress or warn.
  Belongs with the v0.11.0 eval suite.
