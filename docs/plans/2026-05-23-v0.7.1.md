# v0.7.1 — Feishu real reply (OpenAPI + ReactAgent loop)

Status: **Done** — 2026-05-23.

## Goal

Turn the v0.7 Feishu adapter from "acknowledges webhooks and pretends
to reply" into "an actual IM bot": every inbound message runs through
`ReactAgent` and the assistant output is delivered back to the user
via Feishu's `OpenAPI`. Token management (`tenant_access_token` cache)
is the second half of the slice.

## What landed

### `xiaoguai-im-feishu`

- New `api` module:
  - `FeishuClient` trait (object-safe) covering the two endpoints we
    need: `tenant_access_token/internal` and `im/v1/messages`.
  - `HttpFeishuClient` implementing the trait via `reqwest`. Defaults
    to `https://open.feishu.cn`; tests / staging override via
    `with_base_url`.
  - `TokenResponse { token, expire_in_secs }` decoded from Feishu's
    auth response.
  - `TokenCache` — in-memory cache with a configurable refresh
    headroom (60s default). Uses a `tokio::sync::Mutex` so concurrent
    callers single-flight the refresh.
- `ReplySink::Api(Arc<ApiSink>)` variant + `ApiSink { client, token_cache }`
  + `FeishuProvider::with_api_sink(encrypt_key, client, app_id, app_secret)`.
- `ImProvider::reply` now: cache → token → `send_text_message`.

### `xiaoguai-im-gateway`

- `router::run_agent_and_reply(state, msg)` builds a `ReactAgent` from
  `AppState.backend / toolbox / agent_defaults`, runs
  `run_to_completion` on the inbound user message (stateless single
  turn — durable conversation history is deferred), extracts the
  latest assistant text, and dispatches it to `provider.reply`.
- `handle_webhook` returns HTTP 200 immediately and spawns the agent
  task in the background — Feishu's webhook retry policy is unkind to
  slow responders, so we accept-and-async.
- The `GatewayState` and the new `run_agent_and_reply` are public so
  tests can await the pipeline directly.

### Cargo

- `xiaoguai-im-feishu` gains `reqwest`, `chrono`, `tokio` (real deps).
- `xiaoguai-im-gateway` promotes `xiaoguai-agent` + `xiaoguai-llm` from
  dev-dependencies to dependencies; adds `tokio-util` (for
  `CancellationToken`).

### Tests

- 3 new `api::tests::*`: token cache miss/hit/refresh/invalidate.
- 1 new provider-level test exercising the full Api-sink path via a
  fake `FeishuClient` recorder.
- Existing `feishu_route::message_event_is_accepted_and_reply_dispatched`
  updated: the reply text is now the `MockBackend`'s scripted output,
  not an echo of the user's text.

## Deferred

- **Durable conversation history per chat thread.** v0.7.1 is single-
  turn — every inbound message starts a fresh `ReactAgent` with no
  history. Wiring `IncomingMessage.conversation_id` to a persisted
  session (PG or Valkey-backed) is the next slice.
- **Tenant/user mapping.** Feishu's `tenant_key` + `open_id` are not
  resolved to internal tenant/user IDs; the agent runs under
  `AppState.agent_defaults` regardless of caller.
- **Long-task handoff.** Replies above Feishu's per-message size limit
  must be paginated or wrapped in a rich-card; deferred until first
  failure in real traffic.
- **DingTalk + WeCom adapters** implementing `ImProvider`.
- **Interactive cards** (rich-message reply with tool-call viz).
- **Encrypted-payload Feishu apps.** v0.7 + v0.7.1 only handle the
  plaintext-payload variant; AES-encrypted webhook bodies will need a
  decrypt step before signature verification.
