# v1.1.4 ‚Äî HA scaffolding (PG logical repl + Valkey Cluster + nginx)

**Status:** done (2026-05-24). Scaffolding-only slice ‚Äî no crate
code changes.

## Why this slice

Roadmap ¬ß3 v1.0+/v1.1 lists "HA (PG replica + Valkey cluster)" as
the next infra investment after the v1 feature-complete close.
v1.1.4 ships the **scaffolding** ‚Äî compose topology, helm overlay,
runbook ‚Äî without touching any crate code. The application-side
wiring (replica-aware read pool, in-process PG failover, native
Valkey cluster client) lands in subsequent slices so each carries
its own risk surface.

The decision is **locked**:

- **PG: logical replication** (not streaming). Per-table grain +
  the same wire protocol we want for future shard-routing
  experiments. Cost is publication management (one
  `REFRESH PUBLICATION` per primary migration ‚Äî runbook ¬ß2).
- **Valkey: cluster mode** (not sentinel). Horizontal sharding
  capability for free; sentinel would force a re-platform when
  the cache outgrows one keyspace.

Bundling all three artifacts (compose + helm + runbook) under one
tag means an operator who finds the slice can stand the topology
up locally, in k8s, and survive day-2 events without grepping the
codebase.

## Pieces

### 1. `deploy/docker-compose.ha.yml` + `deploy/ha/`

Drop-in alternative to `deploy/docker-compose.yml`. Topology:

- `pg-primary` ‚Äî postgres:16-alpine, `wal_level=logical`,
  `max_wal_senders=10`, `max_replication_slots=10`. Init script
  (`deploy/ha/pg-primary-init.sh`) creates the `xiaoguai_repl`
  replication role + `xiaoguai_pub` publication (`FOR ALL TABLES`)
  + appends pg_hba.conf lines for the replication role.
- `pg-subscriber-a` + `pg-subscriber-b` ‚Äî same image. Init script
  (`deploy/ha/pg-subscriber-init.sh`) waits up to 60s for the
  primary's schema via `pg_dump --schema-only`, applies it
  locally, then creates `SUBSCRIPTION xiaoguai_sub_{a,b}` against
  the primary's publication. Subscribers are read-ready but **not
  yet routed** ‚Äî see Deferred.
- `valkey-cluster-1`..`-6` ‚Äî valkey/valkey:8-alpine, each running
  `--cluster-enabled yes`. AOF on, 5s node timeout.
- `valkey-cluster-init` ‚Äî one-shot that runs
  `valkey-cli --cluster create ‚Ä¶ --cluster-replicas 1`. Idempotent
  on re-`up` (cluster-already-formed exit 1 is swallowed). 3
  primary + 3 replica topology with auto-assigned slot ranges.
- `xiaoguai-core-1` + `xiaoguai-core-2` ‚Äî same Dockerfile as the
  single-node compose. Each gets its own `XIAOGUAI_NODE_ID` so the
  audit chain attributes entries. Both point at `pg-primary` for
  reads + writes (replica reads = v1.1.4.1). Each points at a
  *different* Valkey node (vc-1 vs vc-2) so single-node death
  halves rather than kills cache availability (workaround for the
  deferred cluster-client migration).
- `nginx` ‚Äî nginx:1.27-alpine on `:7600`. Config in
  `deploy/ha/nginx.conf`: `least_conn`, `max_fails=2
  fail_timeout=30s`, SSE-friendly defaults (HTTP/1.1, buffering
  off, 1h read timeout), `proxy_next_upstream` on 502/503/504
  with `tries=2`. Waits for both cores to be `healthy` before
  starting.

Every container has a healthcheck. The compose dependency graph
encodes the bootstrap order (primary ‚Üí subscribers ‚Üí valkey nodes
‚Üí cluster-init ‚Üí cores ‚Üí nginx).

### 2. `docs/runbooks/ha.md`

New runbook covering:

1. ASCII topology diagram (client ‚Üí nginx ‚Üí 2√ó core ‚Üí primary + 2
   subscribers; separate Valkey block).
2. Manual bootstrap order with copy-paste compose commands + the
   one-time `ALTER SUBSCRIPTION ‚Ä¶ REFRESH PUBLICATION` you run
   after cores migrate the primary for the first time.
3. Failover playbooks:
   - **PG primary loss** ‚Äî manual promotion of a subscriber
     (logical repl doesn't auto-promote). DROP SUBSCRIPTION on the
     promoted node, CREATE PUBLICATION, repoint cores via env +
     restart, repoint the other subscriber. RPO ‚â§ replication lag
     (typically < 1s); RTO bounded by core restart (~30s).
   - **Valkey node loss** ‚Äî automatic (cluster replicas
     self-promote within the 5s node timeout). Caveat about
     today's single-node client connection means losing the
     specific dialed node degrades that core's cache until
     restart.
   - **xiaoguai-core instance loss** ‚Äî nginx reroutes
     automatically via `max_fails`/`fail_timeout`.
4. Lag monitoring queries ‚Äî `pg_stat_replication` for PG (with a
   table of acceptable lag per RO endpoint, anchoring the
   v1.1.4.1 design), `cluster info` + `cluster nodes` +
   `info replication` for Valkey.
5. Backup + PITR sketch ‚Äî `pg_basebackup` + `archive_command` to
   S3 + `recovery_target_time` restore flow.
6. Explicit "what today's code is NOT" section pointing each
   reader at the matching Deferred row below.

### 3. `deploy/helm/xiaoguai/values-ha.yaml`

Overlay values layered on top of `values.yaml`. The chart **does
not** install PG or Valkey ‚Äî operators bring their own (CNPG +
spotahome/ot redis-operator). The overlay documents the topology
expectations + tightens probes + enables HPA + pod
anti-affinity + PDB.

Notable keys:

- `replicaCount: 2` + `podAntiAffinity.enabled: true` (host
  anti-affinity) + `podDisruptionBudget.minAvailable: 1`.
- `postgresql.architecture: replication` block ‚Äî pure
  documentation, declares the CNPG Cluster CR shape the operator
  must satisfy (`wal_level: logical`, `publication: xiaoguai_pub`,
  `subscribers: 2`).
- `valkey.architecture: cluster` block ‚Äî same: declares the
  expected RedisCluster CR shape (6 nodes, 1 replica per primary).
- `secrets.databaseRead` left commented ‚Äî uncomment when v1.1.4.1
  ships the `pool_read` config knob.
- Tightened liveness/readiness `failureThreshold` for faster pod
  eviction under load.

## Deferred

The HA story is iterative on purpose. Each item below is a single,
shippable slice with a clear surface ‚Äî they are deliberately *not*
bundled into v1.1.4 because each carries its own risk.

- **v1.1.4.1 ‚Äî Replica-aware read pool routing.** Add
  `XIAOGUAI_DATABASE__READ_URL` config + a second `PgPool` in
  `xiaoguai-storage::AppState`. Route the four
  staleness-tolerant endpoints to it: `GET /v1/admin/today`,
  `GET /v1/admin/sessions`, `GET /v1/admin/audit`,
  `GET /v1/eval/runs`. Lag tolerance table in
  `docs/runbooks/ha.md` ¬ß4.1 already documents the targets. New
  config knob is opt-in (absent ‚áí both pools point at the same
  URL ‚áí today's behaviour).

- **v1.1.4.2 ‚Äî In-process PG primary-loss failover.** Today
  `pg-primary` loss requires the manual playbook in `ha.md` ¬ß3.1
  (promote + edit env + restart cores). Add a
  `XIAOGUAI_HA__FALLBACK_DSN` knob; on `connection_terminated` /
  `cannot_connect_now` errors, retry once against the fallback
  DSN before failing the request. Doesn't promote ‚Äî assumes an
  external mechanism (CNPG operator in k8s; runbook in compose)
  already promoted a subscriber. Tightens RTO from ~30s to
  ~5s in the cases where the operator promoted within the retry
  window.

- **v1.1.4.x ‚Äî Valkey cluster-aware client.** Today the `redis`
  crate uses `redis::Client::open(url)` against a single cluster
  node. `-MOVED` redirects work for slot lookups, but losing the
  specific dialed node won't fail over client-side. Migration to
  `redis::cluster::ClusterClient` is its own slice ‚Äî touches
  `xiaoguai-storage::cache.rs` + every call site that opens
  connections. Sequenced after v1.1.4.1/.2 because the workaround
  in v1.1.4 (each core dials a different node) bounds the blast
  radius.

- **Per-tenant publication scoping.** Today `xiaoguai_pub` is
  `FOR ALL TABLES`. A future workload might want
  `token_usage`-off-the-wire to save WAL bandwidth, or per-tenant
  subscribers for data residency. Mechanism is a publication
  change + matching subscription `REFRESH`; no code change.
  Deferred until a real use case shows up.

- **Multi-region / geo-distributed PG.** Logical repl is region-
  capable but the latency profile changes the failover playbook
  (RPO grows, conflict resolution becomes a concern). Out of
  scope for v1.1.4 ‚Äî single-region only.

## Design decisions worth flagging

- **Compose port `:7600` for the HA stack, not `:8080`.** The
  baseline single-node compose stays on `:8080`; HA exposes on
  `:7600` so an operator can run both side-by-side on the same
  host while practising failover. Avoids the "I forgot which
  stack I'm hitting" foot-gun.

- **Bootstrap subscribers via `pg_dump --schema-only` retry loop,
  not a baked image.** The subscribers don't know the schema at
  build time ‚Äî it's owned by `xiaoguai-storage::migrations` which
  runs from the core. The init script polls the primary up to
  60s and tolerates an empty schema (the subscription is created
  empty; the operator runs `ALTER SUBSCRIPTION ‚Ä¶ REFRESH
  PUBLICATION` once the cores have migrated). Trades startup
  noise for not having two sources of truth on the DDL.

- **`FOR ALL TABLES` not per-table.** Same reason as the v0.6
  RLS-aware migrations: enumerating tables in a config file
  guarantees the config drifts. The cost is one
  `REFRESH PUBLICATION` per migration; the runbook bakes this
  into the release pipeline guidance.

- **nginx for L7, not HAProxy.** Both work; nginx wins on SSE
  defaults (we ship streaming responses, and nginx's
  `proxy_buffering off` + `proxy_http_version 1.1` is one less
  thing to tune). HAProxy users can swap in their own config ‚Äî
  the only contract is "round-robin or least-conn between the
  two cores on :8080, retry on 5xx".

- **Each core dials a different Valkey node by default.** Until
  the cluster-client migration lands, the workaround for "lose
  the dialed node ‚áí lose cache for that core" is "spread the
  dial across nodes so single-node death halves rather than
  kills cache availability." Documented in the runbook so it
  isn't surprising during incident review.

- **Helm chart doesn't install PG/Valkey.** values-ha.yaml
  documents the expected CR shapes but defers to the operator's
  CNPG + redis-operator install. Reason: production k8s shops
  already have PG/Redis operators they trust; bundling sub-charts
  would force a re-deploy of those layers. Same call as the
  baseline `values.yaml` already makes.

- **`xiaoguai-core smoke` as the healthcheck.** Existing binary
  subcommand that dials every dependency and exits non-zero on
  failure. Reuses the v0.6 smoke command rather than inventing
  an HTTP `/healthz`-only probe, so the healthcheck catches
  "core booted but can't reach PG/Valkey" as well as "process
  not running".

- **No crate code changes.** This is non-negotiable for the
  slice. Application-side HA wiring is intentionally factored
  into v1.1.4.1 and v1.1.4.2 because each touches different
  files + introduces different failure modes. v1.1.4 is the
  topology contract; the application learns to use it later.
