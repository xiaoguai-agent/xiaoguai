# v0.12.x.1 — per-tenant webhook tokens + `CompositeExecutor` + admin-ui Scheduler pane

**Status:** done (2026-05-24).

## Why this slice

Three small scheduler closers, bundled because they all touch the same
`scheduler_bridge.rs` adapter file and the same `AppState` and admin
router.

| Piece | Hole it closes (per HANDOFF §1) |
|---|---|
| Per-tenant webhook tokens | v0.12.0 left the webhook route admin-bearer-gated; external integrators (GitHub push, Slack events) needed an admin token. This patch adds a sibling public route at `/v1/scheduler/webhooks/:route_id` (NOT under `/admin`) that authenticates per-request via an opaque token in the `X-Xiaoguai-Token` header. |
| `CompositeExecutor` | v0.12.2 shipped `RagReindexExecutor` but never wired it; production used `RuntimeJobExecutor` for every payload regardless of `payload.kind`. This patch lands the one-arm dispatcher that routes by `kind` and falls through to the runtime default. |
| Admin-ui Scheduler pane | v0.12.1 shipped the NL-compile + upsert backend endpoints but no UI. This patch adds the pane (between Today and Eval), a Jobs table with "Run now", a Tokens subsection (list + create + revoke), and a "Create from description" tab that wraps compile → preview → save. |

## Pieces

### Patch 1 — per-tenant webhook tokens

#### Storage

New migration `crates/xiaoguai-storage/migrations/0008_scheduler_webhook_tokens.sql`:

```sql
CREATE TABLE scheduler_webhook_tokens (
    token        TEXT PRIMARY KEY,
    tenant_id    TEXT NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    route_id     TEXT NOT NULL,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_used_at TIMESTAMPTZ
);
```

RLS-enabled with the same `current_setting('app.current_tenant_id')`
pattern as the rest of the schema. Indexes on `tenant_id` and
`route_id` for the admin list paths.

#### API traits

Two new trait pairs in `crates/xiaoguai-api/src/scheduler.rs`:

* `WebhookTokenValidator::validate(token, route_id) -> Option<tenant_id>`
  — returns `Ok(None)` on mismatch; the handler maps to 401.
* `WebhookTokenAdmin` — `create / list / revoke` for the admin CRUD.

Plus the small read-only `WebhookTokenRecord` DTO used on the wire.

In-memory test helpers (`StaticWebhookTokenValidator`,
`InMemoryWebhookTokenAdmin`) ship alongside so route tests don't need a
DB.

#### HTTP routes

| Method | Path | Auth | Status |
|---|---|---|---|
| POST | `/v1/scheduler/webhooks/:route_id` | `X-Xiaoguai-Token` header | 503 / 401 / 404 / 202 |
| POST | `/v1/admin/scheduler/tokens` | admin bearer | 201 |
| GET | `/v1/admin/scheduler/tokens?tenant_id=` | admin bearer | 200 |
| DELETE | `/v1/admin/scheduler/tokens/:token` | admin bearer | 204 / 404 |

The public route sits OUTSIDE the bearer/Casbin middleware stack on
purpose (the whole point is that external integrators don't have an
admin bearer). Rate-limit middleware still applies so a flood of bad
tokens can't saturate the runner.

The existing admin route at `/v1/admin/scheduler/webhooks/:route_id`
stays put — internal callers keep the no-token shortcut.

#### Production impls

`PgWebhookTokenValidator` + `PgWebhookTokenAdmin` in
`crates/xiaoguai-core/src/scheduler_bridge.rs`. The validator
best-effort updates `last_used_at` on every successful match; update
failures log + continue (the audit row is the source of truth, not the
timestamp).

Tokens are generated as `uuid::Uuid::new_v4().simple()` — 32 hex chars,
122 bits of entropy. Captured exactly once on the create-response;
rotation is "delete + create".

### Patch 2 — `CompositeExecutor`

New `crates/xiaoguai-scheduler/src/composite_executor.rs`:

```rust
pub struct CompositeExecutor {
    default: Arc<dyn JobExecutor>,
    by_kind: HashMap<String, Arc<dyn JobExecutor>>,
}
```

Dispatch:

```rust
fn pick(&self, job: &ScheduledJob) -> &Arc<dyn JobExecutor> {
    if let Some(kind) = job.payload.get("kind").and_then(|v| v.as_str()) {
        if let Some(found) = self.by_kind.get(kind) { return found; }
    }
    &self.default
}
```

Wired in `xiaoguai-core/src/main.rs::run_serve` inside the
`scheduler.enabled` block:

```rust
let executor: Arc<dyn JobExecutor> = Arc::new(
    CompositeExecutor::new(runtime_executor)
        .register("rag_reindex", rag_executor),
);
```

The default arm preserves v0.12.0 behaviour exactly: every existing
scheduled job (no `kind` field in payload) keeps running through
`RuntimeJobExecutor`. The `RagReindexExecutor` now actually executes
when an operator creates a job with `payload.kind == "rag_reindex"`.

`RagReindexExecutor::new` had a `#[allow(dead_code)]` from v0.12.2 (the
deferred-wiring note); removed now that production calls it.

### Patch 3 — admin-ui Scheduler pane

`frontend/admin-ui/src/panes/Scheduler.tsx` — new pane between Today and
Eval in the sidebar (App.tsx route + nav order both updated).

Two tabs:

* **Jobs** — table with `id` / `name` / `trigger` / `enabled` /
  `last_fire_at` / `next_fire_at` / Run-now button. Each row's
  "Run now" hits `POST /v1/admin/scheduler/jobs/:id/fire-now` (new in
  this patch) which calls `JobRunner::fire_now`.
  
  Underneath is a "Webhook tokens" subsection: filter-by-tenant text
  input, create-token form (tenant_id + route_id → mint button), and
  the token table with Revoke buttons. Freshly-minted tokens display
  in a "capture now" banner because they aren't shown again.

* **Create from description** — textarea + optional tenant_id input →
  Compile button hits `POST /v1/admin/scheduler/jobs/compile` (already
  shipped in v0.12.1) → renders the rationale + pretty-printed JSON →
  Save button hits `POST /v1/admin/scheduler/jobs` (also v0.12.1).

New `@xiaoguai/shared` types: `ScheduledJobSummary`, `WebhookToken`,
`CompileScheduledJobRequest`, `CompileScheduledJobResponse`. Plus the
matching client methods (`listScheduledJobs`, `fireScheduledJob`,
`compileScheduledJob`, `upsertScheduledJob`, `listWebhookTokens`,
`createWebhookToken`, `revokeWebhookToken`).

#### Backend additions for the pane

New trait `ScheduledJobsReader` in `xiaoguai-api/src/scheduler.rs`:

```rust
#[async_trait]
pub trait ScheduledJobsReader: Send + Sync {
    async fn list(&self, limit: i64) -> Result<Vec<ScheduledJobSummary>, ScheduledJobsReadError>;
    async fn fire_now(&self, job_id: &str) -> Result<(), ScheduledJobsReadError>;
}
```

Production impl `PgScheduledJobsReader` in `scheduler_bridge.rs` wraps
the live `PgJobRepository` + the live `Arc<JobRunner>` (so `fire_now`
goes through the real runner's `fire_now` API). `StaticScheduledJobsReader`
test helper ships alongside.

`list` cheats slightly for v0.12.x.1: it calls `list_due` with
`now + 10 years` as the upper bound so every enabled job shows up. A
proper `list_all` repo method (covering disabled rows too) is deferred
— see Deferred below.

## Deferred

* **`list_all` on `JobRepository`.** The admin-ui Jobs table needs to
  show disabled jobs too. v0.12.x.1 cheats by passing `now + 10 years`
  to `list_due` so every enabled row appears; disabled rows are
  invisible. Add a `list_all(limit, offset)` method when an operator
  reports it.
* **Pagination on the Jobs table.** Hardcoded `limit: 200` for now.
  When a deployment hits that, add cursor/offset.
* **CSS polish for the new pane.** The pane works against the existing
  CSS classes; no custom styles shipped. Browser eyeball pass owed.
* **Per-tenant rate-limit override on the public webhook.** Today every
  external token hits the global `RateLimiter` (20 r/s, 40 burst). A
  per-tenant override (so a GitHub-push-storm doesn't starve other
  tenants) is a v1.1+ thing.
* **Token rotation primitive.** Today rotation is "create then
  revoke". A `rotate_token(token) -> new_token` admin endpoint that
  hands back the new token + keeps the old one valid for a grace
  window would be nicer for zero-downtime rotation on a hot route.
* **Token scoping by IP or method.** The token is bound to
  `(tenant, route_id)` only; ip-allowlist or HMAC body signatures are
  v1.1+ work.

## Design decisions worth flagging

* **Public route lives outside the bearer/Casbin layer stack.**
  Mounting it inside `v1` and special-casing the auth middleware would
  have meant two failure modes (middleware fires for a non-bearer
  request, route handler also rejects). One handler, one auth seam.
  The price is the `public_v1` router branch in `routes/mod.rs`.
* **Validator returns `Option<tenant_id>`, not a typed error.** Same
  shape as `WebhookPusher::push` — `Ok(0)` means "no jobs bound",
  `Ok(None)` means "token doesn't match". Lets the handler map 401 vs
  500 without inspecting error message strings.
* **Token captured exactly once.** The create-response is the only
  place the token appears in plaintext. Subsequent list responses also
  include the token (for the admin to find a leaked one) — a future
  hardening would mask it in list and require an explicit `?reveal=1`.
* **`CompositeExecutor` registers by string `kind`.** Considered a
  typed enum but the dispatch arm is owned by the binary, not the
  scheduler crate. Strings + a stable JSON schema beat passing a Rust
  enum across the trait boundary.
* **`RagReindexExecutor` wired with `InMemoryRagClient` by default.**
  No `[rag]` config block exists yet; a deployment that wants real
  re-indexes pre-populates an `R2RClient` in the bridge. The slot is
  there; the production-grade wiring lives in v1.1+ when RAG config
  lands. This preserves "tests pass" without claiming we ship a real
  vector store today.
* **`PgScheduledJobsReader` cheats on `list`.** Reusing `list_due`
  with `now + 10 years` ducks adding a new `list_all` trait method
  this slice. Trade-off explicitly noted in Deferred; the v0.12.x.1
  audience for the pane is operator-with-handful-of-jobs, not the
  100-job production deployment. Add `list_all` when a real user has
  100 jobs.
