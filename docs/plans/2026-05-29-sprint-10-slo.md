## 1. Context

Sprint-9 (v1.7.0) shipped the planner/worker/critic triangle. Sprint-10 operationalises **DEC-022**: turn the four Google SRE golden signals (latency, traffic, errors, saturation) into **first-class SLO contracts** with burn-rate alerts. The architecture detail landed in design-repo PR #5 (`LLD-OBS-001`); this plan executes the implementation.

The infrastructure is already there: `crates/xiaoguai-observability` has 15 Prometheus metrics; `deploy/helm/xiaoguai-observability/templates/prometheus-rules.yaml` ships 17 alert groups including Wave-3's three SLO groups (`wave3-slo-recording-rules`, `wave3-slo-burn-rate-alerts`, `wave3-slo-meta`) as the prototype. Sprint-10 generalises the same pattern to the API tier and publishes the SLO table as a contract artefact.

R.E.S.T axis: primarily **Reliability** (SLO breaches page operators before users feel them) and **Traceability** (the SLO declaration is auditable, the burn-rate gauge is observable).

The original 2026-05-30 roadmap allotted ~6.5 dev-days; S10-7 (harness §16 frame) was already absorbed into Step 1 PR #5, so this sprint shrinks to **~6 dev-days**.

## 4. Cross-sprint risks (sprint-10-specific)

| Risk | Mitigation |
|---|---|
| `slo.md` YAML schema drift between S10-1 author and S10-3 reader | Define the schema as a Rust serde struct in S10-3 first; S10-1 writes against that struct's `Deserialize` impl. Schema becomes the source of truth, not the prose. |
| Wave-3 `wave3-slo-*` rule names collide with new `slo-api-tier-*` if alert routing is incautious | `team` label disambiguates; runbook page-chain section makes it explicit which on-call gets which group. Codified in S10-1. |
| `xiaoguai_slo_burn_rate` gauge cardinality explosion when many tenants override | Tenant label only present when override is active (LLD §4.4); base cardinality 80 series, per-override tenant adds 4 × 2 = 8. At 100 overriding tenants = 880 series. Within Prometheus comfort. Document the budget in S10-3. |
| `promtool test rules` not present in CI image | Add to the helm-chart-validate CI job (small Dockerfile delta); S10-2 ships the test fixtures inline so the rule groups are testable even before infra wires up. |
| Alertmanager webhook delivery is best-effort — missed alert means no `WatchEvent` | `xiaoguai-watch` already has a polling fallback for `SqlSource`; S10-5 documents that the Alertmanager sink is augmentation, not the sole detection path. Slow-burn (6 h) alerts auto-re-fire so a missed delivery surfaces within 5–15 min. |
| Default SLO values too tight on Ollama-only deployments (no GPU) | Documented escape valve: operator sets `tenant_settings.settings->>'slo_latency_p95_ms'` per the §4.7 override mechanism. Runbook S10-6 includes "latency SLO too tight" as one of the 8 entries. |

## 5. Out of scope (sprint-10)

- **Renaming `xiaoguai_http_request_duration_seconds`** to align with DEC-022's prose. LLD-OBS-001 footnote queues this as an HLD touch-up; the rename invalidates 17 alert groups + 7 dashboards and is its own sprint.
- **Migrating wave-3 SLO alert groups** to the new `slo-api-tier-*` naming. Parallel coexistence per DEC-LLD-OBS-004.
- **Per-pane SLO** (admin-ui has its own surfaces — Today timeline freshness, Audit chain verify latency). Reserved for sprint-11+; sprint-10 only covers API surfaces.
- **Multi-tenant SLO dashboards as separate Grafana folders.** One overview dashboard with a tenant filter is enough; per-tenant materialised dashboards are deferred.
- **SLO-driven autoscaling.** The gauge could feed an HPA, but DEC-022 explicitly scopes itself to alerting, not control plane.
- **Auto-calibration of SLO thresholds from session-7 production data.** Reviewer sign-off in sprint-8 explicitly used DEC-022 defaults as-is.
