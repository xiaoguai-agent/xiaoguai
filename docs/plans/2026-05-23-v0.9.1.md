# v0.9.1 — MCP server publish

**Status:** done.

## What shipped

Before this tag, xiaoguai was a one-way MCP citizen: it could consume
tools from external MCP servers (`StdioMcpClient` / `HttpMcpClient`),
but couldn't *be* a server. Dify v1.6 / n8n MCP Server Trigger /
Flowise v1.8 all ship two-way MCP — without it xiaoguai is a leaf in
the agent ecosystem.

v0.9.1 mounts `/v1/mcp/serve` (Streamable HTTP) on the xiaoguai-api
router. External agents speak MCP to it; the handler dispatches every
`tools/list` / `tools/call` back into the **same `Toolbox`** the
internal agent loop uses. So whatever tools an internal agent sees,
an external agent sees verbatim.

Publishing is **opt-in** — `AppState.mcp_publish_enabled = false` by
default, flipped by env var `XIAOGUAI_MCP__PUBLISH=true` in
`xiaoguai-core`. Exposing tools to the network is an explicit
operator decision; we don't accidentally publish.

## Pieces

### `xiaoguai-api::mcp_serve::XiaoguaiMcpServer`

`crates/xiaoguai-api/src/mcp_serve.rs`. A `ServerHandler` view over
`Arc<Toolbox>`. Implements:

* `list_tools` — sorts the `Toolbox` by name for stable external
  output, builds `rmcp::model::Tool` per entry from the existing
  `ToolDescriptor` (name + description + JSON-schema parameters).
* `call_tool` — looks up the tool in the `Toolbox`, dispatches to its
  registered `Arc<dyn McpClient>`, converts the resulting
  `xiaoguai_mcp::ToolResult` back to `rmcp::CallToolResult`. Unknown
  tool name → `invalid_params`. Underlying client error →
  `internal_error`. Tool returning `is_error: true` →
  `CallToolResult::error(...)`.
* `get_info` — advertises `tools` capability with `list_changed:
  false`; server name "xiaoguai" + crate version.

Images / resources in `ToolResult.blocks` are filtered today (text
only). Wider block types ride with `ContentBlock::Citation` in
v0.9.3 and beyond.

### Router wiring

`crates/xiaoguai-api/src/routes/mod.rs`. The MCP serve route sits
**outside** the v1 layer stack on purpose — bearer / Casbin / per-
tenant rate limit are the wrong default for an MCP server (external
agents authenticate via the MCP transport's own auth header). When
`state.mcp_publish_enabled = false` nothing is mounted; `GET
/v1/mcp/serve` simply 404s.

### AppState

New field `mcp_publish_enabled: bool`. Default `false` everywhere
(all 7 test constructors updated to `false`). `xiaoguai-core` reads
the env var.

### Dependency

`xiaoguai-api/Cargo.toml`: rmcp features bumped to include `server`
+ `transport-streamable-http-server` +
`transport-streamable-http-server-session`.

## Tests

`tests/mcp_serve_e2e.rs`:

1. `published_server_exposes_toolbox_tools` — full router spin-up,
   `HttpMcpClient` connects via TCP, asserts sorted `list_tools`
   output matches the `Toolbox` seed.
2. `published_call_tool_round_trips_through_underlying_client` — call
   `echo` on the published surface, assert text dispatched through
   the registered `EchoBackend` returns verbatim.
3. `unknown_tool_returns_error_response` — `call_tool("does_not_exist")`
   surfaces an error containing "unknown tool".
4. `publishing_disabled_means_endpoint_404s` — with
   `mcp_publish_enabled = false`, `HttpMcpClient::connect` fails
   within 5 s (no hang).

Plus 2 inline unit tests covering `Toolbox`→`ServerHandler` ordering
and `JsonValue → JsonObject` defensive conversion.

## How to enable in production

```bash
XIAOGUAI_MCP__PUBLISH=true xiaoguai-core serve
```

External clients connect via Streamable HTTP at
`https://<host>/v1/mcp/serve`. Auth: today none enforced at the MCP
layer (the route bypasses the v1 bearer middleware). A follow-up
should add either per-client tokens or reuse the bearer flow — see
deferred list below.

## Deferred

- **Auth on the published endpoint** — currently anonymous. Two
  paths: (a) require a bearer header validated against a separate
  `mcp_publish_tokens` table; (b) reuse the existing
  `TokenValidator` and gate behind the same middleware as `/v1/**`.
  (b) is simpler but ties MCP auth to JWT, which most MCP SaaS
  servers don't expect.
- **Per-tenant tool filtering** — today the published `Toolbox` is
  the global `AppState.toolbox`. Per-caller filtering is the natural
  pair with the v0.7.3 `im_identities` mapping (and would let us
  publish "this user's Obsidian tools" without leaking the rest).
- **`tools/list` change notifications** — `list_changed: false` for
  now. When v0.10.x scheduler adds dynamic tool registration we'll
  need to flip this and emit notifications.
- **Images / resource blocks** — text-only passthrough today; the
  full block mapping lands with v0.9.3's `ContentBlock::Citation`
  rework.
