# v0.9.4 — MCP Marketplace

**Status:** done.

## What shipped

Closes the v0.9.x P0 band. xiaoguai now has a curated MCP server
catalog with one-click install — no marketplace backend, no remote
fetch; the catalog is a single JSON file baked into the binary that
operators audit in-repo.

Roadmap principle: **"Skill of the platform is in curation, not in
hosting."** LobeHub's MCP marketplace is ~10k entries; we explicitly
don't compete on size. The eight entries we ship are the ones the
community signals as table-stakes — filesystem, github, postgres,
sqlite, obsidian, slack, fetch, memory.

## Pieces

### Backend

* `crates/xiaoguai-api/src/marketplace.rs`:
  - `MarketplaceEntry` schema (slug + name + description + category +
    transport + version + command/args/endpoint + env_keys + source_url).
  - `list_marketplace()` handler returns the parsed catalog.
  - `install_from_marketplace()` looks up the slug, materialises an
    `McpServer`, and calls `McpServerRepository::create`. Optional
    `tenant_id` in the request body for per-tenant installs.
  - `OnceLock` caches the parsed catalog — JSON parse failures are
    `expect`'d at first use to fail loud on a malformed ship.
  - Catalog file: `catalog/mcp_marketplace.json` (8 entries, version 1).
* `crates/xiaoguai-api/src/routes/mod.rs`: two new routes —
  `GET /v1/mcp/marketplace` and `POST /v1/mcp/marketplace/install`.

### Frontend

* `@xiaoguai/shared`: `MarketplaceEntry`, `MarketplaceResponse`,
  `InstallMarketplaceRequest/Response` TS types +
  `client.listMarketplace()` / `client.installMarketplace(req)`.
* `admin-ui`:
  - New `panes/Marketplace.tsx` — fetches the catalog, groups by
    category, renders cards with Install button. Idle / installing /
    installed / error per-card status.
  - `App.tsx` — Marketplace is the new default landing nav target
    (replaces "MCP Servers" as the initial route).
  - CSS: `.marketplace-grid` (auto-fill 280px columns) + card
    layout. Token-driven so dark mode works without extra rules.

## Tests

`crates/xiaoguai-api/tests/marketplace_e2e.rs` — 5 cases:

1. `list_marketplace_returns_catalog` — 200, version ≥ 1,
   includes `filesystem` + `github`.
2. `install_writes_row_to_repo` — POST `{slug: "filesystem"}` writes
   an `mcp_servers` row via the in-memory test repo; row has the
   expected name + stdio transport + null tenant.
3. `install_404s_for_unknown_slug` — POST `{slug: "totally-fake"}`
   returns 404.
4. `install_503s_when_repo_not_wired` — when
   `AppState.mcp_servers = None`, install returns 503.
5. `install_respects_tenant_id_when_supplied` — POST with
   `{tenant_id: "ten_a"}` writes a tenant-scoped row.

Inline unit tests (in `marketplace.rs`):
- Catalog parses.
- Every entry has a recognised transport + matching command/endpoint.
- Slugs are unique.

## Deferred

- **Wire `McpSupervisor` to start an installed entry immediately.**
  Today install writes the DB row; the supervisor only picks it up on
  next process restart. Pairs naturally with v0.9.4.1.
- **Live env-var prompt at install time.** GitHub / Postgres / Slack
  entries declare required env keys but the UI just lists them — a
  modal with one input per key + writing to a secret store would be
  the polished form.
- **Catalog auto-updates.** Today the catalog is baked into the
  binary. A signed remote-fetch path (verify signature against a
  known maintainer key) is the natural next step but adds trust-chain
  complexity that doesn't fit v0.9.x.
- **Marketplace publish API.** We curate; we don't host. Operators
  who want to add private entries either patch the JSON in their
  fork or rely on the existing `xiaoguai mcp register` CLI path.
