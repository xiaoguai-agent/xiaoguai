# v0.9.0 — MCP Streamable-HTTP transport

**Status:** done.

## What shipped

Before this tag, `xiaoguai-mcp` only spoke stdio (rmcp's
`TokioChildProcess`). That meant the supervisor could host community
MCP servers shipped as npm packages or local binaries but **not** the
~80% of the ecosystem that exposes HTTP endpoints —
`OpenWebUI` tool servers, the official `GitHub` MCP server, Continue's
hosted bundles, and most marketplace entries on `LobeHub`.

v0.9.0 adds a second transport: rmcp's **Streamable HTTP** (the
current MCP spec, which consolidates the legacy standalone-SSE
variant into one endpoint that POSTs JSON-RPC requests and SSE-streams
responses). One transport now covers both the modern spec and SSE-
streamed-response legacy servers.

## Pieces

### `xiaoguai-mcp::http::HttpMcpClient`

`crates/xiaoguai-mcp/src/http.rs`. Constructed via
`HttpClientConfig::new(endpoint)` with builder-style `.with_auth(...)`
and `.with_header(name, value)`. Implements the same `McpClient`
trait as `StdioMcpClient` — supervisor / api / agent code consumes
both via `Arc<dyn McpClient>` and never branches at call sites.

Defaults inherited from rmcp:
- Exponential-backoff retry on SSE reconnect.
- `allow_stateless = true` (works with stateless servers).
- `reinit_on_expired_session = true` (one-shot recovery on HTTP 404).

Header validation happens client-side before any network I/O —
CRLF-injected names are rejected with `McpError::InvalidArgument`,
never sent to reqwest.

### Workspace dep changes

`Cargo.toml`: added rmcp features
`transport-streamable-http-client`,
`transport-streamable-http-client-reqwest`, `reqwest` (rustls TLS).

`xiaoguai-mcp/Cargo.toml`:
- Production: pulls `http` 1.x for `HeaderName/HeaderValue`.
- Dev-only: rmcp `server` + `transport-streamable-http-server` +
  `transport-streamable-http-server-session`, axum, tower,
  tokio-util — used solely by the in-process integration test below.

### Tests

`tests/http_client_e2e.rs` boots an in-process `StreamableHttpService`
behind axum on `127.0.0.1:0` with an `EchoServer` `ServerHandler`,
then exercises `HttpMcpClient`:

1. `connect_lists_tools_and_calls_echo` — full handshake →
   `list_tools` (asserts schema flows through) → `call_tool`
   (asserts text + block count).
2. `call_tool_rejects_non_object_arguments` — array args rejected
   client-side without hitting the wire.
3. `custom_headers_flow_through_to_request` — builder accepts
   well-formed auth + per-tenant headers; handshake still succeeds.
4. `malformed_custom_header_name_is_rejected_before_connect` — CRLF
   injection refused at config build, no server contact.

`src/http.rs` unit tests cover the builder + the header-name
validation predicate directly.

## Acceptance

`cargo test --workspace`: **220 passed / 0 failed / 58 ignored**
(+6 over v0.6.4). `cargo clippy --workspace --all-targets -- -D
warnings` clean. `cargo fmt --check` clean.

Manual smoke: pointing `HttpMcpClient` at the official `GitHub` MCP
server (`https://api.githubcopilot.com/mcp/`) with a PAT in the
`auth_header` value completes the handshake and lists ~80 tools.

## Deferred (v0.9.0.x)

- **Wiring `HttpMcpClient` into `McpSupervisor::start_from_db`** —
  today `mcp_servers` rows with `transport = "http"` or `"sse"` will
  parse but the supervisor's bootstrap code in `xiaoguai-core` still
  only knows how to spawn stdio. Mechanical change; pairs naturally
  with v0.9.4 (the marketplace).
- **OAuth flow for MCP servers that need it.** Continue.dev's MCP
  layer handles auto-OAuth handshake; we accept a static bearer
  today. Sufficient for the v0.9.x marketplace which only needs PAT-
  style auth.
- **Cgroup / seccomp / netns sandbox.** Still in the MCP backlog
  (pre-existing). Less urgent for HTTP transport (the supervisor
  doesn't spawn a child process) but still nice for stdio.
- **Health-check pings.** Same as above; deferred.
