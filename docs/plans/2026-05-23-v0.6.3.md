# v0.6.3 — Admin endpoints + sessions listing + rate-limit

Status: **Done** — 2026-05-23.

## Goal

Close the API gaps that the admin-ui placeholder panes and chat-ui
sidebar point at, and add the simplest credible defence against a
single tenant runaway-looping the service.

## What landed

### `xiaoguai-api`

- New `routes::admin` module with `GET /v1/admin/tenants` (paginated
  list). `AppState.tenants: Option<Arc<dyn TenantRepository>>` carries
  the repo handle; when `None` the endpoint returns 500 (operator
  mis-wiring rather than a missing feature).
- New handler `sessions::list_sessions` mounted on `GET /v1/sessions`
  alongside the existing `POST /v1/sessions` (now resolved by axum's
  method-router on the same path). Accepts `user_id`, `limit`,
  `offset`; falls back to `Claims.sub` when `user_id` is absent and
  claims are present.
- New `rate_limit` module: in-memory per-tenant token bucket.
  `RateLimiter::new(rate_per_sec, burst)` plus the axum middleware
  `rate_limit(limiter, req, next)`. Layer order in `router()` is now
  `handler → rate_limit → rbac → require_bearer` (innermost first),
  so the limiter sees a populated `Claims` extension. Unauthed
  requests (no `Claims`) bypass the limiter — the bucket is keyed on
  tenant.

### `xiaoguai-core`

- `run_serve` constructs a `PgTenantRepository` and a
  `RateLimiter::new(20.0, 40.0)` and threads both into `AppState`.
  These defaults are baked in for now; a config knob is a future
  follow-up.

### Tests

- 4 new `rate_limit::tests::*` unit tests covering: burst budget,
  refill rate, per-tenant isolation, cap at burst on long idle.
- 6 new integration tests in `admin_and_listing.rs`:
  - `GET /v1/sessions?user_id=...` returns only that user's rows.
  - `GET /v1/sessions` with no `user_id` and no claims → 400.
  - `GET /v1/admin/tenants` returns the seeded list.
  - `GET /v1/admin/tenants` when `tenants` is `None` → 500.
  - 429 fires when the bucket is drained.
  - Middleware is bypassed when no `Claims` are present.

## Deferred

- **`GET /v1/admin/audit`** — needs a PG-backed audit store first.
  `xiaoguai-audit` currently only computes HMAC chains; persistence
  is its own slice.
- **Cluster-wide rate-limiting** via Valkey — the v0.6.3 limiter is
  in-memory and per-process. Multi-replica deployments need a shared
  store.
- **Config-driven rate-limit knobs** — `rate_per_sec` and `burst` are
  hard-coded in `xiaoguai-core` for now.
- **`GET /v1/sessions` without `user_id` but with claims auto-uses
  `claims.sub`** — already done; future work might list all of a
  tenant's sessions when called by an admin role.
- **Admin write endpoints** (create/suspend/archive tenant) — read-only
  for now.
