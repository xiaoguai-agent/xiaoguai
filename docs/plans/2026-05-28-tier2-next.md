## D.1 — Agent-authored skills (HotL-gated)

### D.1.1 Context

xiaoguai already has a *human-authored* skill-pack marketplace
(`xiaoguai skills install`, `xiaoguai skills list`). The pi/Hermes
roadmap wants the agent itself to author new skill manifests at runtime
— with the HotL gate enforcing that a human approves before any
agent-authored skill becomes loadable. This is the
"data-driven evolution" principle from
`xiaoguai-agent-design/docs/harness-engineering.md` §5 made literal.

### D.1.2 Success criteria

1. A new tool `propose_skill` is registered with the agent loop. Its
   schema accepts `{name, description, version, system_prompt,
   tool_allowlist[], proposed_by, reason}`.
2. When the agent calls `propose_skill`, the HotL gate (#61) is consulted
   with bucket `skill_author` and the proposed manifest as context. On
   `Deny` the proposal is dropped and the reason returned to the agent.
   On `Allow` (or `Escalate → approved`) the manifest is persisted as
   `pending` in a new `skill_proposals` Postgres table.
3. A new HTTP endpoint `POST /v1/skills/proposals/:id/approve` (requires
   tenant-admin role) flips a proposal from `pending` → `installed` and
   writes the manifest to `~/.xiaoguai/skills/` so `xiaoguai skills
   reload` picks it up.
4. Integration test: a mock-backend agent drafts a skill, gate denies →
   agent retries with a smaller `tool_allowlist`, gate allows → admin
   approves → `xiaoguai skills list` shows the new skill.
5. Audit log shows three rows: `skill.propose`, `skill.hotl_gate`,
   `skill.approve`.
6. The new tool is **off by default** — operators must explicitly enable
   it per tenant via `xiaoguai tenant config set
   allow_skill_authoring=true`.

### D.1.4 Step-by-step

#### Step D.1.4.1 — Schema design

Add migration `20260528_skill_proposals.sql`:

```sql
CREATE TABLE skill_proposals (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  proposed_by     TEXT NOT NULL,        -- agent identity
  name            TEXT NOT NULL,
  description     TEXT,
  version         TEXT NOT NULL,
  manifest_json   JSONB NOT NULL,       -- full skill manifest
  status          TEXT NOT NULL CHECK (status IN ('pending','approved','rejected','installed')),
  reason          TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  decided_at      TIMESTAMPTZ,
  decided_by      TEXT
);
CREATE INDEX idx_skill_proposals_tenant_status ON skill_proposals(tenant_id, status);
```

### D.1.5 Risks (specific to D.1)

| Risk | Mitigation |
|---|---|
| Skill schema not yet stable across implementations | Lock to the current marketplace schema; agent-authored skills are a strict subset (no native code, no MCP server URLs not on a tenant allow-list) |
| Agent generates skills that essentially `eval` user input | The `tool_allowlist` in the manifest is whitelist-only; manifest must reference existing tools by name, not declare new MCP servers |
| Spam: agent proposes 1000 skills | HotL bucket `skill_author` budget capped at 5 proposals/tenant/day in default policy |
| Tenant admin doesn't notice pending proposals | Notification via the existing `xiaoguai watch` system — new event type `skill_proposed` |

### D.1.6 Out of scope (D.1)

- Multi-tenant publishing of skills to the global marketplace.
- Versioned skill upgrades (each proposal is a fresh manifest for now).
- UI for the approval flow (admin-ui can pick it up in a follow-up).
- Agent self-revising a denied proposal automatically (it can retry but
  the same proposal counts against the budget).

## D.2 — Session compaction for long local-model sessions

### D.2.1 Context

`crates/xiaoguai-agent/src/history.rs` today implements `slide(messages,
window)` — a fixed-size sliding window that drops oldest messages. Good
enough for stateless chat; bad for long agentic workflows on Ollama
(qwen2.5-coder default has 32k context, so a 100-turn session with tool
calls easily exceeds it). The file itself says:

> LLM-driven summarisation (and the `/compact` slash command) are tracked
> in the v0.5.4.1 backlog.

Plan D.2 lands the v0.5.4.1 work.

### D.2.2 Success criteria

1. A new public function
   `compact(messages: Vec<Message>, model: &dyn LlmBackend, max_tokens:
   usize) -> Vec<Message>` exists in `crates/xiaoguai-agent/src/history.rs`
   alongside `slide`. It is called *instead of* `slide` when token count
   exceeds `max_tokens * 0.75`.
2. `compact` keeps the system message + last N (configurable, default 6)
   user/assistant pairs verbatim, and replaces older messages with a
   single synthetic `system` message containing an LLM-generated summary.
3. Token-count estimation is deterministic (use `tiktoken`-equivalent or a
   conservative whitespace approximation; do **not** depend on Ollama).
4. A new `/compact` slash-command in `xiaoguai chat` and the REST API
   triggers an immediate compaction regardless of threshold.
5. Integration test: synthetic 100-turn conversation including 30 tool
   calls is compacted; the resulting history (a) preserves the
   `tool_call_id`-pairing invariant `history.rs::slide` already enforces;
   (b) total token count is ≤ `max_tokens * 0.5`; (c) the agent can
   answer a follow-up question that depends on a fact mentioned 80 turns
   ago.
6. If the summarization LLM call fails or times out, the function falls
   back to the existing `slide` behaviour and emits a
   `xiaoguai_compaction_fallback_total` Prometheus counter.
7. The configuration knobs are surfaced in `local.yaml`:
   ```yaml
   agent:
     compaction:
       enabled: true
       max_context_tokens: 30000   # leave headroom below 32k for tool defs
       trigger_at_pct: 75
       keep_recent_turns: 6
   ```
8. `docs/runbooks/compaction.md` documents tuning advice for different
   models.

### D.2.3 Prerequisites

| What | Verify by |
|---|---|
| `xiaoguai-llm` exposes a token-count utility | `grep -r 'fn count_tokens' crates/xiaoguai-llm/src/` (write one if missing) |
| Ollama backend is reachable for the integration test | `curl -sf localhost:11434/api/tags` |
| `MockBackend` can be scripted with multi-step responses | `grep MockBackend crates/xiaoguai-llm/src/` |

### D.2.4 Step-by-step

#### Step D.2.4.1 — Add a token estimator

In `crates/xiaoguai-llm/src/token_count.rs` (new file):

```rust
pub fn estimate_tokens(s: &str) -> usize {
    // Whitespace-conservative; ~4 chars/token is the common GPT heuristic
    (s.chars().count() + 3) / 4
}

pub fn estimate_message_tokens(messages: &[Message]) -> usize { ... }
```

**VC:** `cargo test -p xiaoguai-llm token_count` passes a unit test that
exercises both functions on small inputs.

#### Step D.2.4.2 — Implement `compact`

In `crates/xiaoguai-agent/src/history.rs`:

```rust
pub async fn compact(
    messages: Vec<Message>,
    backend: &dyn LlmBackend,
    cfg: CompactionConfig,
) -> Vec<Message>
```

Algorithm:
1. Partition: system, head (everything except last `keep_recent_turns *
   2` non-system messages), tail.
2. If `head` is empty → return original.
3. Render `head` as plain text; call `backend.summarize(head_text)` with
   a system prompt like *"Summarise the following agent conversation in
   ≤ 500 tokens. Keep concrete facts (names, IDs, file paths). Drop
   commentary."*
4. On success: return `[system_prompt, synthetic_summary_message, ...tail]`.
5. On failure: log + bump
   `xiaoguai_compaction_fallback_total`, fall through to
   `slide(messages, cfg.keep_recent_turns * 2 + 1)`.
6. Re-apply the dangling-`Tool`-message guard from `slide()`.

**VC:** unit tests in `history.rs::tests` cover happy path, fallback,
and the `tool_call_id` invariant.

#### Step D.2.4.3 — Wire into the agent loop

In `crates/xiaoguai-agent/src/loop_.rs`:

- Before sending each request, estimate tokens.
- If `est_tokens > cfg.max_context_tokens * cfg.trigger_at_pct / 100`,
  call `compact(...)` and use the result.

**VC:** existing integration test
`crates/xiaoguai-agent/tests/loop_basic.rs` still passes; a new test
`tests/compaction.rs` runs a 100-turn synthetic session and asserts
post-compaction token count.

#### Step D.2.4.4 — Slash-command + REST trigger

CLI: `xiaoguai chat --compact` flag forces compaction on the
in-session history.

REST: `POST /v1/sessions/{id}/compact` admin endpoint forces compaction.

**VC:** `curl -X POST /v1/sessions/<id>/compact` returns 200 and the

#### Step D.2.4.5 — Metrics

**VC:** `curl localhost:7601/metrics | grep xiaoguai_compaction` shows

#### Step D.2.4.7 — Docs + PR

Write `docs/runbooks/compaction.md` covering:
- when to tune `trigger_at_pct`
- recommended `max_context_tokens` per model (qwen2.5-coder, llama3.1,
  gpt-4o-mini, claude-haiku-4-5)
- how to debug "agent forgot something it should know"

PR: `feat(tier-2): session compaction for long local-model sessions`.

**VC:** CI green.

### D.2.5 Risks (specific to D.2)

| Risk | Mitigation |
|---|---|
| Summary quality on small local models is poor; agent loses critical facts | Default config is `enabled: false`; operators opt-in per tenant. Documentation includes example "bad summary" failure mode. |
| Compaction itself takes 2-5s on a 7B Ollama model — adds noticeable latency | Run compaction async to the request (snapshot history at request start, return response from un-compacted history, persist compacted version for the next turn). |
| Token estimator is approximate; we might trigger too late and overflow | Conservative 4-char/token estimate **rounds up**. Add a hard upper bound check + retry-on-overflow path. |
| `tool_call_id` invariant breaks if compaction drops the assistant message but keeps a `Tool` response | The dangling-`Tool` guard from `slide()` is reused (step D.2.4.2 algorithm step 6). |
| Operators on Anthropic/OpenAI prefer their *provider's* native summarization | Compaction is optional; document that Claude's `cache_control` is the better tool for those providers and don't enable compaction by default for non-local backends. |

### D.2.6 Out of scope (D.2)

- Provider-native compaction (Claude `cache_control`, OpenAI conversation
  IDs) — track separately.
- Hierarchical summarization (summary-of-summaries for very long
  sessions). v0.5.4.1 ships the flat version.
- Vector-based "recall by similarity" from historical turns
  (`xiaoguai-memory` already covers long-term facts).
- A UI for showing what was compacted (admin-ui can add later).

## Recommendation

**Pick D.2 (session compaction)** as the next ship. Rationale:

1. **Higher demonstrated payoff today.** The air-gap mode shipped in #60
   means real users will run long local-model sessions; the current
   slide-window is a known degradation point (file comment in
   `history.rs` flags it).
2. **Smaller blast radius.** D.2 is contained in `xiaoguai-agent` and
   `xiaoguai-llm`; D.1 touches storage, auth, api, agent, tasks, and a
   new HTTP endpoint.
3. **Reuses existing machinery.** Ollama backend, MockBackend,
   `LlmBackend` trait, `slide()` guard — all reused. D.1 needs new
   schema, new endpoint, new policy bucket.
4. **R.E.S.T alignment.** D.2 strengthens E (efficiency) and R
   (reliability) directly. D.1 is about future capability — important
   but not blocking.

D.1 is a strong follow-up *after* D.2 ships, and after agent-authored
skills get more shape from operator feedback.
