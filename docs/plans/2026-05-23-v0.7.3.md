# v0.7.3 — PG-persisted IM history + tenant mapping

**Status:** done.

## What shipped

v0.7.2 kept IM conversation history in process memory (`ConversationHistory`,
a `HashMap<conversation_id, Vec<LlmMessage>>`). Two problems:

1. **Multi-replica split-brain.** Two Feishu webhooks for the same chat
   could land on different pods; the second pod would not see the first's
   history.
2. **Process restart wipes context.** A core deploy / OOM kill drops
   every active conversation.

v0.7.3 lifts the store behind a new `ImHistoryStore` trait and adds a
PG-backed implementation that resolves the external IM identity to the
internal tenant/user/session model on first sight, then reads / writes
the conversation turns through the same `MessageRepository` the REST
API uses.

## Pieces

### Migration `0006_im_identity.sql`

Two small mapping tables:

```
im_identities    (provider, tenant_external_id, user_external_id)
  → (tenant_id, user_id)

im_conversations (provider, tenant_external_id, conversation_id)
  → session_id
```

Both populated on first webhook for a given identity / chat. After that,
PK-index lookups stay fast and tenant / user / session rows are reused.
Neither table is itself RLS-scoped — they are the bootstrap indirection
that *finds* the tenant before downstream RLS-aware writes can run.

### `PgImIdentityRepository`

`crates/xiaoguai-storage/src/repositories/im.rs`. Two helpers:

* `resolve_or_create_identity(ExternalIdentity, display_hint)` — looks up
  `(provider, tenant_ext, user_ext)`. On miss: creates a synthetic
  tenant (`name = "im:{provider}:{tenant_ext}"`), a synthetic user
  (`email = "{user_ext}@{tenant_ext}.{provider}.im.invalid"` — a reserved
  zone that cannot collide with a real directory address), inserts the
  mapping. Concurrent first-touch is safe: `ON CONFLICT DO NOTHING` plus
  a re-select.
* `resolve_or_create_conversation(ExternalConversation, identity, model)`
  — same shape, creates the `sessions` row inside the identity's tenant
  on first sight and stores the mapping.

### `ImHistoryStore` trait + impls

`crates/xiaoguai-im-gateway/src/history.rs`:

```rust
#[async_trait]
pub trait ImHistoryStore: Send + Sync {
    async fn snapshot(&self, ident: &ConversationIdent) -> Result<Vec<LlmMessage>, HistoryError>;
    async fn extend(&self, ident: &ConversationIdent, msgs: Vec<LlmMessage>) -> Result<(), HistoryError>;
}
```

* `ConversationHistory` (existing in-process `HashMap` store) gets a
  thin async impl that ignores `tenant_external_id` / `user_external_id`
  and keys only by `conversation_id` — preserves the v0.7.2 semantics
  for single-replica deployments and the test suite.
* `PgImHistoryStore` (new in `pg_history.rs`) resolves identity +
  conversation, reads the trailing `max_turns` messages via
  `MessageRepository::list_by_session`, and writes each turn via
  `MessageRepository::append`. Text-only: `Role::Tool` turns are
  filtered out (consistent with v0.7.2; the IM gateway only renders
  text replies).

### Router rewire

`GatewayState.history` is now `Arc<dyn ImHistoryStore>` (was
`Arc<ConversationHistory>`). `run_agent_and_reply` builds a
`ConversationIdent` from the inbound `IncomingMessage` and dispatches
through the trait. Production wiring substitutes the PG store; tests
keep the in-memory default.

## Tests

* `crates/xiaoguai-im-gateway` unit (`history.rs`): trait round-trip via
  the in-memory impl. `pg_history.rs`: pure-function conversions
  (`LlmMessage` ↔ `DomainMessage`).
* `crates/xiaoguai-im-gateway` integration (`feishu_route.rs`):
  unchanged behaviour — multi-turn history accumulation still works via
  the trait dispatch path. The existing tests now use `Arc<dyn
  ImHistoryStore>` instead of the concrete type.
* `crates/xiaoguai-storage/tests/im_identity_repo.rs` (4 ignored
  testcontainer tests): first-webhook auto-creates rows; second webhook
  reuses; conversations are scoped per `(provider, tenant_ext)`;
  different `tenant_ext` produce isolated tenants.

## Deferred (v0.7.4+)

* Operator switch in `xiaoguai-core` to pick PG vs in-process via
  configuration — right now production wiring still needs a small
  bootstrap change in the core binary to construct `PgImHistoryStore`
  and pass it into `mount_feishu_with_history`.
* Persist `Role::Tool` turns in the IM transcript so multi-turn tool
  use survives across webhook deliveries. Today they're filtered out.
* Token / character cap on stored conversations (currently unbounded
  per chat; relies on the trailing-window read).
* Tenant directory administration UX: today the synthetic tenant /
  user rows show up in `/v1/admin/tenants` next to real ones. Decide
  whether to flag them with a `source = "im:feishu"` column.
