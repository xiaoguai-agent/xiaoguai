# v1.1.8 — promote security pipeline to gating

**Status:** done (2026-05-24).

## Why this slice

v0.12.2 closed the v1 feature set. The CI security posture lagged
behind: `cargo-deny` ran but was advisory-only (no explicit GPL ban,
no copyleft guardrail); `cargo-audit` wasn't wired at all; no SCA on
PRs; nothing scanned the published container image.

v1.1.8 promotes the dependency-security pipeline from "we have the
infra" to "merges block on real findings". Three independent
guardrails, one tag.

## Pieces

### 1. `cargo-deny` promoted to gating

`.github/workflows/deny.yml` — added a header comment marking the
promotion. The job was already calling `EmbarkStudios/cargo-deny-action@v2`
without `continue-on-error: true`, so the action's non-zero exit was
already failing the job in principle; what was missing was an explicit
allow/deny posture in `deny.toml` that actually rejected anything we
care about.

`deny.toml` changes:

- Top-of-`[licenses]` comment makes the Apache-2.0 + copyleft-incompatible
  posture explicit. The allow-list is closed: anything not listed is
  denied. GPL-*, AGPL-*, LGPL-* therefore fail. (Today nothing in the
  tree pulls them in — this is a forward guard.)
- `[advisories].ignore` extended to mirror `.cargo/audit.toml` entries
  (rsa via unused sqlx-mysql, instant via notify). Every ignore now
  carries a written reason. Header comment commits us to quarterly
  review.

### 2. `cargo-audit` nightly cron

`.github/workflows/audit.yml` — new file.

- Cron `0 8 * * *` UTC (16:00 Beijing). Also `workflow_dispatch` so on-
  call can fire it manually.
- `cargo install cargo-audit --locked` (pinned by lockfile), then
  `cargo audit --deny warnings` (escalates unmaintained / yanked to
  errors).
- On failure: writes a structured Markdown report to `issue-body.md`
  with the run URL + the captured `cargo audit` output, then opens a
  GitHub issue via `peter-evans/create-issue-from-file@v5` with labels
  `security`, `automated`, `cargo-audit`.
- Permissions scoped: `contents: read`, `issues: write`. No write to
  code, no PR-create permissions.

Ignore list lives in `.cargo/audit.toml` so the same set is honored by
both the cron and any local developer invocation. Keep it in
lock-step with `deny.toml` `[advisories].ignore`.

### 3. Snyk SCA + container scan

`.github/workflows/snyk.yml` — new file.

- Fires on `push` to main + every `pull_request` to main.
- Two jobs: `snyk-rust` (SCA via `snyk/actions/rust@master`) and
  `snyk-docker` (container scan via `snyk/actions/docker@master`).
- The docker job builds `deploy/Dockerfile` locally with
  `docker/build-push-action@v6` (`load: true, push: false`) before
  handing it to Snyk — no need to push a scratch tag to ghcr.
- Threshold: `--severity-threshold=high` gates the workflow. Medium and
  low findings still show up in the Snyk UI but don't block.
- Token gating: every step that needs `SNYK_TOKEN` checks for it via
  `steps.token.outputs.available`. If the secret is unset (fork PR,
  external contributor), the job logs a `::notice::` and skips
  silently — no red X for contributors who can't fix it. Also gated
  upstream via `if:` clause that blocks fork PRs entirely from running
  the Snyk job (forks can't see secrets anyway).

## Deferred (v1.1.8.1+)

- **cargo-vet bootstrap.** `cargo install cargo-vet --locked; cargo
  vet init; cargo vet check` is a one-line workflow but the
  `supply-chain/audits.toml` it generates needs human review of every
  dep before `cargo vet check` passes. Bootstrap session needs ~1 hour
  of attestation work. Deferred to v1.1.8.1 so v1.1.8 ships the
  gating story today.
- **Dependabot.** GitHub-native, configured via `.github/dependabot.yml`
  not a workflow file. Separate setup. v1.1.8.2.
- **GitHub Advanced Security (CodeQL).** Paid tier on private repos;
  free on public. Whether this repo is public-or-private at v1 release
  time decides whether we wire it. Defer until that call is made.
- **Snyk Code (SAST) on top of Snyk Open Source (SCA).** Same token,
  adds `snyk/actions/setup` + `snyk code test`. Lower-priority because
  Rust SAST coverage in Snyk Code is still maturing.
- **Issue de-duplication for the audit cron.** Today the cron opens a
  new issue every failed run. If an advisory stays open for a week,
  that's 7 issues. Either we layer a "check if an open issue with this
  title prefix exists" step, or accept the noise and triage manually.
- **Fixing the 4 ignored advisories.** When upstream `notify` cuts a
  `notify-types` release without `instant`, drop the ignore. When
  `sqlx` ships feature-gated mysql/rsa decoupling, drop that ignore.
  Track at quarterly review.

## Design decisions worth flagging

- **Two ignore files (`.cargo/audit.toml` + `deny.toml`).** Couldn't
  avoid it — `cargo-audit` doesn't read `deny.toml` and `cargo-deny`
  doesn't read `audit.toml`. The CLAUDE.md-style discipline is "any
  add in one must be added to the other with the same reason"; the
  written reason makes drift obvious in code review.
- **Cron at 08:00 UTC instead of midnight.** RustSec advisories
  typically land US business hours; running the cron 4-6 hours after
  the bulk of the publish window catches same-day advisories. Beijing
  16:00 is also when this team is most likely to triage.
- **`peter-evans/create-issue-from-file@v5` rather than direct gh api
  call.** The action handles auth + body escaping + idempotency
  edge-cases. Pinning to a tagged release means we get behavior
  guarantees that a hand-rolled curl wouldn't.
- **Snyk thresholds at high, not medium.** Medium-severity in Snyk
  often flags supply-chain risks that aren't exploitable in our shape
  (e.g., a parser CVE on a file format we never parse). Setting the
  gate at high keeps the signal-to-noise high enough that PR authors
  trust the wall. Medium findings are still visible — just non-gating.
- **Snyk gracefully skips on missing token.** External contributor PRs
  can't see the SNYK_TOKEN secret; if the job hard-failed without it,
  every fork PR would be red. Token check + `if:` guard means the job
  is invisible to those PRs and main-push still scans.
- **No `continue-on-error: true` anywhere.** Security findings should
  fail loudly. The only soft-skip is the SNYK_TOKEN one above, and it
  skips the *step*, not the workflow — the workflow still passes.
