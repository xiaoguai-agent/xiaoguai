# v0.11.0 — `xiaoguai-eval` crate

**Status:** done.

## What shipped

First slice of the P1 differentiation band (roadmap §3, v0.11
row). The audit-first console (v0.11.1) and the eval pane (v0.11.2)
both need a regression-eval substrate underneath; v0.11.0 lands
that substrate as a standalone crate so neither downstream slice
has to invent it.

The contract from roadmap §5.4 is non-negotiable: **no new DB
tables.** An eval case is `(input_messages, mock_script,
assertions)`; the substrate is the existing `sessions` +
`audit_log` rows plus `MockBackend` for deterministic traces. The
v0.11.2 "convert prod run to regression case" button will read a
`sessions.id`, pull its `messages` + `audit_log.tool.invoke` rows,
and project them into the same `EvalCase` shape this crate
defines.

## Pieces

### `crates/xiaoguai-eval/` — new workspace member

Mirrors the `xiaoguai-rag` / `xiaoguai-scheduler` style: one crate
with a flat module set (`types` / `grader` / `runner` / `suite` /
`report`), pedantic clippy clean, `forbid(unsafe_code)`. Cargo
deps stay minimal: `xiaoguai-agent`, `xiaoguai-audit`,
`xiaoguai-llm`, plus workspace `serde` / `serde_yaml` / `regex` /
`thiserror` / `tokio` / `tracing` / `chrono`.

### `types.rs` — wire format

* `MockTurn { text, tool_calls }` — one scripted assistant turn.
  Local shape (instead of re-using `xiaoguai_llm::ScriptStep`) so
  YAML cases don't carry `FinishReason`, which is a runtime concern
  cases shouldn't pin.
* `MockScript { turns }` — full script, replays last turn
  indefinitely past the end (matches `MockBackend::with_script`
  semantics so the agent loop never deadlocks past the script).
* `Assertion` enum, five variants:
  * **Outcome graders:** `FinalMessageContains`,
    `FinalMessageRegex`, `ToolInvocationCount`.
  * **Transcript graders:** `AgentEventSequence` (subsequence
    match on the snake_case event tag), `ToolCallSequence`
    (subsequence by tool name + optional argument substring).
* `EvalCase { id, input_messages, mock_script, assertions, tags }`,
  `EvalSuite { name, cases }`, `CaseStatus = Pass | Fail
  { reasons }`, `EvalResult { case_id, status, transcript_len,
  duration_ms }`, `EvalReport { suite, started_at, finished_at,
  results, pass_rate }`.

### `grader.rs` — pure assertion logic

Each assertion compiles to a `fn(events, messages) -> Result<(),
String>`. Reasons are human-readable and quote the actual value
back to the case author. Regex compile errors fail the case with a
clear reason rather than crashing the runner — the assertion is
the grader, not the runner.

The `AgentEventSequence` matcher is a subsequence — interleaving
`text_delta`s between two expected `tool_call_started`s doesn't
fail the case. Streaming would otherwise make every assertion
brittle.

The `ToolCallSequence` matcher tries the raw arguments string
first and then a canonicalised JSON re-encode, so a case can pin
"the arguments contained `\"city\":\"Berlin\"`" without committing
to the model's exact whitespace.

### `runner.rs` — agent loop driver

* `EvalAgentBuilder` async trait — pluggable factory. v0.11.0
  ships `DefaultEvalAgentBuilder` (empty `Toolbox`, configurable
  `max_iterations`); v0.11.1+ can wire a real toolbox builder when
  capability suites need MCP tools.
* `EvalRunner::new(Arc<dyn EvalAgentBuilder>)` —
  `run_suite(&EvalSuite) -> EvalReport`. Per-case grader failures
  become `Fail` rows; only actual agent-loop crashes bubble up as
  `RunnerError::Agent`.
* `mock_from_script(&MockScript) -> MockBackend` — fan the
  per-case script into the upstream `MockBackend::with_script`.
* `pretty_summary(&EvalReport) -> String` — multi-line summary the
  CLI prints to stdout.

### `suite.rs` — YAML loader

`EvalSuite::load_from_dir(name, &Path)` walks `*.eval.yaml` files
*directly* under `dir` (no recursion — suites stay flat so
`--suite regression` vs `--suite capability` is visually obvious
from the layout). One file = one case; alphabetical ordering for
stable test output. Errors carry the offending path so a
malformed file is one `grep` away.

### `report.rs` — JSON writer

`EvalReport::write_json(&Path)` writes pretty-printed JSON. The
v0.11.2 eval pane will consume the same shape; serializing
straight off `EvalReport` keeps the contract one source of truth.

### `examples/eval/regression/*.eval.yaml` — shipped fixtures

Two trivial cases (`greeting.eval.yaml`, `no_tool_call.eval.yaml`)
prove the on-disk schema with the real loader + real
`MockBackend` + real `ReactAgent`. Their integration test
(`tests/example_suite.rs`) is the canary that fires before any
downstream consumer notices a refactor break.

### `xiaoguai eval run` CLI subcommand

`crates/xiaoguai-cli/src/commands/eval.rs` plus a new `Eval`
variant in `Cmd` + `EvalCmd::Run` subcommand in `main.rs`. Matches
the existing `Chat` / `Provider` / `Mcp` / `Remote` pattern:
business logic in `commands/eval.rs`, parsing + dispatch in
`main.rs`. CLI exits non-zero when any case fails so CI can gate
on the return code without parsing stdout.

## Deferred

- **`Assertion::RepoStateMatches(JsonPath, Value)`.** Roadmap
  §5.4's outcome-grader list mentions "final state of the
  in-memory repo", but v0.11.0 doesn't have a well-defined
  "snapshot" surface across repos that an eval could pin against
  without hard-coding a particular repo's serde shape. The eval
  substrate doesn't suffer from the omission — the four shipped
  graders cover every regression case the v0.10.x band could
  produce. Revisit alongside the v0.11.1 console once the audit
  schema settles on what "today's state" looks like.
- **Capability suites with real `Toolbox`.** Today
  `DefaultEvalAgentBuilder` ships an empty `Toolbox`, so any
  `ToolCallSequence` assertion against a non-trivial tool gets a
  `ToolCallFinished { ok: false, error: "tool ... not in
  toolbox" }`. The trait was deliberately built `Arc<dyn
  EvalAgentBuilder>` so v0.11.x can ship a `McpEvalAgentBuilder`
  that boots an MCP supervisor per case, but the shape of "which
  MCP server, with what config" is exactly the v0.9.4 marketplace
  question — pair them naturally.
- **PG-backed eval persistence.** Reports are JSON on disk; the
  v0.11.2 eval pane will likely keep them there (eval runs are
  artifacts, not first-class data). If real-world feedback shows
  operators want a "recent eval runs" view in the admin-ui, a
  thin `pg_eval_reports` table is the obvious follow-up against
  the existing `EvalReport` shape.
- **Streaming runner.** `EvalRunner::run_suite` runs cases
  sequentially and returns the full `EvalReport` at the end. A
  `tokio::sync::mpsc<EvalResult>` variant would let the CLI
  print per-case lines as they land; the v0.11.2 console will
  probably want it. Trivial follow-up against the existing
  runner.
- **Parallel case execution.** Sequential is fine for the
  per-case work this crate ships (no I/O beyond the in-memory
  `MockBackend`). A `cases_in_parallel: usize` option on a new
  `RunnerOptions` is the natural shape once a real backend +
  toolbox arrive in v0.11.x.

## Design decisions worth flagging

- **Substrate is the existing tables, full stop.** Tempting to
  add `eval_runs` / `eval_cases` tables now to make the v0.11.2
  console "easier" later. Refused: every PG migration costs us
  RLS + index review + downgrade story. Reports are
  artifacts — keep them on disk.
- **`EvalAgentBuilder` over a concrete `ReactAgent` parameter.**
  The runner could take an `Arc<ReactAgent>` directly; we
  instead take a *factory* so each case gets a fresh
  `MockBackend` (the upstream backend is `Arc<Mutex<>>`-backed
  and stateful across calls — sharing one across cases would
  bleed script state between them). Builder also gives v0.11.x
  the seam for tool-equipped agents without changing the runner.
- **YAML, not JSON, for case files.** Operators hand-author
  these; YAML is the obvious choice. The `serde` derives are
  format-agnostic so a JSON loader is a five-line addition if
  anyone asks for it.
- **Subsequence match for transcript graders.** Strict equality
  on streamed `AgentEvent`s is too brittle — every `text_delta`
  chunk would have to be pinned exactly. Subsequence lets a case
  pin the *load-bearing* sequence ("a `tool_call_started` for
  `search` followed by a `done`") without committing to the
  every-byte transcript.
- **Reasons accumulate per case, not bail-at-first.** A case
  author sees the full failure picture in one run, not one
  fix-and-re-run cycle per assertion.
- **CLI exits non-zero on any failure.** The eval suite is a CI
  gate first, a developer convenience second. Stdout is the
  human-readable summary; the JSON report is for tooling.
- **Sketch test, not real query, for the audit substrate.** The
  `tests/regression_from_audit.rs` projection function is hand-
  built — it documents the shape and traps schema drift, but the
  *real* PG query lives in `xiaoguai-storage` where the sqlx
  types are in scope. Keeps the eval crate free of a sqlx
  dependency.
