# v0.10.3 — push sinks (Feishu / Telegram / Email / Inbox)

**Status:** done.

## What shipped

Fourth and final P1 slice in the v0.10.x band. v0.10.0 shipped the
`PushSink` trait + `LoggingSink` stub; v0.10.2 added the
proactive-fire path and `PushPayload.reason`. v0.10.3 fills in the
real sinks that the scheduler binary will actually wire — one per
roadmap §3 P1 channel:

* **Feishu** — reuses the v0.7.1 `FeishuClient` + `TokenCache` so
  there's exactly one place in the codebase that knows how to fetch
  and cache a Feishu `tenant_access_token`.
* **Telegram** — Bot API `sendMessage` over `reqwest`. Stateless.
* **Email** — JSON webhook to a relay (rationale below). No SMTP
  dependency.
* **Inbox** — in-memory FIFO with `pop_all()`, ready to be drained by
  the v0.11.1 audit-first console's inbox pane.

The reason-required rule from roadmap §5.5 lands at the sink edge
this tag: a new `PushPayload::is_proactive` field tells each sink
whether the fire originated from `Trigger::Proactive`, and a one-line
helper `PushPayload::require_reason_when_proactive()` returns
`SinkError::Invalid("reason required")` when proactive + empty reason
collide. Every real sink calls it at the top of `deliver`, so the
guarantee is uniform whether the sink is Feishu, Telegram, Email,
Inbox, or a future custom one.

## Pieces

### `PushPayload.is_proactive: bool`

Added with `#[serde(default)]` for back-compat with v0.10.0–v0.10.2
persisted rows. The runner populates it from `job.trigger.is_proactive()`
when building the payload, so the sink can enforce the
reason-required rule without needing to receive the full `Trigger`
enum. v0.10.2 already shipped `reason: String` with the same back-compat
treatment.

### `PushPayload::require_reason_when_proactive`

```rust
pub fn require_reason_when_proactive(&self) -> Result<(), SinkError> {
    if self.is_proactive && self.reason.trim().is_empty() {
        return Err(SinkError::Invalid("reason required".into()));
    }
    Ok(())
}
```

Every real sink under `crate::sinks` calls this at the top of
`deliver`. `LoggingSink` (dev / capture / tests) is intentionally
permissive — same as v0.10.0.

### `crate::sinks::feishu` — `FeishuPushSink`

* Owns one `xiaoguai_im_feishu::TokenCache` per instance, built from
  the v0.7.1 `FeishuClient` trait. **No code is duplicated** from the
  v0.7.1 outbound path; the sink is a thin adapter that turns a
  `PushPayload` into a `send_text_message(token, chat_id, text)` call.
* Renders the payload as plain Feishu text. Proactive fires get a
  `【主动推送】<reason>` prefix so receivers see *why* they're getting
  the push first.
* `FeishuSinkConfig { app_id, app_secret, chat_id }` — serde-derived
  so it can deserialize directly from `Settings::scheduler.sinks.feishu`.

### `crate::sinks::telegram` — `TelegramPushSink`

* POSTs to `<base_url>/bot<token>/sendMessage`. `base_url` defaults
  to `https://api.telegram.org` and is overridable for the mockito
  tests.
* Renders to plain text with a `🔔 ` bell prefix on proactive fires.
* Non-success HTTP status surfaces as `SinkError::Delivery` with the
  status + body in the message — operator-friendly diagnostics.

### `crate::sinks::email` — `EmailPushSink`

* JSON POST to `webhook_url` with the shape
  `{ to, from, subject, body, payload }`. The full original payload
  rides along so a smart relay can build HTML if it wants.
* **No SMTP.** v0.10.3 deliberately avoids adding `lettre` (200KB of
  dependencies for transient retry / DKIM / SPF / bounce handling
  that any operator's existing relay already does better). The
  webhook pattern stays trivial; SMTP-native is a separate decision
  for whoever needs it.
* `render_subject` / `render_body` are public for assertion in tests.

### `crate::sinks::inbox` — `InboxPushSink`

* In-memory `VecDeque<InboxMessage>` with `pop_all()` /
  `len()` / `is_empty()`. The v0.11.1 audit-first console pane drains
  via `pop_all` and renders.
* Persistence is **deferred** — see "Deferred" below. The contract
  (FIFO of `InboxMessage { enqueued_at, payload }`) is identical to
  what the PG-backed impl will be in v0.12.0.

### `xiaoguai-config` — `scheduler.sinks` block

New `SchedulerSettings { sinks: SchedulerSinkSettings }` with one
`Option<serde_json::Value>` field per sink (feishu / telegram / email
/ inbox). Storing the per-sink config as opaque JSON keeps
`xiaoguai-config` free of a dependency on `xiaoguai-scheduler` (which
would create a circular crate graph the moment a sink wanted to read
a config knob). The operator binary calls
`serde_json::from_value::<FeishuSinkConfig>(...)` when constructing
each sink. New field is `#[serde(default)]` — existing config.yaml
files from v0.10.0/v0.10.1/v0.10.2 still parse.

### Cargo wiring

* `xiaoguai-scheduler` now depends on `xiaoguai-im-feishu` (for
  `FeishuClient` + `TokenCache` + `TokenResponse`) and
  `xiaoguai-im-gateway` (for `ProviderError`). Track B's changes are
  to the **gateway** side of `xiaoguai-im-feishu`; the Cargo.toml
  add here only touches dependency graph, not source — track B's
  diff and ours don't overlap.
* `reqwest` added (workspace version) as a runtime dep for the
  Telegram + Email HTTP clients.
* `mockito` added as a dev-dep for the Telegram + Email tests.

## Deferred

- **Operator wiring in `xiaoguai-core`.** Still deferred from
  v0.10.0; this tag adds the building blocks, v0.12.0 lands the
  `tokio::spawn(JobRunner::run_loop)` + sink-construction code that
  reads `settings.scheduler.sinks.*` and assembles the runner.
- **Inbox persistence.** v0.10.3 keeps the inbox in-memory; PG-backed
  storage lands in v0.12.0 alongside the rest of the scheduler PG
  repos. The trait contract (`pop_all() -> Vec<InboxMessage>`) is
  identical either way.
- **SMTP-native email.** Webhook-to-relay is the only path shipped.
  If feedback shows operators want xiaoguai-core to speak SMTP
  directly, a `LettrePushSink` against the same `PushSink` trait is
  the obvious follow-up.
- **DingTalk + WeCom sinks.** `xiaoguai-im-dingtalk` /
  `xiaoguai-im-wecom` exist as crate stubs (per the workspace
  member list); concrete adapters land in v0.7.4 (IM finishers).
  Once those ship, mirroring the `FeishuPushSink` pattern is
  ~50 lines per sink.
- **Per-tenant sink config.** Today a sink's `chat_id` / `to` /
  `bot_token` / `chat_id` are global per sink instance. Wiring one
  sink instance per `(tenant, destination)` pair is trivial against
  the existing `ScheduledJob.sinks: Vec<String>` field — out of
  scope for v0.10.3 since "small team" deployments don't need it.
- **Rich Feishu cards / Telegram inline keyboards.** v0.10.3 ships
  plain text bodies. Interactive cards live in the same "v0.7.x IM
  finishers" backlog as DingTalk/WeCom and pair naturally with the
  v0.11.1 console redesign.
- **Sink retry inside the sink.** A failed Telegram delivery (HTTP
  500) bubbles up as `SinkError::Delivery`; the runner logs and
  moves on. The job's own `RetryPolicy` is intentionally not
  applied to sink failures — sinks deliver job *results*, and
  re-running the executor isn't the right answer to "Telegram is
  having a bad day". A per-sink retry queue is the right shape and
  belongs in v0.12.0 once we have the PG inbox to act as a buffer.

## Design decisions worth flagging

- **Reason-required is enforced at the sink edge, not the runner.**
  The runner already knows whether a fire was proactive (it picked
  the path), so it could refuse to call `deliver` for
  proactive-empty-reason payloads up-front. We deliberately put the
  guard *at the sink* so any custom sink outside this crate (a
  user's Slack adapter, a future LINE adapter) gets the rule for
  free by calling one helper. One enforcement point > N enforcement
  points.
- **`PushPayload::is_proactive` instead of carrying the whole
  `Trigger`.** A `Trigger` field would pull `is_proactive`'s six
  variant tags into the payload's serialised JSON; the bool is one
  byte and means exactly what the sink needs.
- **Feishu sink reuses `FeishuClient` rather than calling `reqwest`
  directly.** The token cache + endpoint URL + error envelope
  handling already live in `xiaoguai-im-feishu`. Duplicating them
  would mean two places to fix a Feishu API change. The Cargo
  dependency cost is one extra crate (already in the workspace).
- **Email is a webhook by design, not SMTP.** Operators serious
  about email have a relay already; operators not serious about
  email don't want to debug DKIM. The webhook shape includes the
  full payload so the relay can do whatever templating it wants.
  If real-world feedback flips this judgement, switching to
  `lettre` is one new sink module against the existing trait.
- **`SchedulerSinkSettings` fields are `Option<serde_json::Value>`,
  not concrete config structs.** Otherwise `xiaoguai-config` would
  need a dependency on `xiaoguai-scheduler` (which depends on
  `xiaoguai-audit`, which transitively pulls in `xiaoguai-storage`)
  — a layering inversion. Storing the raw JSON keeps the config
  crate at the bottom of the graph; the binary deserializes at sink
  construction time, exactly where the type is in scope anyway.
- **Inbox is `pop_all`, not `peek` + ack.** The v0.11.1 console
  drains and renders — there's no per-message ack flow. If a future
  consumer needs at-least-once semantics (vs. the current
  at-most-once on console restart), the PG-backed v0.12.0 impl can
  add a `peek(limit)` / `ack(ids)` pair without changing the trait.
- **Doc-test `ignore` on the FeishuPushSink example.** The example
  references types and bindings from the v0.12.0 operator wiring
  that don't exist yet. `ignore` prevents the test runner from
  failing while keeping the example visible in `cargo doc`.
