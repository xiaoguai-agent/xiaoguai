# v0.7.2 — IM conversation history (in-process)

Status: **Done** — 2026-05-23.

## Goal

Make the Feishu bot understand follow-up turns. v0.7.1 ran the agent
on the inbound text alone — every webhook delivery started a fresh
context. v0.7.2 carries a per-`conversation_id` sliding-window history
so subsequent webhooks pick up where the previous turn left off.

## Why in-process, not PG

The `messages` table is FK'd to `sessions`, which is FK'd to `users`
and `tenants`. Auto-creating those rows from Feishu's
`tenant_key`/`open_id` requires a tenant/user mapping policy that
isn't a one-liner — and it interacts with the v0.6.1 RLS layer in
non-obvious ways. v0.7.2 keeps history in a `parking_lot::Mutex<HashMap>`
inside `GatewayState` so single-replica deployments get coherent
multi-turn chat *now*; multi-replica + persistence is a follow-up.

## What landed

### `xiaoguai-im-gateway`

- New `history` module with `ConversationHistory { max_turns,
  inner: Mutex<HashMap<String, Vec<LlmMessage>>> }`. Operations:
  `snapshot`, `append`, `extend`, `clear`. The window evicts oldest
  messages when it exceeds `max_turns`.
- `GatewayState` gains `history: Arc<ConversationHistory>`.
- New `mount_feishu_with_history(...)` constructor; existing
  `mount_feishu` keeps the same signature and now defaults to a
  fresh `ConversationHistory::new(DEFAULT_HISTORY_TURNS)` (20).
- `run_agent_and_reply`:
  1. Snapshots history for `msg.conversation_id`.
  2. Appends the inbound user message and runs
     `ReactAgent::run_to_completion` on the full window.
  3. Picks the latest assistant text.
  4. Appends `(user_msg, assistant_msg)` to the history store.
  5. Calls `provider.reply(...)`.

### Cargo

- `xiaoguai-im-gateway` promotes `parking_lot` from dev-dependencies
  to dependencies.

### Tests

- 6 new unit tests in `history::tests::*`: empty snapshot, round-trip,
  sliding-window eviction, conversation isolation, clear, extend.
- New integration test in `feishu_route.rs`:
  `conversation_history_accumulates_per_chat` drives three turns
  through `run_agent_and_reply` (two for one chat_id, one for
  another) and asserts the per-chat windows match what the scripted
  `MockBackend` returned.

## Deferred to v0.7.3

- **PG-persisted IM history.** Today the window is process-local;
  multi-replica deployments split-brain on whichever replica receives
  the webhook. A Valkey-backed `ConversationHistory` trait impl is
  the minimum credible fix; PG + auto-create of tenants/users is the
  proper one.
- **Tenant + user mapping** from Feishu `tenant_key` / `open_id` to
  internal IDs (with auto-create). Same scope as PG persistence.
- **Per-conversation reset command** (`/reset`, `/forget`) — the
  `ConversationHistory::clear` API is exposed but no IM command
  wires it.
- **History TTL** — currently entries linger until process restart;
  a time-based eviction would bound memory under churn.
- **DingTalk + WeCom + interactive cards + encrypted-payload Feishu**
  — same backlog the v0.7.1 plan named.
