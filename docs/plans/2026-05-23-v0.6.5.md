# v0.6.5 — `PgAuditSink` bootstrap + chain verify endpoint + IM tenant routing

**Status:** done.

## What shipped

Closes three v0.6.5 backlog items in one tag because they share the same
theme: making per-tenant scoping actually visible across every entry
point — REST audit, IM gateway, and verified chain integrity.

1. **`PgAuditSink` wired into `xiaoguai-core`.** The production binary
   now reads the audit signing key from the env var named by
   `[audit].signing_key_env` (default `XIAOGUAI_AUDIT_SIGNING_KEY`) and
   constructs one `PgAuditSink` that backs both `AuditReader` (read) and
   the new `AuditVerifier` (chain integrity). When the env var is unset
   we log a warning and leave the slots at `None` — the dev `hmac_key`
   in `AuditSettings` is reserved for `smoke` and tests; using it for
   production would silently desync a real chain.
2. **`GET /v1/admin/audit/verify`.** New admin endpoint behind the
   existing rate-limit + Casbin + bearer-token stack. Returns
   `200 {"ok": true, "verified_count": N}` on success and
   `200 {"ok": false, "broken_at": rowid}` when the chain breaks
   (200-with-flag instead of 5xx so dashboards can scrape the `ok`
   field as an alerting signal). 503 when no verifier wired; 400 when
   `tenant_id` is missing or empty.
3. **IM tenant propagation.** `ImHistoryStore::resolve_tenant` (new
   trait method, default impl returns `None`) is consulted on every
   webhook; the resolved internal tenant flows into
   `AgentConfig.tenant_id` so v0.6.4's per-tenant `LlmRouter` defaults
   finally apply to Feishu traffic. The in-memory store keeps the
   single-tenant default (returns `None`); `PgImHistoryStore` returns
   the synthetic / real tenant resolved during the first webhook.

## Pieces

### Config

* `[audit].signing_key_env: String` — defaults to
  `"XIAOGUAI_AUDIT_SIGNING_KEY"`. Documented in `xiaoguai-config` next
  to `hmac_key`. Matches the existing `[auth]` knob style (string field
  on the settings struct, no nested module).

### API

* `xiaoguai-api::audit`:
  - `AuditVerifier` async trait with one method
    (`verify_tenant(tenant) -> VerifyReport`).
  - `VerifyReport::Ok { verified_count }` / `Broken { broken_at }`.
  - `StaticAuditVerifier` in-memory impl for tests + dev.
* `AppState.audit_verifier: Option<Arc<dyn AuditVerifier>>` (parallel
  to `audit`).
* `routes::admin::verify_audit` handler.
* `crate::lib` re-exports `AuditVerifier`, `StaticAuditVerifier`,
  `VerifyReport`.

### Core binary

* New `xiaoguai-core/src/audit_bridge.rs` with `PgAuditAdapter`
  wrapping `Arc<PgAuditSink>` and implementing both `AuditReader` and
  `AuditVerifier`. Lives in core (not api) because api intentionally
  doesn't depend on `xiaoguai-audit` — keeping the API layer
  concrete-storage-agnostic. One sink instance services both endpoints
  (no risk of two sinks with divergent keys).
* `run_serve` reads `std::env::var(&settings.audit.signing_key_env)`;
  on success builds the adapter and wires it into both `AppState`
  slots, on miss logs a warning.

### IM gateway

* `ImHistoryStore::resolve_tenant(&ConversationIdent) -> Option<String>`
  default-impl method returns `None`; `PgImHistoryStore` overrides to
  return the resolved tenant.
* `run_agent_and_reply` consults the store before building
  `ReactAgent`; injects `agent_cfg.tenant_id = Some(resolved)` only when
  the store reports a tenant. In-memory store path is unchanged.

## Tests

`xiaoguai-api/tests/admin_and_listing.rs` — 4 new audit-verify cases:
* `admin_audit_verify_returns_ok_for_unbroken_chain` — 200 + ok:true +
  verified_count.
* `admin_audit_verify_reports_broken_chain` — 200 + ok:false +
  broken_at:rowid.
* `admin_audit_verify_503s_when_verifier_not_wired`.
* `admin_audit_verify_400s_when_tenant_missing`.

`xiaoguai-im-gateway/tests/feishu_route.rs` — 1 new tenant routing
case:
* `agent_inherits_tenant_resolved_by_history_store` — drives
  `run_agent_and_reply` through a stub history store that always reports
  `"ten_fixed"`; asserts the resolved tenant lands in
  `ChatRequest.tenant_id` via a recording `LlmBackend`.

`xiaoguai-im-gateway/src/pg_history.rs` — `domain_to_llm` and the now-
updated `llm_to_domain` test (the v0.7.4 change to persist tool turns is
also covered here so the v0.6.5 → v0.7.4 transition stays single-tag-
shippable).

`crates/xiaoguai-storage` `PgAuditSink::verify_tenant` integration test
already exists from earlier audit work (unchanged); the new adapter is
covered indirectly through the static verifier tests + by the explicit
adapter mapping. A live PG integration test for the adapter pairs with
the testcontainer audit suite already in `xiaoguai-storage` — held back
to keep this tag scoped.
