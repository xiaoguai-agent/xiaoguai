# v0.12.2 ŌĆö file watcher operator wiring + incremental RAG re-index + Obsidian catalog entry

**Status:** done (2026-05-24).

## Why this slice

Roadmap ┬¦3 P1-closer bundles three deliverables under v0.12.2:

1. The v0.10.1 `FileWatchSource` shipped but `xiaoguai-core::run_serve`
   never instantiates it. Operators wanting reactive jobs on file
   change had no path. Wire it now.
2. `xiaoguai-rag::RagClient` defines `ingest` (full-document) but no
   incremental re-index entry point. A file watcher that re-ingests
   on every change wants a path-shaped seam so backends don't have to
   re-read the file twice. Extend the trait + implement on both
   `R2RClient` and `InMemoryRagClient`.
3. Obsidian shows up as a community MCP server. v0.9.4 shipped a
   catalog entry that under-described it (read-only, env-var-driven)
   when the real `obsidian-mcp` server is read/write and takes the
   vault path as a positional CLI arg. Correct the entry; the
   write-capable first-party Obsidian connector stays a v1.1+
   roadmap item.

Bundling them lands the single user story ŌĆö "watch my Obsidian
vault, re-index the RAG collection on change" ŌĆö end-to-end in one
tag, even though piece 3 is only the catalog half.

## Pieces

### 1. File-watch operator wiring (`xiaoguai-core` + `xiaoguai-config`)

`xiaoguai-config::SchedulerSettings` grows a `file_watch`
sub-block:

```rust
pub struct FileWatchSettings {
    pub enabled: bool,                  // default false
    pub routes: Vec<FileWatchRoute>,    // static config routes
    pub load_routes_from_db: bool,      // default true
}

pub struct FileWatchRoute {             // shape-matches scheduler's type
    pub job_id: String,
    pub path: String,
}
```

`xiaoguai-config` doesn't depend on `xiaoguai-scheduler`, so we
mirror the scheduler's `FileWatchRoute` shape here and convert at
the binary layer. Same precedent as `SchedulerSinkSettings` storing
opaque `serde_json::Value` blocks.

`xiaoguai-core/src/main.rs::run_serve` ŌĆö extended the existing
v0.12.0 scheduler block to also start a `FileWatchSource` on the
same event channel as the webhook source. The new code lives right
after the `webhook_source` start call:

```rust
if settings.scheduler.file_watch.enabled {
    if let Err(e) = crate::scheduler_bridge::spawn_file_watch_source(
        &settings.scheduler.file_watch,
        jobs.as_ref(),
        event_tx,
    )
    .await
    {
        tracing::error!(error = %e, "serve: file_watch source bootstrap failed; continuing without it");
    }
}
```

`event_tx` is `.clone()`d on the webhook call so both sources push
into the same `mpsc::Receiver` the runner drains.

Route discovery merges two sources:

1. **Static config routes** (`[scheduler.file_watch].routes`) ŌĆö ops
   scenarios that shouldn't require a DB write. Operator edits
   `config.yaml`, restarts.
2. **DB-defined routes** ŌĆö `JobRepository::list_reactive()` returns
   every enabled job whose trigger is `Trigger::FileWatch`. Each
   becomes a `(job_id, path)` binding.

Static routes win on `(job_id, path)` conflict. Per-route
registration errors log + continue ŌĆö one bad path doesn't kill the
source.

**New trait method** `JobRepository::list_reactive()`. Default impl
returns `Vec::new()` so unrelated backends keep compiling.
`InMemoryJobRepository` filters by `j.enabled && j.trigger.is_reactive()`.
`PgJobRepository` pushes the discriminant filter into SQL via
`trigger->>'type' IN ('file_watch','webhook','git_push','db_poll')`
so we don't pull every enabled row through the app layer.

### 2. `RagClient::reindex_path`

`xiaoguai-rag::RagClient` gains:

```rust
async fn reindex_path(&self, _collection_id: &str, _path: &Path) -> RagResult<usize> {
    Err(RagError::Unsupported("reindex_path not implemented for this backend".into()))
}
```

Default impl returns the new `RagError::Unsupported` variant so the
caller can detect missing capability without parsing strings.
Existing backends compile unchanged.

`InMemoryRagClient::reindex_path` reads the file via
`tokio::fs::read_to_string`, ingests with
`source_uri = format!("file://{}", path.display())` so the v0.9.2
idempotency contract (replace on `source_uri` match) does the
right thing. Returns the chunk count (one per line for this toy
backend).

`R2RClient::reindex_path` reads the file the same way, posts via
the existing `ingest()` path. R2R's chunk_count is async (returned
on the initial response as 0); we return `n.max(1)` so callers see
"we touched the backend" rather than "no work done".

Binary file re-index (PDF / docx via R2R's multipart upload) is
deferred to v0.12.2.1.

`RagMcpAdapter::map_rag_err` learns the new variant ŌĆö Unsupported
maps to `McpError::InvalidArgument` (agent-visible "this isn't
supported" rather than a transport error suggesting a wire-level
problem).

### 3. `RagReindexExecutor` ŌĆö shipped, not yet wired

`xiaoguai-core/src/scheduler_bridge.rs::RagReindexExecutor`
implements `JobExecutor` for payloads shaped like:

```json
{"kind": "rag_reindex", "collection_id": "notes", "path": "/var/notes/topic.md"}
```

Returns an `ExecutionOutcome` whose `output_preview` reports
chunk count. Tested in isolation against `InMemoryRagClient`.

**NOT yet wired into the operator binary's executor selection.**
Production binaries still use `RuntimeJobExecutor` exclusively for
every scheduled job. The payload-dispatching `CompositeExecutor`
that routes `rag_reindex` payloads to `RagReindexExecutor` and
everything else to `RuntimeJobExecutor` is deferred to v0.12.2.1
(see Deferred below). The trait extension + impls are the
load-bearing pieces; the dispatching is a single match on
`job.payload.get("kind")`.

## Deferred (v0.12.2.1)

- **Payload-dispatching `CompositeExecutor`.** Today every
  scheduled job runs through `RuntimeJobExecutor`. The
  `RagReindexExecutor` ships and is tested, but no caller picks it.
  The simplest wiring: a `CompositeExecutor { runtime, rag }` that
  matches on `job.payload.get("kind")` and forwards. One file, one
  match arm, one extra `Arc` on `AppState` ŌĆö held back so v0.12.2
  stays focused on the seams (trait + impls + operator wiring) and
  v0.12.2.1 takes care of the production switch.
- **Binary file re-index via R2R multipart.** Today
  `R2RClient::reindex_path` posts via `raw_text`. PDFs / docx need
  the multipart path. Adapter-side; surfaces when a user reports a
  PDF watcher.
- **Per-collection RAG routing.** Today an operator wires one
  `RagReindexExecutor` per binary; multi-tenant deployments wanting
  per-tenant RAG endpoints need a `RagClientResolver` similar to
  v0.6.4's per-tenant `LlmRouter`. Not yet justified by usage.
- **Debounce.** v0.10.1 already noted `notify-debouncer-full` as a
  future option; relevant here because RAG re-index is expensive
  (chunking + embedding). A burst from `git checkout` could fire
  20 re-indexes in 200ms. Revisit if profiling shows it.
- **Write-capable first-party Obsidian connector.** The community
  `obsidian-mcp` is good enough for the v1.0 audience; a Rust-
  native connector with structured frontmatter writes + Dataview
  query awareness is a v1.1+ thing. Tracked in roadmap ┬¦3
  v0.12.2 row's deferral list.

## Design decisions worth flagging

- **Default trait impl returns `Unsupported`, not a panic.** The
  v0.9.2 contract said "backends that can't compute lines MUST
  remediate at ingest time" ŌĆö same pattern here. Backends opt in;
  silent fallback to a less-capable path would mask "your file
  watcher isn't actually re-indexing".
- **`list_reactive` is a new required-ish trait method with a
  default impl.** Same backward-compat trick as
  `RagClient::reindex_path`. Production backends override it; toy
  backends in tests inherit the empty `Vec`. Keeps the v0.10.0
  in-memory tests untouched.
- **Static config routes override DB-defined ones on conflict.**
  An operator who explicitly wrote a route into `config.yaml`
  probably means it; the DB row is the dynamic / convenient path.
  De-dup key is `(job_id, path)` ŌĆö different jobs watching the
  same path is intentional and both fire.
- **`spawn_file_watch_source` errors log + continue, don't bail
  the server.** Same pattern as the v0.6.5 audit reader ŌĆö a
  misconfigured file-watch path is operator config drift, not a
  fatal server condition. The `JobRunner` keeps serving cron jobs
  + webhook deliveries; only the missing source goes dark.
- **`RagReindexExecutor` lives in `xiaoguai-core`, not
  `xiaoguai-rag` or `xiaoguai-scheduler`.** Putting it in
  `xiaoguai-rag` would force a `xiaoguai-rag ŌåÆ xiaoguai-scheduler`
  dep (so it could see `JobExecutor`). Putting it in
  `xiaoguai-scheduler` would force a `xiaoguai-scheduler ŌåÆ
  xiaoguai-rag` dep. The binary already depends on both, so the
  bridge belongs there ŌĆö same shape as `WebhookSourceAdapter` and
  `PgSchedulerAuditAppender` already in the file.
- **Obsidian entry isn't a new slug ŌĆö it's a correction.** The
  v0.9.4 entry pointed at the right server but mis-declared its
  shape. A second `obsidian-rw` slug would have left the broken
  read-only entry in the catalog forever. One slug, accurate now.
