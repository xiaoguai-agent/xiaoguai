# v0.6.4 — audit endpoint + per-tenant LlmRouter

**Status:** done.

## What shipped

Two finishers off the v0.6.3 backlog land in the same tag because they
share the same theme: making the per-tenant pieces actually visible at
the request boundary.

1. **`GET /v1/admin/audit?tenant_id=...&limit=...&since=...&until=...`**
   — JSON-friendly view of the HMAC-chained audit log. Returns 503 when
   the operator hasn't wired the reader (the production wiring still
   needs an audit signing-key boot step); 400 when `tenant_id` is
   missing or empty; 200 with rows otherwise. Gated by the
   `system_admin` Casbin policy in auth-required mode.
2. **Per-tenant `LlmRouter`** — the `LlmBackend` impl on `LlmRouter` now
   reads the new optional `ChatRequest::tenant_id` field and builds a
   `ResolveCtx { tenant_id: Some(...) }` from it, so the v0.6.2
   tenant-default + tenant-fallback rules finally apply for live REST
   requests. The session handler sets the field from
   `session.tenant_id` on every turn.

## Pieces

### Backend: per-tenant router

* `ChatRequest` gains an optional `tenant_id: Option<String>` field.
  Backends ignore it; only the router consumes it. Storing the routing
  hint on the request keeps `LlmBackend` object-safe and unchanged
  (no breaking trait shift).
* `AgentConfig` gains a matching `tenant_id: Option<String>` propagated
  onto every `ChatRequest` built by the agent loop.
* `xiaoguai-api` `send_message` handler sets
  `cfg.tenant_id = Some(session.tenant_id)` after loading the session,
  so each REST turn now picks the per-tenant default. IM gateway path
  unchanged for v0.6.4 — adds onto the v0.7.3 PG-history work later.

### Backend: audit endpoint

* New module `xiaoguai-api::audit` with:
  - `AuditReader` async trait (`list(tenant_id, since, until, limit)`).
  - `AuditEntryView` wire shape (hex-encoded HMACs, JSON details).
  - `StaticAuditReader` in-memory impl for tests + dev.
* `AppState.audit: Option<Arc<dyn AuditReader>>` (parallel to the
  existing `tenants` / `rate_limiter` optional slots).
* `routes::admin::list_audit` handler wired at
  `GET /v1/admin/audit`, behind the existing rate-limit + Casbin +
  bearer-token layer stack.
* `ApiError` gains `InvalidRequest` (HTTP 400, code
  `invalid_request`) and `ServiceUnavailable` (HTTP 503, code
  `service_unavailable`).

### Frontend

* `@xiaoguai/shared` adds `TenantResponse`, `AuditEntryView`,
  `ListAuditQuery` wire types and `XiaoguaiClient.listTenants(...)` +
  `XiaoguaiClient.listAudit(q)`.
* `admin-ui`'s `AuditPane` swaps its placeholder for a live table:
  per-tenant input, refresh button, error banner, and an audit table
  (id / ts / actor / action / resource / last-8 HMAC).

## Tests

* `xiaoguai-llm/tests/router.rs`: two new tests prove the
  `LlmBackend` impl routes by `ChatRequest::tenant_id` and falls back
  to system defaults when the field is `None`.
* `xiaoguai-api/src/audit.rs`: two unit tests cover
  `StaticAuditReader` (tenant filter + time bounds + limit).
* `xiaoguai-api/tests/admin_and_listing.rs`: three new integration
  tests cover the `GET /v1/admin/audit` endpoint (200 with rows, 503
  when reader missing, 400 when tenant_id missing).

`cargo test --workspace`: **214 passed / 0 failed / 58 ignored**
(+7 over v0.7.3). `cargo clippy --workspace --all-targets -- -D
warnings`: clean. `cargo fmt --check`: clean. Frontend
`pnpm -r typecheck` + `pnpm -F chat-ui build` + `pnpm -F admin-ui
build`: clean.

## Deferred (v0.6.5+)

* Operator bootstrap to actually wire a `PgAuditSink` into
  `AppState.audit` in `xiaoguai-core/src/main.rs` — needs a signing-key
  config knob (today the only `AuditReader` impl that ships is
  `StaticAuditReader` for tests; production deploys see HTTP 503 from
  the audit endpoint until that lands).
* Chain integrity verification surfaced via a separate endpoint
  (`PgAuditSink::verify_tenant` exists in `xiaoguai-audit`; not exposed
  yet).
* IM gateway path through `LlmRouter` — `mount_feishu`'s agent build
  uses `state.app.agent_defaults.clone()` directly. Threading the
  per-conversation tenant resolved by v0.7.3's `PgImHistoryStore`
  through to a per-request `AgentConfig.tenant_id` is the natural next
  slice.
* Admin-ui filter UI (date range pickers, free-text search). The
  current pane shows the latest 100 rows for the typed tenant; that's
  enough to verify the chain end-to-end but not a polished operator
  view.
