# v0.12.0 — `xiaoguai-runtime` + PG scheduler repos + operator wiring + webhook HTTP route

**Status:** done (2026-05-24).

## Why this slice

After v0.11.x, three call sites all drive the same `ReactAgent` with
near-identical glue:

| Caller | File | What it does |
|---|---|---|
| REST | `crates/xiaoguai-api/src/routes/sessions.rs::send_message` | persist user msg → load history → `ReactAgent::run_stream` → SSE events → finalize task persists new msgs |
| IM | `crates/xiaoguai-im-gateway/src/router.rs::run_agent_and_reply` | snapshot history → append user → `ReactAgent::run_to_completion` → pick reply text → extend history with full new slice |
| Scheduler | `crates/xiaoguai-scheduler/src/executor.rs::EchoExecutor` | currently a stub; production agent executor is the v0.12.0 deliverable |

Each variant duplicates 30–80 lines of "build agent, run, find the
reply text, persist". v0.12.0 extracts that into one place and
rewires the three callers so a future bug fix in the loop lands once.

Roadmap §3 also bundles three other deferred items into this tag:

- **PG repository impls** for `scheduled_jobs` + `scheduled_job_runs`
  (migration `0007_scheduled_jobs.sql` already shipped in v0.10.0).
- **Operator wiring** in `xiaoguai-core/src/main.rs::run_serve`:
  spawn `JobRunner::run_loop` on a 30s ticker; pass PG repos when
  available, in-memory fallback when not.
- **Webhook HTTP route** in `xiaoguai-api`:
  `POST /v1/scheduler/webhooks/:route_id` →
  `WebhookSource::push(route_id, body)` so external pokes can fire
  reactive jobs.

Bundling them now means one code path lands once instead of three.

## Pieces

### New crate — `crates/xiaoguai-runtime/`

```
crates/xiaoguai-runtime/
├── Cargo.toml
└── src/
    ├── lib.rs        — RuntimeContext + entry points
    ├── outcome.rs    — RuntimeOutcome (AgentOutcome + reply_text + new_messages)
    ├── sink.rs       — RuntimeSink trait (event + finish hooks)
    └── error.rs      — RuntimeError
```

Public surface (final):

```rust
pub struct RuntimeContext {
    pub backend: Arc<dyn LlmBackend>,
    pub toolbox: Arc<Toolbox>,
    pub agent_config: AgentConfig,
}

pub struct RuntimeOutcome {
    pub stop_reason: StopReason,
    pub iterations: u32,
    /// Full conversation including the input history.
    pub messages: Vec<Message>,
    /// Messages produced *during this run* — i.e. everything after the
    /// last user message that matches the inbound prompt. Convenience
    /// for callers that want to persist only the delta.
    pub new_messages: Vec<Message>,
    /// Last assistant text message; empty if the loop produced none.
    pub reply_text: String,
}

#[async_trait]
pub trait RuntimeSink: Send + Sync {
    /// Called for each agent event as the loop streams.
    /// Default: no-op (caller doesn't want streaming).
    async fn on_event(&self, _event: &AgentEvent) -> Result<(), RuntimeError> {
        Ok(())
    }
    /// Called once after the loop terminates; gets full outcome.
    async fn on_finish(&self, _outcome: &RuntimeOutcome) -> Result<(), RuntimeError> {
        Ok(())
    }
}

/// One-shot: blocks until the loop terminates, returns enriched outcome.
pub async fn run_to_completion(
    ctx: &RuntimeContext,
    history: Vec<Message>,
    cancel: CancellationToken,
) -> Result<RuntimeOutcome, RuntimeError>;

/// Streaming: returns (join, stream) so the caller can wire SSE etc.
/// The join future yields the enriched outcome.
pub fn run_streamed(
    ctx: &RuntimeContext,
    history: Vec<Message>,
    cancel: CancellationToken,
) -> (
    JoinHandle<Result<RuntimeOutcome, RuntimeError>>,
    ReceiverStream<AgentEvent>,
);

/// Sink-driven: each event hits sink.on_event; finish hits sink.on_finish.
pub async fn run_to_sink<S: RuntimeSink>(
    ctx: &RuntimeContext,
    history: Vec<Message>,
    sink: S,
    cancel: CancellationToken,
) -> Result<RuntimeOutcome, RuntimeError>;
```

`RuntimeOutcome::reply_text` is computed by walking `messages` in
reverse and picking the first assistant message with non-empty
content (current REST + IM logic, lifted here).

`RuntimeOutcome::new_messages` is computed by finding the position
of the last user message whose `content` equals the *inbound*
prompt (passed at the top of `history`) and slicing from there.
Mirrors v0.7.4's `to_persist` logic.

Deps: `xiaoguai-agent`, `xiaoguai-llm`, `tokio`, `tokio-stream`,
`tokio-util` (for `CancellationToken`), `futures`, `async-trait`,
`thiserror`. **No** dep on `xiaoguai-api` / `xiaoguai-core` /
`xiaoguai-storage` — the runtime is the *common substrate*, not a
shim above them.

### Callers rewired

**REST (`xiaoguai-api/src/routes/sessions.rs`):**

`send_message` now calls `xiaoguai_runtime::run_streamed(&ctx,
history, cancel)`. The SSE stream wraps the `ReceiverStream<AgentEvent>`
unchanged. The finalize task awaits the join handle, reads
`outcome.new_messages`, and persists each via `MessageRepository`
(skipping the user message which we wrote up-front). Net change:
~20 lines deleted; `agent` construction, `find reply text` logic,
and direct `ReactAgent::run_stream` all migrate behind the runtime.

**IM (`xiaoguai-im-gateway/src/router.rs`):**

`run_agent_and_reply` now calls `xiaoguai_runtime::run_to_completion`.
The new-message persistence uses `outcome.new_messages` instead of
the v0.7.4 `rposition` hand-roll. Net change: ~30 lines deleted.

**Scheduler (`xiaoguai-scheduler/src/executor.rs`):**

`EchoExecutor` stays as the test/smoke executor. New
`RuntimeJobExecutor` lives in a new module
`crates/xiaoguai-scheduler/src/runtime_executor.rs` (gated behind
`xiaoguai-runtime` dep):

```rust
pub struct RuntimeJobExecutor {
    ctx: Arc<RuntimeContext>,
}

#[async_trait]
impl JobExecutor for RuntimeJobExecutor {
    async fn execute(&self, job: &ScheduledJob, _attempt: u32)
        -> Result<ExecutionOutcome, String>
    {
        let prompt = job.payload.get("prompt").and_then(...)
            .ok_or("payload.prompt missing")?;
        let cfg = self.ctx.agent_config.with_tenant(job.tenant_id.clone());
        let history = vec![LlmMessage::user(prompt)];
        let outcome = run_to_completion(...).await
            .map_err(|e| e.to_string())?;
        Ok(ExecutionOutcome {
            output_preview: truncate(outcome.reply_text, 500),
            session_id: None,  // scheduler jobs don't bind to a session yet
        })
    }
}
```

Per-job session binding (so an audit-first console can drill into
the scheduler-driven transcript) is **deferred to v0.12.1** — needs a
small "synthetic session per scheduled run" pattern that's tangential
to the runtime extraction itself.

### PG repository impls

New module `crates/xiaoguai-scheduler/src/pg_repository.rs`:

```rust
pub struct PgJobRepository { pool: PgPool }
pub struct PgJobRunRepository { pool: PgPool }
```

Both impl their respective traits using `sqlx` against the existing
`scheduled_jobs` / `scheduled_job_runs` tables (migration 0007).
RLS-aware: every query opens a tx and calls `set_config(
'app.current_tenant_id', $1, true)` via the shared `begin_tenant_tx`
helper from `xiaoguai-storage`. Reuses the trick from v0.6.1.

Add `xiaoguai-scheduler` Cargo dep on `xiaoguai-storage` (currently
just on `xiaoguai-types` + `xiaoguai-audit`) so the PG impl can call
into the shared tenant_ctx helper. The InMemory impls stay.

Tests: `tests/pg_repository_e2e.rs` testcontainers-based, marked
`#[ignore]` for CI fast-path consistency with the rest of the
storage e2e suite.

### Operator wiring (`xiaoguai-core/src/main.rs::run_serve`)

After `AppState` is built and the API future is started, add:

```rust
let (scheduler_handle, webhook_source) = if /* scheduler enabled */ {
    let jobs = Arc::new(PgJobRepository::new(pool.clone()));
    let runs = Arc::new(PgJobRunRepository::new(pool.clone()));
    let executor = Arc::new(RuntimeJobExecutor::new(runtime_ctx));
    let audit_appender = build_scheduler_audit_appender(...);  // wraps PgAuditSink
    let webhook = Arc::new(WebhookSource::new());
    let (tx, rx) = event_channel();
    webhook.start(tx.clone()).await?;
    // (Future) file_watch source too, when v0.10.1's deferred filewatch wiring lands.

    let runner = JobRunner::new(jobs, runs, executor, audit_appender)
        .with_options(RunnerOptions {
            max_jobs_per_tick: 32,
            max_retry_sleep_secs: 30,
            budget_limit_per_user_per_day: DEFAULT_PROACTIVE_BUDGET_PER_DAY,
        });
    let runner = Arc::new(runner);
    let runner_for_task = runner.clone();
    let handle = tokio::spawn(async move {
        runner_for_task.run_loop(rx, Some(Duration::from_secs(30))).await
    });
    (Some(handle), Some(webhook))
} else {
    (None, None)
};
```

Config: `[scheduler] enabled = true` (default `false`) +
`tick_interval_secs = 30` (default 30) added to `xiaoguai-config`.
Off by default so existing deployments don't change behavior.

The `webhook_source` ref is added to `AppState` so the new HTTP
route can call `webhook.push(...)`.

### Webhook HTTP route (`xiaoguai-api`)

New route in the admin/v1 router:

```
POST /v1/scheduler/webhooks/:route_id   body: any JSON
  → AppState.webhook_source.push(route_id, body).await
  → 202 Accepted { delivered: N }
  → 404 if no jobs bound to route_id (delivered == 0)
  → 503 if webhook_source is None
```

Behind admin auth like the other scheduler endpoints (we'll add
per-tenant tokens in v0.12.1; for v0.12.0 admin-auth is the minimum
viable gate).

Add `webhook_source: Option<Arc<WebhookSource>>` to `AppState`.

Tests: route test verifies 202 + delivered count, 404 on unknown
route, 503 when unwired.

### Audit appender for scheduler runs

Today `RecordingAuditAppender` + `NullAuditAppender` are the only
impls. The operator binary needs a real one that hits the same
`PgAuditSink` already wired in v0.6.5. Small adapter in
`crates/xiaoguai-core/src/scheduler_bridge.rs`:

```rust
pub struct PgSchedulerAuditAppender { sink: Arc<PgAuditSink> }
impl AuditAppender for PgSchedulerAuditAppender {
    async fn append(&self, entry: AuditEntry) -> Result<(), String> {
        self.sink.append_one(&entry).await.map_err(|e| e.to_string())
    }
}
```

(Or reuse the existing `PgAuditAdapter` from `audit_bridge.rs` if its
shape already covers `append`. Check during step 6.)

## Pieces actually shipped (delta vs the original plan)

The plan's "audit appender" section anticipated reusing
`PgAuditAdapter` from v0.6.5; in the end the scheduler audit
appender shape (`Result<(), String>`) didn't match the existing
adapter's `AuditReader`/`AuditVerifier` surface (`Result<T,
AuditError>`), so a tiny `PgSchedulerAuditAppender` ships in
`crates/xiaoguai-core/src/scheduler_bridge.rs` instead. Same
behaviour, smaller surface than reusing the v0.6.5 adapter would
have demanded.

The plan also called out a possible cycle when adding
`webhook_source: Option<Arc<WebhookSource>>` to `AppState`. The
cycle would have been
`xiaoguai-scheduler → xiaoguai-im-feishu → xiaoguai-im-gateway →
xiaoguai-api → xiaoguai-scheduler`. v0.12.0 ships a `WebhookPusher`
trait in `xiaoguai-api` (mirroring the v0.6.5 `AuditReader` and
v0.11.1 `TodayReader` patterns); the `WebhookSourceAdapter` in
`xiaoguai-core` is the production impl. `AppState` carries
`webhook_pusher: Option<Arc<dyn WebhookPusher>>` rather than the
concrete source, so the cycle stays broken.

## Deferred

- **Per-scheduled-run synthetic session.** The
  `scheduled_job_runs.session_id` column already exists (migration
  0007); v0.12.1 fills it so the audit-first console can drill into
  a scheduled run's transcript.
- **Per-tenant webhook auth tokens.** v0.12.0 ships admin-auth-gated
  webhook routes; per-tenant API keys deferred to v0.12.1 or later.
- **FileWatchSource operator wiring.** The crate already supports
  `FileWatchSource`; wiring it from a `[scheduler.file_watch]`
  config block lives in v0.12.2 alongside the incremental-RAG work.
- **Streaming progress for scheduled runs.** Today the scheduler
  doesn't surface intermediate `AgentEvent`s — only the final
  outcome reaches `ExecutionOutcome`. v0.11.1 console drills into
  the linked session for transcript; once v0.12.1 wires the synthetic
  session, the existing chat-ui transcript view covers this.
- **Per-route HTTP method customisation for webhooks.** v0.12.0 is
  `POST` only; some integrators (GitHub push events specifically)
  send `POST` so this is fine; the data variant `Trigger::GitPush`
  is data-only in v0.10.1.
- **Cancellation hook on `JobRunner::run_loop`.** Today cancellation
  is via task abort (per v0.10.1 design); a built-in token would be
  cleaner but adds zero expressive power. Revisit if an operator
  actually wants graceful drain semantics.

## Design decisions worth flagging

- **Runtime is a substrate crate, not a façade.** It does NOT
  depend on `xiaoguai-api` / `xiaoguai-storage` / `xiaoguai-core`
  — the dependency arrow points the other way. Callers wrap the
  runtime, not the reverse. This keeps the runtime trivially
  test-stubbable.
- **`RuntimeSink` has default no-op impls** for both methods so a
  caller that only cares about `on_finish` (or only `on_event`)
  ships zero boilerplate. The `JobExecutor` wrapper provides exactly
  one impl that ignores events and records `on_finish`.
- **`new_messages` slicing uses content-equality** for the user
  prompt, same trick as v0.7.4. Documented gotcha: if the user
  message gets dropped by the slide window (very long thread), the
  slice falls back to the full message tail. Same behavior as today;
  not a regression.
- **PG repos open one transaction per call** (the existing v0.6.1
  pattern). Batching the per-attempt insert + status update into one
  tx is a v0.12.0.x optimisation if profiling shows it matters.
- **Scheduler is off by default** in `xiaoguai-core` to preserve
  existing deployment behavior. Operators flip
  `XIAOGUAI_SCHEDULER__ENABLED=true` to opt in.
- **Reactive-source wiring picks WebhookSource only** in v0.12.0.
  `FileWatchSource` waits for v0.12.2 so the slice stays focused;
  the wiring shape is identical, so a future addition is one
  function call long.
