# v0.9.4.1 — `McpSupervisor` live-pickup on marketplace install

**Status:** done.

## What shipped

Closes the last v0.9.4 deferral. Marketplace installs now reach the
running supervisor so the freshly added server is reachable without a
process restart.

1. **`McpSupervisor::reload_from_db(&dyn McpServerRepository, tenant)`.**
   Diffs the persisted manifest against the live client set: new rows
   get spawned via the matching transport, vanished rows get stopped.
   Cheap to call repeatedly — idempotent on the no-op path.
2. **Install handler calls reload.** `POST /v1/mcp/marketplace/install`
   checks `AppState.mcp_supervisor` and (when wired) invokes
   `reload_from_db` after the DB write succeeds. Spawn failures are
   logged but don't fail the install — the DB row is the source of
   truth, and a missing upstream binary is the operator's problem to
   fix once.
3. **`AppState.mcp_supervisor: Option<Arc<McpSupervisor>>`.** New
   optional slot parallel to `mcp_servers`. Default `None` so existing
   handlers / tests are unaffected; `xiaoguai-core` always wires
   `Some(Arc::new(McpSupervisor::new()))` in production.

## Pieces

### `xiaoguai-mcp::supervisor`

* `reload_from_db` implementation:
  - Pulls `list_global` or `list_for_tenant` depending on scope.
  - Filters `enabled = true` rows.
  - Diffs against `running_servers` under the live lock.
  - Stops any in-scope key not in the desired set (other tenants'
    clients stay live — caller decides their reconciliation cadence).
  - Calls a private `spawn_client(&server)` for each new key. Best-
    effort: spawn failures log a warning and continue (one bad row
    shouldn't sink the whole reload).
* `spawn_client` covers `McpTransport::Stdio` today (the only
  transport the v0.9.4 marketplace catalog uses). Http / Sse return
  `McpError::InvalidArgument("… supervisor pickup not implemented
  yet")` so future marketplace entries flag clearly. The v0.9.0
  `HttpMcpClient::new` needs per-call auth/header config that the
  `mcp_servers` schema doesn't capture; rather than over-fit the
  schema now, the http path is deferred.

### `xiaoguai-api`

* `AppState.mcp_supervisor: Option<Arc<McpSupervisor>>` (new field).
* `marketplace::install_from_marketplace` now invokes
  `supervisor.reload_from_db(repo, tenant)` after the row is written,
  catching + logging any reload error so the install handler still
  returns 200.

### `xiaoguai-core`

* `run_serve` constructs the supervisor alongside the rest of the
  state and threads it into `AppState.mcp_supervisor`.

### Test fixtures

* Every existing AppState construction (8 test files + 1 binary) gains
  `mcp_supervisor: None`. No behaviour change for tests that don't
  care; the new marketplace e2e case opts in.

## Tests

`xiaoguai-mcp/tests/supervisor.rs::reload_from_db`:
* `picks_up_new_row_and_stops_removed_one` — 3-step reconciliation
  exercises start-on-insert + skip-already-running + stop-on-delete.
* `disabled_rows_are_not_started`.
Both use the existing `mock-mcp-server` fixture binary so the test
stays Docker-free and the spawn path matches production.

`xiaoguai-api/tests/marketplace_e2e.rs`:
* `install_invokes_supervisor_reload_when_wired` — drives the live
  HTTP install handler with a supervisor wired into AppState. The
  marketplace catalog's `fetch` entry tries to spawn `uvx`, which
  isn't on CI's PATH; the install still returns 200 (handler swallows
  spawn errors) and the supervisor was reached (no panic). The strong
  guarantees about the started set live in the `xiaoguai-mcp` test
  above where the binary is controlled.

`build_state_with_supervisor` test helper composes a state with both
mcp repo + supervisor without forcing every marketplace test to pass
both.

## Deferred

* **Http / Sse transport spawn from `mcp_servers` rows.** The schema
  needs to grow per-row auth + custom-header columns first; the
  marketplace catalog ships stdio entries only today, so deferring
  doesn't block any user-visible workflow.
* **Periodic reconciliation timer.** Today reload runs only on
  install. A drift detector that ticks every N seconds (e.g. catches
  manual `DELETE FROM mcp_servers`) lands when the v0.12.0 runtime
  extraction needs cross-replica consistency.
* **Reload result surfaced in the install response.** Today the spawn
  failure goes to logs only — the response body is just `{id, slug,
  name}`. A `{install_status: "live" | "queued" | "spawn_failed",
  message?: string}` extension would let the admin-ui pane render a
  yellow vs green dot per row.
* **Tenant fan-out.** Install with `tenant_id` reloads only that
  tenant's scope (system-wide + that tenant's rows). Installs without
  `tenant_id` reload the global scope. Multi-tenant batch reload (e.g.
  after a marketplace catalog version bump) would need a separate
  "reload all tenants" admin endpoint.
