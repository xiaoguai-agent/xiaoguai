# v1.1.7 ‚Äî pip wheel for the `xiaoguai` CLI

**Status:** done (2026-05-24, branch `feat/c8-pip`, unpushed).

## Why this slice

Roadmap ¬ß3.v1.1 ("distribution surface area") asks for more than
one install channel. The Rust crate ships through `cargo install`;
container images and SBOMs ship through `release.yml`. The C7
worktree adds standalone tarballs. This worktree adds the
`pip install xiaoguai` route.

Python is the dominant install vector for the agent-curious crowd
(LangChain / AutoGPT / MCP-client users), even when the actual
runtime is native. A `pip` install needs zero Rust toolchain on the
end-user's machine. The cost is one extra CI matrix and a small
Python launcher.

The launcher carries no agent logic ‚Äî native bindings via PyO3
remain a v1.2+ scope decision (`python/xiaoguai` is positioned to
absorb them later without an import-path break).

## Pieces

### 1. `python/xiaoguai/` package

```
python/xiaoguai/
‚îú‚îÄ‚îÄ pyproject.toml
‚îú‚îÄ‚îÄ README.md
‚îú‚îÄ‚îÄ LICENSE                       (symlink to repo root LICENSE)
‚îú‚îÄ‚îÄ src/xiaoguai/
‚îÇ   ‚îú‚îÄ‚îÄ __init__.py               (exposes __version__)
‚îÇ   ‚îú‚îÄ‚îÄ __main__.py               (the console-script entry point)
‚îÇ   ‚îú‚îÄ‚îÄ _binary.py                (find bundled native binary)
‚îÇ   ‚îú‚îÄ‚îÄ _version.py               (reads $XIAOGUAI_VERSION at build)
‚îÇ   ‚îî‚îÄ‚îÄ _binary/                  (populated by CI per platform)
‚îî‚îÄ‚îÄ tests/
    ‚îú‚îÄ‚îÄ conftest.py
    ‚îî‚îÄ‚îÄ test_invokes_binary.py
```

**Launcher design.** `__main__.main()` resolves the bundled binary
via `_binary.find_binary()` and `execvp`s into it on POSIX
(replacing the Python interpreter so signal handling stays clean
and the process tree stays flat). Windows falls back to
`subprocess.run` because `execvp` semantics differ there ‚Äî out of
scope for v1.1.7 either way.

**Missing-binary behaviour.** When `_binary.find_binary()` returns
`None` (e.g. an editable install from a fresh checkout) we print a
two-paragraph diagnostic to stderr and exit 2. The message points
at the three remedies: install a published wheel, build from source,
or stage a binary manually. `XIAOGUAI_PY_DEBUG=1` shows the
resolution path the launcher tried.

**Version sourcing.** `_version.py` reads `XIAOGUAI_VERSION` from
the environment with a `"0.0.0+local"` fallback. CI exports
`XIAOGUAI_VERSION=<tag-without-v>` before invoking `cibuildwheel`
so the wheel metadata matches the git tag without requiring a
`setup.py` rewrite per release.

**Package-data glob.** `pyproject.toml` declares
`xiaoguai = ["_binary/**/*"]` under `[tool.setuptools.package-data]`.
CI stages the binary under
`src/xiaoguai/_binary/<target-triple>/xiaoguai` before
`cibuildwheel` runs; the glob picks it up.

### 2. Target matrix

| Target triple                  | Runner               | cibw archs |
|--------------------------------|----------------------|------------|
| `aarch64-apple-darwin`         | `macos-14` (M-series)| `arm64`    |
| `x86_64-apple-darwin`          | `macos-13` (Intel)   | `x86_64`   |
| `x86_64-unknown-linux-gnu`     | `ubuntu-latest`      | `x86_64`   |
| `aarch64-unknown-linux-gnu`    | `ubuntu-24.04-arm`   | `aarch64`  |

Each target runs on its native runner; no cross-compilation or
QEMU emulation. Linux wheels use the `manylinux_2_28` base image
(matches the existing release.yml floor).

### 3. `.github/workflows/pip-wheel.yml`

NEW file (not modifying `release.yml`, `rust.yml`, or any other
existing workflow ‚Äî C7 and C9 sibling worktrees also touch
`.github/workflows/`, keeping changes in separate files avoids
merge conflicts).

Triggers: `push` on `v*` tags + `workflow_dispatch` with an optional
version override (handy for testing without cutting a tag).

Per-target job:

1. Checkout.
2. Compute `XIAOGUAI_VERSION` from the tag (or override).
3. Install Rust 1.88.0 + cache.
4. `cargo build --release --target <triple> -p xiaoguai-cli`.
5. Copy the binary to
   `python/xiaoguai/src/xiaoguai/_binary/<triple>/xiaoguai`.
6. `cibuildwheel` (cp310-only, no PyPy / musllinux / win32).
7. Inspect the wheel to confirm `_binary/<triple>/xiaoguai` is
   actually inside it (the painful failure mode is "wheel built,
   binary missing" ‚Äî catch it in CI rather than at install time).
8. Upload wheels as `wheels-<triple>` artifacts.

`publish` job present but gated `if: false` with a TODO pointing at
PyPI trusted-publisher setup. Token + workflow_id config deferred.

### 4. Tests

`tests/test_invokes_binary.py`:

- **`test_target_triple_resolves_on_supported_platform`** ‚Äî unit
  test that `_binary.current_target_triple()` returns a known
  triple on macOS / Linux. Catches drift between the workflow
  matrix and the launcher dispatch.
- **`test_missing_binary_prints_helpful_error`** ‚Äî unit test that
  monkey-patches `find_binary -> None` and asserts the launcher
  exits with code 2. Locks in the user-facing failure mode.
- **`test_module_invocation_prints_help`** ‚Äî integration test that
  shells out to `python -m xiaoguai --help` and looks for "chat"
  in the combined stdout/stderr (which `xiaoguai --help` emits as
  a subcommand name). Skipped when no binary is bundled, so
  `pytest` from a fresh checkout doesn't fail noisily.
- **`test_console_script_help`** ‚Äî same shape via the `xiaoguai`
  PATH entry from the installed wheel.

## Deferred (v1.1.7.1+)

- **Native Python bindings via PyO3.** Huge scope. Wraps
  `xiaoguai-agent::ReactAgent` + `xiaoguai-llm::LlmBackend` so
  Python callers can drive the agent loop in-process rather than
  through a subprocess. Probably its own crate
  `xiaoguai-pybind` + a separate `xiaoguaipy` PyPI package to
  keep this `xiaoguai` wrapper simple. v1.2.
- **PyPI publishing automation.** Workflow scaffold present
  (`publish` job, `if: false`). Activating it needs PyPI account
  ownership, trusted-publisher OIDC config, and a dry-run on
  test.pypi.org. v1.1.8.
- **conda-forge package.** A `meta.yaml` recipe under
  `python/conda-recipe/` would let `conda install xiaoguai` work.
  v1.1.9 ‚Äî depends on PyPI publishing landing first because
  conda-forge bots watch PyPI for upstream releases.
- **Windows wheel.** Adds `x86_64-pc-windows-msvc` to the matrix.
  Launcher already has a Windows branch (`subprocess.run` instead
  of `execvp`); the gap is signing + the Rust binary building
  cleanly on Windows. v1.1.10 if any user asks.
- **musllinux (Alpine) wheel.** Adds
  `x86_64-unknown-linux-musl`. Static-link `xiaoguai-cli`'s
  OpenSSL dependency or switch to rustls ‚Äî separate work item.
- **Universal2 macOS wheel.** Currently we ship two separate
  arm64 / x86_64 wheels. `universal2` would halve the matrix
  entries at the cost of bundle size + harder code signing per
  arch. Not justified by demand.
- **Source distribution (`sdist`).** Today only wheels ship. An
  `sdist` would require carrying a Rust toolchain in the install
  path (or vendoring pre-built binaries via package-data, which
  defeats the point). Skip indefinitely.

## Design decisions worth flagging

- **`execvp` over `subprocess` on POSIX.** Keeps the process tree
  flat so `Ctrl+C`, job control, and OOM handling reach the Rust
  binary cleanly. The Python interpreter is fully replaced ‚Äî no
  wrapper overhead, no orphaned PIDs.
- **`XIAOGUAI_VERSION` env var instead of dynamic version from
  git.** setuptools' `setuptools-scm` would also work, but env-var
  injection means the CI workflow is the single source of truth
  for "what version is this wheel". Matches how the Cargo
  workspace handles its `version.workspace = true`.
- **Per-target subdirectory under `_binary/`, not a flat layout.**
  A future universal2 macOS wheel can drop two binaries side by
  side (`aarch64-apple-darwin/xiaoguai` +
  `x86_64-apple-darwin/xiaoguai`) and the launcher picks the
  right one at runtime. Flat layout would have made that mutually
  exclusive.
- **CI matrix uses native runners, not QEMU.** macOS arm64 (M1)
  and Linux aarch64 are GitHub-hosted now (`macos-14`,
  `ubuntu-24.04-arm`). QEMU cross-build worked when those runners
  didn't exist; today it's just slower + less reliable.
- **`cibuildwheel` `build-frontend = "build"` instead of `pip`.**
  Subtle: with `pip` as the frontend, cibuildwheel will try to
  re-resolve dependencies inside the manylinux container, which
  in airgapped CI lanes fails. `build` uses the PEP 517 frontend
  directly with no resolution step.
- **`cp310-*` only despite the Python tag not mattering for a
  binary launcher.** cibuildwheel doesn't produce `py3-none-<plat>`
  wheels natively. cp310 is the lowest interpreter we support;
  the wheel will install on cp311 / cp312 via the manylinux
  back-compat rules.
- **Symlink LICENSE instead of copying.** Keeps the licence text
  single-sourced; setuptools follows symlinks during sdist /
  wheel construction. The git tree carries the link, not a
  duplicated file.
- **`publish` job present but `if: false`.** Carrying the
  plumbing in-tree means the day we get PyPI access we flip one
  line. Carrying it as a TODO comment instead would mean
  re-inventing the publish step from scratch later.
