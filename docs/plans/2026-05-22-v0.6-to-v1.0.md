# v0.6 → v1.0 — carve v1.0 into shippable slices

The architecture doc's roadmap (`docs/architecture/2026-05-21-design.md` §15)
jumps v0.5 → v1.0 in one breath. Reality: v1.0 has Chat UI + Admin UI + IM
gateway + packaging + compliance — each its own session. v0.6/0.7/0.8 are
intermediate slices that let us stop at any one of them with a coherent
demo.

## v0.6 — backend hardening — **DONE 2026-05-22 (tag `v0.6`)**

**Goal:** lift the v0.5.5 API from "minimum viable" to "auth-aware,
multi-tenant-correct, with a real MCP listing endpoint that downstream
clients (Chat UI in v0.8) can consume".

**Shipped:**
- `xiaoguai-api::auth::TokenValidator` trait + `Claims` + `StubValidator`
  + `JwtTokenValidator<JwtValidator>` adapter. Axum middleware
  `require_bearer` 401s missing/invalid tokens and injects `Claims` as a
  request extension. Engaged when `AppState.auth = Some(_)`; healthz stays
  public always.
- `AppState` gains `mcp_servers` (Option) + `auth` (Option).
- `POST /v1/sessions` honors claims over body identity so authed clients
  can't spoof `user_id`/`tenant_id`.
- `GET /v1/mcp/servers` — globals for anonymous callers; globals +
  tenant rows for authed callers. Returns 500 with a clear message when
  no registry is wired in.
- `xiaoguai-core serve` wires `PgMcpServerRepository` into the state.

**Deferred to v0.6.1:**
- WebSocket fallback (`/v1/sessions/:id/ws`).
- OpenAPI spec via `utoipa` (would touch every handler annotation).
- `/v1/admin/tenants` + `/v1/admin/audit`.
- Per-request RLS context: `SET LOCAL app.current_tenant_id = $1` on each
  txn. Needs reshaping the Pg repo trait methods to take a context.
- Real `JwtValidator` engaged in `xiaoguai-core serve` (waiting on a
  config switch + JWKS endpoint).
- Casbin role enforcement on a per-route basis.

**Acceptance:** 144 → **152** passed / 0 failed / 51 ignored. `clippy`
+ `fmt --check` clean.

## v0.7 — IM gateway + Feishu minimum

**Goal:** make Xiaoguai reachable from a real chat surface so that users
without a frontend get a meaningful demo. Slice from v1.0 "IM gateway +
飞书 adapter".

### Tasks (outline)

- **T1:** `xiaoguai-im-gateway` crate concrete shape — `ImProvider` trait
  (`verify_signature`, `parse_event`, `reply`), `IncomingMessage` /
  `OutgoingReply` types, `route(event, state)` that maps to a Xiaoguai
  session + spawns ReactAgent.
- **T2:** `xiaoguai-im-feishu` impl of `ImProvider` — HMAC signature
  verification, parse Feishu v2 event JSON, post reply via Feishu OpenAPI
  send-message endpoint. Tenant + user id derived from `open_id`.
- **T3:** Axum `POST /v1/im/feishu/webhook` route mounted under the same
  app state — verifies, dispatches, returns 200 immediately while the
  agent runs async; reply is pushed back via Feishu OpenAPI when done.
- **T4:** Tests with fixture event payloads + stub Feishu HTTP transport.

### Deferred to v0.7.1

- Interactive cards (rich messages w/ tool-call viz).
- Long-task handoff (replies > Feishu's hard size cap).
- 钉钉 + 企微 adapters.
- Rate-limit per channel.

## v0.8 — Chat UI MVP

**Goal:** ship a browser surface that can drive a session end-to-end so
the project becomes screencast-able. Slice from v1.0 "Chat UI".

### Tasks (outline)

- **T1:** `frontend/` Vite + React + TS scaffold; `npm-workspaces`
  optional.
- **T2:** API client (`src/api.ts`) — typed wrappers over our 6 endpoints
  + SSE consumer using browser `EventSource`.
- **T3:** Pages:
  - `/` — session list + "new session" button.
  - `/sessions/:id` — chat surface: message list (text + tool-call
    blocks), input box, send → SSE-render the assistant turn live.
- **T4:** Minimal styling (Tailwind utility classes; no full design
  system yet).
- **T5:** README in `frontend/` with `npm run dev` + `npm run build`
  instructions.

### Deferred to v0.8.1

- MCP studio (browse tools, dry-run).
- Admin UI (tenant / RBAC / audit panes).
- Auth flow (OIDC) in the browser — depends on v0.6.1.
- Multi-modal blocks (images, files).

## v0.9 — Admin UI + packaging skeleton

**Goal:** the operator-facing pieces so v1.0 is one polish-pass away.

### Tasks (outline)

- **T2:** `docker-compose.yml` for the canonical stack (xiaoguai-core +
  Postgres + Valkey + Ollama + optional MCP fs server).
- **T3:** Multi-stage `Dockerfile` for `xiaoguai-core` + `xiaoguai` CLI.
- **T4:** Smoke test that brings the stack up via testcontainers-rs and
  hits `/healthz` + `POST /v1/sessions`.

## v1.0 — production polish

**Goal:** the final hardening pass before a public release.

### Tasks (outline)

- **T1:** Helm chart skeleton (values for replicas, ingress, secrets refs).
- **T2:** Bare-metal tarball + systemd unit + uninstall script.
- **T3:** SBOM (cargo-sbom / cyclonedx) + cosign signing of release images.
- **T4:** Multi-arch image builds (amd64 + arm64) via GHA buildx.
- **T5:** 等保 2.0 三级 self-check checklist + GDPR DPIA template skeleton
  under `docs/compliance/`.
- **T6:** Full docs — `docs/user-guide/quickstart.md`, `runbooks/`,
  `docs/api/` (mostly auto-generated from utoipa once v0.6.1 lands).

v0.6/0.7/0.8/0.9 are stepping stones that each leave the tree in a
runnable state; v1.0 is the only "no green-fields" milestone — it
operationalises everything we already built.
