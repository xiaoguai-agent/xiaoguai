# v0.10.2 — proactive triggers

**Status:** done.

## What shipped

Third P1 slice from the v0.9–v0.12 roadmap, and the differentiating
piece per roadmap §3 — "the piece no OSS platform ships well". v0.10.0
was passive (Cron / Interval). v0.10.1 added reactive (FileWatch /
Webhook / GitPush / DbPoll). v0.10.2 closes the
`passive → reactive → proactive` ladder: a job can now describe a
**check-prompt** that a cheap model evaluates on every interval, and
the executor only fires when the check returns a non-empty reason.

Two non-negotiables from roadmap §5.5 land with this tag:

1. **Per-user-per-day push budget**, default **3**, configurable.
   Exhausting it produces a `scheduler.proactive_denied` audit row so
   the v0.11.1 audit-first console can show *why* a tenant stopped
   getting pings — not just that they did.
2. **Reason field on push payloads.** Scheduled / reactive fires
   populate it with empty string by default; proactive fires populate
   it with the checker's verdict. v0.10.3 sinks refuse delivery when
   `payload.reason.is_empty()` *and* the originating trigger was
   `Trigger::Proactive` (the field is `#[serde(default)]` so older
   persisted rows decode cleanly).

The runner extension is surgical: the v0.10.1 fire-event /
fire-now / tick path is unchanged for non-proactive triggers. The only
new branch in `JobRunner::fire` is "if this is a proactive trigger,
route through `proactive_gate` before the executor loop and either
skip cleanly or carry the reason forward into every audit row + push
payload for the fire".

## Pieces

### `Trigger::Proactive` variant

```rust
Trigger::Proactive { check_prompt: String, interval_secs: u64 }
```

Constructor `Trigger::proactive(prompt, secs)` rejects empty prompt
and zero interval. Proactive triggers are
[`Trigger::is_scheduled`] (they tick on `interval_secs`, so
`JobRepository::list_due` finds them) but also
[`Trigger::is_proactive`] (so the runner knows to route through the
checker first). `next_fire_after` advances by `interval_secs`
identically to `Interval`.

Serde shape (matches the rest of the enum's snake_case tagged form):

```json
{ "type": "proactive", "check_prompt": "Is there anything new?",
  "interval_secs": 1800 }
```

### `crate::proactive` — checker seam

```rust
trait ProactiveChecker {
    async fn should_fire(
        &self,
        prompt: &str,
        ctx: ProactiveCtx,
    ) -> Result<Option<String>, ProactiveError>;
}
```

`ProactiveCtx { job_id, tenant_id }` is the narrow context the
checker sees — deliberately not the full `ScheduledJob` row so the
v0.11.2 cheap-model wiring doesn't grow a dependency on the
repository.

Three test impls ship: `NeverFireChecker` (production default until
the cheap model is wired — fail-safe), `AlwaysFireChecker { reason }`,
`ScriptedChecker` (FIFO queue of decisions for the
say-no-then-yes-then-error tests).

### `crate::budget` — `BudgetLedger`

```rust
trait BudgetLedger {
    async fn check_and_debit(&self, user_id: &str, day: NaiveDate)
        -> Result<bool, BudgetError>;
    async fn remaining(&self, user_id: &str, day: NaiveDate)
        -> Result<u32, BudgetError>;
}
```

`InMemoryBudgetLedger::new(limit_per_day)` +
`InMemoryBudgetLedger::with_default_limit()` (3/day, hard-coded
constant `DEFAULT_PROACTIVE_BUDGET_PER_DAY = 3` so the roadmap
default is grep-discoverable). Key is `(user_id, day)`; user_id is
`job.tenant_id` (falling back to `"system"` — the same string the
audit module uses for unscoped rows so the budget and audit log key
identically).

`check_and_debit` is atomic under the `parking_lot::Mutex` guarding
the in-memory map — the runner relies on this for the
"first-tick-wins, second-tick-denied" semantics.

### `PushPayload.reason: String`

New field with `#[serde(default)]`. The runner populates it with the
checker's reason for proactive fires; empty for everything else. v0.10.3
sinks read it. A regression test (`payload_decodes_without_reason_field_for_back_compat`)
locks the back-compat behaviour for v0.10.0/v0.10.1 persisted rows.

### `JobRunner` extensions

- `with_proactive_checker(Arc<dyn ProactiveChecker>)` — chainable
  builder; absent ⇒ proactive jobs skip silently (fail-safe).
- `with_budget_ledger(Arc<dyn BudgetLedger>)` — chainable; absent ⇒
  proactive jobs skip silently even when the checker says yes (also
  fail-safe — roadmap §5.5 is non-negotiable).
- `RunnerOptions::budget_limit_per_user_per_day` (default 3) for the
  operator binary that auto-builds the in-memory ledger from the
  config knob in a later tag.
- `JobRunner::fire` gains one branch up front: if `trigger.is_proactive`,
  call `proactive_gate(job, prompt, extra_detail)`. On `Skip`, advance
  the schedule and return without touching the executor / runs /
  audit / sinks. On `Fire(reason)`, fall through to the existing
  attempt loop but thread `Some(&reason)` into `write_audit` and
  `push_to_sinks`.
- `proactive_gate` produces a `scheduler.proactive_denied` audit row
  on the budget-exhausted branch (with `outcome: "budget_exhausted"`
  and the still-set `proactive_reason`), so the v0.11.1 console has
  the data it needs to render "3 of 3 pushes used today — Alice".
  Other skip paths (checker says no, checker errors, missing
  checker/ledger) suppress audit entirely so a fast tick interval
  doesn't saturate the audit log.

### Re-exports from `lib.rs`

```rust
pub use budget::{
    BudgetError, BudgetLedger, InMemoryBudgetLedger,
    DEFAULT_PROACTIVE_BUDGET_PER_DAY,

pub use proactive::{
    AlwaysFireChecker, NeverFireChecker, ProactiveChecker,
    ProactiveCtx, ProactiveError, ScriptedChecker,
};
```

## Deferred

- **PG-backed `BudgetLedger`.** The in-memory ledger is the
  production contract; the PG impl pairs naturally with the v0.12.0
  scheduler PG repos (same migration adds the
  `scheduler_proactive_budget` table keyed on `(user_id, day_utc)`).
- **Cheap-model `ProactiveChecker` impl.** Until v0.11.2 wires a real
  Haiku / 3B-local checker, the production default stays
  `NeverFireChecker` (fail-safe). The trait is the contract; the
  backend swap is a single `.with_proactive_checker(Arc::new(...))`
  call in the operator binary.
- **Per-tenant budget overrides.** Today the limit is global. A
  tenant-aware ledger (`pro` plan gets 10/day, `free` gets 3) is
  trivial to add against the same trait — out of scope for v0.10.2.
- **Local-time day boundary.** The ledger keys on UTC `NaiveDate`.
  Tenants who want their "day" to start at midnight local time will
  notice the cap reset is offset. Out of scope; the obvious fix is a
  per-tenant timezone column once we have multi-region operators.
- **Sink-side reason enforcement.** Sinks currently *can* read
  `payload.reason` but v0.10.2 doesn't ship any real sinks. The
  reason-required rule lands with the real sinks in v0.10.3 (Feishu /
  Telegram / Email / Inbox).
- **Admin-ui budget dashboard.** `BudgetLedger::remaining` exists
  precisely so v0.11.1 can render "Alice has 2/3 left today". Not in
  scope for this tag — no admin-ui changes.

## Design decisions worth flagging

- **Proactive is `is_scheduled() = true`.** The natural alternative
  would be to make proactive its own third category alongside scheduled
  and reactive, but it ticks on a wall-clock interval just like
  `Interval` — the only difference is the runner-side fork. Keeping it
  scheduled means `JobRepository::list_due` finds it without
  modification, and the PG impl gets the same `WHERE jsonb_extract_path_text(
  trigger, 'type') IN ('cron', 'interval', 'proactive')` predicate in
  v0.12.0.
- **`is_proactive()` is a separate predicate.** Lets the runner
  switch on "needs checker gate" without re-matching the enum (matters
  when the PG row's `trigger` column is decoded once and inspected by
  several call sites).
- **Fail-safe wiring everywhere.** No checker installed ⇒ proactive
  jobs tick but don't fire. No ledger installed ⇒ same. Checker
  returns `Err` ⇒ skip silently this tick. Roadmap §5.5 demands the
  budget; a misconfigured deployment must not be able to bypass it,
  even by leaving a piece out.
- **Schedule advances on every tick regardless of fire decision.**
  Otherwise a proactive job whose checker says no for a while would
  stay "due" forever and burn list_due slots. The audit chain
  preserves the actual history; the schedule tracks "when do I ask
  again".
- **`scheduler.proactive_denied` is its own audit action.** Splitting
  it from `scheduler.job_run` means the v0.11.1 console can filter
  exhaustion events into a separate pane (and operators can alarm on
  them without false-positives from real job failures).
- **No audit row when the checker says no.** A 60-second proactive
  trigger answering "no" 99% of the time would produce 1440 audit
  rows/day for one job. The audit chain is for the *fired-or-denied*
  decisions, not the "checked and nothing to do" cases. If diagnostic
  visibility into the no-fire path matters later, the checker can log
  on its own — same pattern as the `LoggingSink` is to push.
- **`user_id` = `tenant_id` (with `"system"` fallback).** Matches the
  rest of the audit model. A future "per-user-within-tenant" split is
  additive: bump the `BudgetLedger::check_and_debit` key to
  `(tenant_id, user_id, day)` and update the wiring; trait signature
  change only.
- **`#[allow(clippy::too_many_lines)]` on `JobRunner::fire`.**
  111-line function after the proactive branch. Splitting the attempt
  loop into a helper would either fragment the run-row lifecycle into
  two functions (bad — the lifecycle is one transaction conceptually)
  or pass 6+ flags between them (also bad). The branch is cohesive
  and reads top-to-bottom; the lint is wrong here.
