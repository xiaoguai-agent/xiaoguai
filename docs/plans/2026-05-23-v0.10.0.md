# v0.10.0 — `xiaoguai-scheduler` crate

**Status:** done.

## What shipped

First P1 slice from the v0.9–v0.12 roadmap. xiaoguai now has a
scheduler primitive that owns the `Trigger × Agent × Audit` seam
the rest of the P1 band builds on (reactive triggers in v0.10.1,
proactive triggers in v0.10.2, push sinks in v0.10.3).

Roadmap principle (§3): platforms with strong triggers (n8n) are
weak on agents; platforms with strong agents (every chat-first
competitor) are weak on triggers. Owning both behind one runtime
is the differentiation. v0.10.0 lays the substrate without locking
us out of either v0.10.1's reactive variants or v0.12.0's
`xiaoguai_runtime::run_agent_to_sink` extraction.

## Pieces

### New crate — `crates/xiaoguai-scheduler/`

- `trigger.rs` — `Trigger` enum with `Cron { expr }` and
  `Interval { secs }` variants. Internally-tagged serde for
  human-friendly JSON in `scheduled_jobs.trigger`. Cron parsed via
  the `cron` crate (6-field, UTC). `next_fire_after(after)` returns
  the next fire strictly after the given instant.
- `retry.rs` — `RetryPolicy` (max_attempts + exponential backoff +
  max_backoff_secs cap). `delay_before_attempt(n)` is `Some(ZERO)`
  for attempt 1, exponential for attempts 2..=max, `None` past max.
- `job.rs` — `ScheduledJob` + `JobRun` + `JobRunStatus`
  (`pending|running|succeeded|failed|cancelled`). Mirrors the PG
  schema 1:1.
- `repository.rs` — `JobRepository` + `JobRunRepository` traits +
  `InMemory*` impls. Production PG impls land with the runtime
  extraction in v0.12.0.
- `executor.rs` — `JobExecutor` trait + `EchoExecutor` (test).
  Production wiring will hand in a real agent-loop executor; the
  trait keeps tests deterministic.
- `audit.rs` — `AuditAppender` trait (just the `append` method
  taking an `xiaoguai_audit::AuditEntry` so the production shim
  wraps `PgAuditSink` 1:1). `NullAuditAppender` + recording test
  impl.
- `sink.rs` — `PushSink` trait + `PushPayload` + `LoggingSink` stub.
  Real sinks (Feishu / Telegram / Email / chat-ui inbox) land in
  v0.10.3 against the same trait.
- `runner.rs` — `JobRunner` orchestrates ticks: list due jobs, fire
  each with its retry sequence, write a `JobRun` row per attempt,
  append an `audit_log` entry per attempt (actor =
  `scheduler:<job_id>`, action = `scheduler.job_run`), push to
  matching sinks on success or final failure. Push payload skipped
  on mid-retry intermediate failures so sinks don't get spammed.

Crate public surface re-exported from `lib.rs`.

### New PG migration — `0007_scheduled_jobs.sql`

Two tables under `xiaoguai-storage/migrations/`:

- `scheduled_jobs` — `id` TEXT PK, `tenant_id` FK to tenants
  (NULL → system-wide), `trigger`/`payload`/`retry_policy`/`sinks`
  as JSONB, `enabled`/`next_fire_at`/`last_fire_at` bookkeeping,
  RLS policy `tenant_id IS NULL OR tenant_id = current_setting(...)`.
- `scheduled_job_runs` — `BIGSERIAL` PK, FK to `scheduled_jobs`,
  `status`/`attempt`/`started_at`/`finished_at` + `session_id` FK
  to `sessions` (SET NULL on session delete so a run row survives
  session cleanup), `error_message`/`output_preview` for the
  audit-first console. RLS policy mirrors `scheduled_jobs`.

Partial index `ix_scheduled_jobs_due` is `WHERE enabled IS TRUE` so
the runner's `list_due` scan only touches active jobs.

- `cron = "0.12"` added to `[workspace.dependencies]`.
- `xiaoguai-scheduler` added to workspace members.

## Tests

- `trigger::tests` (6) — cron parses 6-field, cron top-of-hour
  alignment, cron rejects garbage, interval advances, interval
  zero rejected, serde round-trip both variants.
- `retry::tests` (4) — default 30/60/120 schedule, no-retry preset
  stops after attempt 1, max_backoff_secs clamps long backoffs,
  attempt 0 rejected.
- `repository::tests` (3) — `list_due` returns jobs with no
  `next_fire_at` (priming), `record_fire` updates bookkeeping,
  run repo assigns monotonic ids.
- `runner::tests` (5) — successful run writes exactly one audit row
  with the expected actor/action/resource/details shape; next-fire
  advanced after run; retry escalates failing-then-succeeding
  executor with one row per attempt; retry exhaustion marks all
  three attempts Failed + pushes to the sink once (on the final
  attempt only); sink only delivers when job's `sinks` list
  includes the sink id.
- `sink::tests` (1) — `LoggingSink` captures payload + exposes id.
- `executor` covered indirectly by runner tests; `audit` covered by
  the recording appender.

## Deferred

- **PG repository impls.** `InMemory*` impls back the trait
  contract for v0.10.0. The PG-backed versions land alongside the
  `xiaoguai_runtime` extraction in v0.12.0 — at that point the IM
  gateway / REST / scheduler all share one runtime + PG access path,
  so it's cheaper to do PG repos once across the family than
  per-tag.
- **Operator wiring in `xiaoguai-core`.** The runner trait surface
  is stable but the production binary doesn't yet spawn a tokio
  task that drives `JobRunner::tick` on a timer. Wiring this is a
  one-screen change; left out of v0.10.0 to keep the tag tight.
- **Reactive triggers** (file watcher / webhook / git push) — v0.10.1.
- **Proactive triggers** (LLM-initiated + per-user budget + reason
  field) — v0.10.2. The push payload schema will gain `reason:
  String` then; adding the field later is non-breaking.
- **Real push sinks** (Feishu / Telegram / Email / chat-ui inbox)
  — v0.10.3.
- **Natural-language job definition** (Manus / Hermes UX where the
  user says "每天 8 点扫 r/LocalLLaMA + HN 推 Telegram" and the
  agent compiles to a `ScheduledJob`) — v0.12.1.
- **Per-job timezone field.** Locked to UTC for v0.10.x. Users
  needing local time offset their Cron expression. Per-job tz adds
  a small column + a `chrono-tz` dependency; revisit when the first
  user files a ticket.
- **Streaming verification of audit chain across scheduler runs.**
  `PgAuditSink::verify_tenant` already loads the entire tenant
  chain; once the scheduler is wired in production we'll add a
  background sweep — landed together with v0.6.5's audit endpoint
  wiring.

## Design decisions worth flagging

- **One `JobRun` row per attempt** (not one per fire). Linear audit
  trail; the console can join `attempt` to show "tried 3×, all
  failed" without parsing JSONB.
- **Audit on every attempt, push only on terminal outcome.** Audit
  is the source of truth and must be exhaustive; push sinks are
  user-facing and a mid-retry "it failed" notification would just
  be noise.
- **`AuditAppender` is its own trait** rather than the runner
  taking `Arc<PgAuditSink>` directly. Keeps `xiaoguai-scheduler`
  off the `xiaoguai-storage` dependency graph (storage will
  eventually own scheduler tables; cycle would be inevitable).
- **`RunnerOptions::max_retry_sleep_secs` defaults to 0.** Tests
  drive the loop without `sleep`s; the production binary sets a
  small (≤30s) cap so a misconfigured retry policy can't stall the
  scheduler thread.
