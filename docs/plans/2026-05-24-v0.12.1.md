# v0.12.1 — natural-language job definition + per-scheduled-run synthetic session

**Status:** done (2026-05-24).

## Why this slice

The v0.12.0 plan deferred two pieces that close obvious holes in the
scheduler story; v0.12.1 lands them together so a single release
unblocks the Manus / Hermes-style UX the roadmap §3 v0.12.1 row calls
out.

| Piece | Hole it closes |
|---|---|
| Natural-language → `ScheduledJob` compile | Operators (and the admin-ui Scheduler pane) can author jobs from "每天 8 点扫 r/LocalLLaMA + HN 推 Telegram" instead of hand-authoring the JSON shape. |
| Per-run synthetic session | `scheduled_job_runs.session_id` stops being `None`. The v0.11.1 audit-first console can drill from a scheduled run into the chat-style transcript via the same session→messages join REST and IM already use. |

Bundling them now: they share the scheduler bridge in
`xiaoguai-core/src/scheduler_bridge.rs` (one more cycle-breaking
adapter each), each adds one tiny trait to its consumer crate (one in
`xiaoguai-api`, one in `xiaoguai-scheduler`), and both share the same
runtime wiring touch in `run_serve`.

## Pieces

### Piece 1 — natural-language job definition

#### Trait surface in `xiaoguai-api`

Two new shim traits land beside [`WebhookPusher`] in
`crates/xiaoguai-api/src/scheduler.rs`:

```rust
#[async_trait]
pub trait NlJobCompiler: Send + Sync {
    async fn compile(
        &self,
        description: &str,
        tenant_id: Option<&str>,
    ) -> Result<(serde_json::Value, String), NlJobCompileError>;
}

#[async_trait]
pub trait ScheduledJobUpserter: Send + Sync {
    async fn upsert(&self, job: serde_json::Value) -> Result<(), ScheduledJobUpsertError>;
}
```

Boundary uses `serde_json::Value` so the api crate stays free of the
`xiaoguai_scheduler::ScheduledJob` type (the cycle is still real:
`xiaoguai-scheduler → xiaoguai-im-feishu → xiaoguai-im-gateway →
xiaoguai-api`). Matches the v0.12.0 `WebhookPusher` shape; production
wires the typed impl from `xiaoguai-core`.

Test helpers ship in the same module: `StaticNlJobCompiler` returns a
pre-canned `(job, rationale)`; `RecordingJobUpserter` accumulates
accepted bodies for assertions.

#### HTTP routes

Two new routes in `routes/mod.rs`:

```
POST /v1/admin/scheduler/jobs/compile   { description, tenant_id? }
  → { suggested_job: <ScheduledJob>, rationale: <string> }
  → 503 when nl_job_compiler is None
  → 400 when description is empty or compiler returned unparseable JSON

POST /v1/admin/scheduler/jobs           <ScheduledJob>
  → 201 { id }
  → 503 when job_upserter is None
  → 400 when body doesn't deserialise as a valid ScheduledJob
```

Behind the same admin/Casbin gate as the rest of `/v1/admin/*`.
Per-tenant API tokens stay deferred (same as the v0.12.0 webhook
route — minimum viable gate today).

#### Production impl `LlmNlJobCompiler`

Lives in `crates/xiaoguai-core/src/scheduler_bridge.rs`. Wraps an
`Arc<dyn LlmBackend>` + default model name. The system prompt enumerates
every safe `Trigger` variant by example (cron / interval / file_watch /
webhook / git_push / db_poll / proactive) plus the exact JSON envelope.
Temperature pinned at 0.1; tenant id forwarded so per-tenant routing
applies. Response handling:

1. Drain `chat_stream` deltas into a single string.
2. Strip ` ```json ` fence if present (insurance against models that
   ignore the "no markdown" instruction).
3. `serde_json::from_str` → fail as `Unparseable` on first error.
4. Regenerate `id` with `uuid::Uuid::new_v4()` so the model can't
   collide; `parse-back as ScheduledJob` to surface schema mismatches
   as 400 instead of letting them flow into the upserter.
5. Build a one-line `rationale` summarising the trigger shape and job
   name — surfaced to the operator in the admin pane.

#### Production impl `PgScheduledJobUpserter`

Same file. Wraps `Arc<PgJobRepository>`. Deserialises the JSON body
into a typed `ScheduledJob` (mapping serde errors → `InvalidJob`) and
calls `upsert`. Repository errors surface as `Repository` → 500.

#### Operator wiring

`run_serve` in `xiaoguai-core/src/main.rs`:

* `LlmNlJobCompiler` is wired **unconditionally** (even when the
  scheduler is disabled) — the compile preview is useful before
  flipping the runner on, and the upserter still gates persistence.
* `PgScheduledJobUpserter` only wires when `scheduler.enabled`, since
  it needs the PG repo handle anyway.
* Both feed into the new `AppState` fields `nl_job_compiler` and
  `job_upserter`. `AppState` shared `pg_session_repo` /
  `pg_message_repo` handles factored out so Piece 2 (below) doesn't
  build duplicates.

#### Frontend

**Deferred to v0.12.1.1.** The admin-ui addition (a Scheduler pane
with a textarea + Save button) is straightforward but mechanical;
backend is the load-bearing piece and ships first to keep this slice
focused. The two routes are stable so the UI work in v0.12.1.1 reduces
to one CRUD form. The v0.12.0 admin-ui shipped without scheduler
controls and that remains the case at v0.12.1.

### Piece 2 — per-scheduled-run synthetic session

#### Trait in `xiaoguai-scheduler`

New trait in `runtime_executor.rs`:

```rust
#[async_trait]
pub trait ScheduledSessionWriter: Send + Sync {
    async fn create_and_record(
        &self,
        job: &ScheduledJob,
        prompt: &str,
        new_messages: &[LlmMessage],
    ) -> Result<String, String>;
}
```

Returns the new `session_id`. Error type stays `String` to match
`JobExecutor::execute`'s existing error channel.

#### `RuntimeJobExecutor` change

Adds `Option<Arc<dyn ScheduledSessionWriter>>` member + builder method
`with_session_writer(...)`. When `Some`, `execute()` calls the writer
after the runtime returns (so a writer failure can't cancel an
already-completed agent run; it surfaces as the `JobRun` row's
error_message) and threads the returned id into
`ExecutionOutcome.session_id`. When `None`, behaviour is unchanged from
v0.12.0 — `session_id` stays `None`. The v0.12.0 `new(ctx)` constructor
keeps the same signature so existing call sites compile unmodified.

#### Production impl `PgScheduledSessionWriter`

Lives in `crates/xiaoguai-core/src/scheduler_bridge.rs`. Holds
`Arc<dyn SessionRepository>` + `Arc<dyn MessageRepository>` — the same
ones `AppState` already carries.

For each job-run:

1. Bail with a clear error if `job.tenant_id` is `None` (RLS gives us
   no useful behaviour for null-tenant sessions and the audit-first
   console can't surface them by user_id anyway).
2. Generate a UUID v4 `session_id`.
3. Synthesise a user id `scheduler:<job_id>` so the console can show a
   "scheduler" badge instead of a real user avatar.
4. Insert a session row with title `"scheduled: <job.name>"`, the
   model name from `job.payload.model` (falls back to `"scheduler"`),
   and the standard `Active` status.
5. Append every message in `new_messages` via `MessageRepository::append`.

#### Operator wiring

In `run_serve`:

```rust
let session_writer: Arc<dyn ScheduledSessionWriter> = Arc::new(
    PgScheduledSessionWriter::new(pg_session_repo.clone(), pg_message_repo.clone()),
);
let executor: Arc<dyn JobExecutor> = Arc::new(
    RuntimeJobExecutor::new(runtime_ctx).with_session_writer(session_writer),
);
```

Only wired when `scheduler.enabled` (same gate as the rest of the
scheduler bootstrap).

## Deferred

* **Admin-ui Scheduler pane** (v0.12.1.1). Backend routes are stable;
  the UI is a textarea + Save button + JSON pretty-print + tenant
  selector. Add to admin-ui as a new sidebar entry below Eval. Target
  bundle delta: +5 kB CSS / +5 kB JS.
* **Per-tenant API tokens for the new routes.** Same as v0.12.0 —
  routes sit behind admin Casbin only. Per-tenant tokens land when
  the scheduler grows multi-tenant external integrators.
* **Streaming compile progress.** The compiler runs synchronously
  (one LLM call); the route blocks. If a slow backend pushes wall
  clock past the operator's patience we can add SSE later — the trait
  shape already supports streaming hooks via `chat_stream`.
* **Tool calls inside scheduled-run synthetic sessions.** Today
  `PgScheduledSessionWriter` writes the `new_messages` slice — which
  includes the user prompt and assistant text turns. Tool calls and
  tool results carried inside `LlmMessage` collapse to text-only
  blocks via the same simplification IM v0.7.4 uses. v0.12.1.x can
  promote them to `ContentBlock::ToolCall` / `ToolResult` once the
  audit-first console renders them differently.
* **Cancel hook for in-flight compiles.** The compiler ignores its
  parent context's cancellation today; an admin who hits Stop in the
  pane has to wait for the LLM to finish. Wire `CancellationToken`
  through when the route grows a cancel button.

## Design decisions worth flagging

* **Compiler returns `serde_json::Value`, not a typed `ScheduledJob`.**
  Keeps the `xiaoguai-api` crate free of the scheduler type definition
  (cycle constraint). The compile route would otherwise need the api
  crate to depend on `xiaoguai-scheduler`, which would re-create the
  v0.12.0 cycle the `WebhookPusher` indirection broke. Cost: one extra
  serde round-trip at the boundary; benefit: dependency graph stays
  acyclic. The compiler in `xiaoguai-core` still
  `serde_json::from_value::<ScheduledJob>(...)` internally before
  returning to surface schema mismatches as `Unparseable` rather than
  silently propagating bad shapes through to the upserter.
* **`id` is always regenerated server-side.** The LLM picks an `id` to
  satisfy the JSON schema; we throw it away and use
  `uuid::Uuid::new_v4()`. Prevents a model from picking a colliding id
  on purpose or by mistake.
* **Compiler wired even when scheduler is disabled.** Compile is a
  read-only preview; the operator can iterate on phrasings without
  flipping the runner on. Save still requires `scheduler.enabled` →
  503 on the upserter route. This matches the spirit of "models are
  unreliable, but systems can be reliable" — the validate-and-show step
  is decoupled from the persist step.
* **Synthetic session created post-runtime, not pre-runtime.** A writer
  failure can't cancel an already-completed agent run. The `JobRun` row
  flips to failed (so the operator notices) but the LLM call wasn't
  wasted. Pre-runtime session creation would mean a writer hiccup
  silently aborts the actual work, which is the wrong failure mode for
  a scheduler that's billed as audit-first.
* **Synthetic user id format `scheduler:<job_id>`.** Stable, prefix-
  recognisable, joins through to `sessions.user_id` so the audit-first
  console's existing tenant scoping just works. Production admins
  could also use the prefix to filter scheduled-run sessions out of
  the regular chat-ui sidebar (orthogonal to this slice).
* **`xiaoguai-runtime` doc-markdown fixup + scheduler `pg_repository`
  `#[allow(clippy::needless_pass_by_value)]` on three helper functions.**
  Both were latent in v0.12.0 — newer clippy versions surface them. The
  `#[allow]` matches the actual helper shape (passed straight into
  `.map_err(...)` which yields owned errors); rewriting the call sites
  to `.map_err(|e| sqlx_err(&e))` would touch ~30 lines for zero benefit.
