# v0.8.3 — chat-ui code-block syntax highlighting + copy button

**Status:** done.

## What shipped

Two chat-ui finishers from the v0.8.2 backlog:

1. **Syntax highlighting** in fenced markdown code blocks rendered inside
   assistant bubbles. Nine languages registered statically (with
   common aliases): bash/sh/shell, json, typescript/ts/tsx,
   javascript/js/jsx, python/py, rust/rs, sql, yaml/yml, markdown/md.
   Unknown languages render as plain monospace inside the same
   copy-button wrapper.
2. **Copy button** on every fenced code block (markdown) and on every
   tool bubble. Top-right overlay, fades in on hover or keyboard
   focus, calls `navigator.clipboard.writeText`, flips the label to
   "Copied!" for 1.2 s, then reverts. No third-party clipboard polyfill.

Both surfaces reuse the v0.8.2 `[data-theme]` tokens — the highlighter
swaps between Prism's `oneLight` and `oneDark` style objects via a
React re-render driven by `useTheme()`. Copy-button colors come from
`--surface-elevated`, `--accent`, `--accent-soft`, `--border`,
`--shadow-focus` — all existing tokens, none redefined.

## Pieces

### `src/codeblock.tsx` (new)

`CodeBlock` + `CopyButton` components.

* `CodeBlock` wraps `PrismLight` (the static-register variant — see
  *Design decisions*). Receives a `language` hint (already validated
  against the registered set) plus the raw `value`. For unrecognized
  languages it renders a plain `<pre><code>` so the copy affordance
  still works.
* `CopyButton` is exported standalone so tool bubbles can host it
  without going through the highlighter path. It owns a 1.2 s
  `setTimeout` for the success label and rejects silently if the
  Clipboard API is unavailable (insecure context / permission
  denied) — the label resets so the user knows it didn't take.
* Languages registered once at module load via
  `SyntaxHighlighter.registerLanguage(name, grammar)`; alias names
  share a grammar reference.

### `src/markdown.tsx` (edit)

* Adds a `components.code` override that detects fenced blocks via the
  `language-*` className react-markdown injects, or via the presence
  of a newline in the value (untagged fences). Inline code stays under
  the cheap `.md code` styling.
* Adds `components.pre` that strips the outer `<pre>` react-markdown
  emits for fences — `CodeBlock` provides its own, and nesting `<pre>`
  inside `<pre>` is invalid HTML.
* Note: react-markdown v10 dropped the `inline` prop, so detection is
  className-based now.

### `src/ChatPage.tsx` (edit)

* Tool bubbles gain `copy-host` class and an embedded `<CopyButton>`
  when they have non-empty text. The copy host is the bubble itself,
  not a per-line affordance.
* Empty / still-streaming tool bubbles skip the button (nothing to
  copy yet).

### `src/styles.css` (edit, +95 lines)

* `.codeblock` — `position: relative` wrapper so the copy button can
  anchor to the top-right corner.
* `.codeblock__plain` — fallback for unknown languages; matches the
  existing `.md pre` look so the rhythm stays consistent.
* `.copy-btn` — corner overlay, `opacity: 0` at rest, reveals on
  `:hover` of `.codeblock` / `.copy-host` or `:focus-visible` on the
  button itself. Accent-soft hover background, `--shadow-focus` on
  keyboard focus.
* `.bubble.tool.copy-host` — adds right-side padding so the absolute
  copy button doesn't overlap `→ name(args)`.

No new color tokens. Both `oneLight` and `oneDark` are imported once
and switched at the React layer, so dark/light correctness rides on
the existing `[data-theme]` attribute — no extra CSS theme fork
required.

The chat-ui has no test harness (matches v0.8.1/v0.8.2 — same
scaffold). Verification is:

* `pnpm -F chat-ui typecheck` → **clean**.
* `pnpm -F chat-ui build` → **clean** (TS strict + vite build).
* `pnpm -F chat-ui lint` → **`echo 'no lint yet'`** (intentional;
  matches v0.8.1/v0.8.2 — frontend lint isn't wired up).
* `pnpm -r typecheck` → **clean** across shared + chat-ui + admin-ui.

Visual quality (hover reveal timing, dark/light contrast on both
themes, the copy-button overlap behavior on narrow viewports) **was
not browser-verified**. Matches the v0.8.1 / v0.8.2 / v0.9.3 / v0.9.4
pattern — chat-ui visuals are tuned by reading the cascade, then
punched out to the human on `pnpm dev`. Sharp edge §6 of the handoff
already calls this out as outstanding for the v0.8.x band.

## Acceptance

* Bundle sizes (production `vite build`):

  | Asset | Before (v0.10.1) | After (v0.8.3) | Delta |
  |-------|------------------|----------------|-------|
  | `index-*.css` | 9.63 kB | 10.73 kB | **+1.10 kB** (+11 %) |
  | `index-*.css` (gzip) | 2.69 kB | 2.87 kB | +0.18 kB |
  | `index-*.js` | 331.66 kB | 430.89 kB | **+99.23 kB** (+30 %) |
  | `index-*.js` (gzip) | 104.32 kB | 134.28 kB | +29.96 kB |

  CSS delta well under the 5 kB budget. **JS delta exceeds the ~80 kB
  guideline by ~19 kB** (≈30 kB gzip in over the wire). The rationale
  for staying with PrismLight despite the overage is in *Design
  decisions* below — the only smaller-on-disk path (`PrismAsyncLight`
  + per-language dynamic chunks) ballooned `dist/` to ~1.8 MB across
  287 separate files, which is a worse production deploy story than
  one 431 kB JS bundle.
* `pnpm -F chat-ui typecheck` clean.
* `pnpm -F chat-ui build` clean.
* `pnpm -F chat-ui lint` runs (no-op).
* `pnpm -r typecheck` clean across the frontend workspace.
* No backend / Rust changes — Cargo workspace untouched; v0.10.1
  smoke pass remains valid.

## Design decisions worth flagging

* **`PrismLight` over `PrismAsyncLight`.** Both expose the same
  `registerLanguage` API. `PrismAsyncLight` *also* pulls
  `refractor`'s dynamic-import tree, which Vite materialises into one
  separate chunk per language (~287 extra files in `dist/`, total
  unpacked dist ~1.8 MB). For a chat surface where the warm path
  needs maybe 2–3 languages, that's a worse deploy story than one
  larger JS file. `PrismLight` keeps the bundle to exactly what we
  register. Cost: ~99 kB JS delta (≈30 kB gzip) vs. the brief's
  ~80 kB target — documented above as accepted.
* **`oneLight` / `oneDark` themes shipped together.** They're plain JS
  style objects; both being in the bundle is ~6 kB. Switching at the
  React layer (driven by `useTheme()`) means the highlighter
  re-renders correctly without needing a CSS theme fork or a
  page reload. No third theme planned.
* **Copy button is a single `<button>`, no portal.** The codeblock is
  `position: relative`; the button is `position: absolute` in its
  top-right corner. No portal / overlay layer means it composes
  trivially inside scrollable bubbles and respects the bubble's own
  stacking context.
* **Copy button reveal uses `opacity: 0` + `:hover` / `:focus-visible`**,
  not `display: none`. This keeps the button in the tab order for
  keyboard navigation while staying invisible to the eye at rest.
* **Tool bubbles get a single bubble-level copy button**, not a
  per-line one. Tool output is a single string (`← {...}` or
  `→ name(args)`) — splitting it line-by-line would add clicks
  without value.
* **No language inference for tool bubbles.** They're already
  monospace-formatted text from the agent loop; trying to detect
  JSON-vs-plain would add a parsing layer for marginal benefit. Copy
  is enough.
* **`navigator.clipboard.writeText` directly, no polyfill.** Modern
  browsers all support it in secure contexts; failure mode is the
  label not flipping to "Copied!", which is a sufficient signal.
  Adding `react-copy-to-clipboard` would have brought in a hidden
  `<textarea>` + `execCommand('copy')` fallback that doesn't help on
  any browser shipping in 2026.
* **No `eslint` added.** chat-ui's `lint` script is a no-op today
  (v0.8.1 / v0.8.2 baseline); introducing a lint config in this tag
  would expand scope. Deferred to whenever the frontend gets its
  linting story.
