# v1.1.5b — Supervisor crate (design doc only)

**Status:** plan only. No code in this tag. v1.1.5a (peer MVP) ships
alongside; this doc describes the orchestration layer we deliberately
*did not* build for v1.1.5a.

## Motivation

v1.1.5a (`docs/architecture/multi-agent-peer.md`) ships peer-to-peer
multi-agent with zero new code — every running xiaoguai-core is
already both an MCP server (v0.9.1) and an MCP client (v0.9.0), so a
"specialist" + "front-door" topology is a deployment shape, not a code
shape.

The peer pattern has three known limits, all called out in the
architecture doc:

1. **No shared session state across hops.** Each `call_tool` on a peer
   is a fresh agent run; the specialist has no view of the front
   door's conversation history.
2. **No cancel propagation.** Cancelling the front-door run does not
   abort an in-flight specialist call.
3. **No audit chain stitch.** Each peer keeps its own chain; cross-peer
   correlation is by timestamp.

For specialist-style "fire and forget" delegations the peer pattern is
fine — and ~80% of the multi-agent use cases people ask about
(summariser, code reviewer, doc Q&A) are exactly that shape. The
remaining 20% need a centralised orchestrator: a *supervisor* that
holds the plan, dispatches subtasks to specialists, accumulates their
results into the same context, and stitches everything together.

That's v1.1.5b.

## Crate proposal: `xiaoguai-orchestrator`

A new top-level crate, mirroring the shape of `xiaoguai-runtime`. It
sits *above* `xiaoguai-runtime` (which owns the single-agent loop) and
beside `xiaoguai-agent` (which owns the Toolbox + ReAct primitives).

```
xiaoguai-orchestrator      Supervisor + Specialist registry +
                           dispatch tool + child-context lifecycle
  ↓ depends on
xiaoguai-runtime           run_to_completion / run_streamed / run_to_sink
  ↓ depends on
xiaoguai-agent             Toolbox + ReactAgent + AgentEvent
```

Adding a separate crate (rather than a module on `xiaoguai-runtime`)
keeps the dependency graph honest: runtime is the substrate every
caller shares, orchestrator is opt-in.

### Types

```rust
/// A registered specialist — name, the RuntimeContext to run it with
/// (its own Toolbox, its own backend selection, its own system prompt),
/// and a one-line description for the supervisor's tool catalogue.
pub struct Specialist {
    pub name: String,
    pub description: String,
    pub ctx: RuntimeContext,
    pub system_prompt: String,
}

/// The orchestration entry point. Owns the supervisor's own
/// RuntimeContext (whose Toolbox is *just* the dispatch tool) plus the
/// registry of children.
pub struct Supervisor {
    ctx: RuntimeContext,
    specialists: Vec<Specialist>,
}

impl Supervisor {
    pub fn new(ctx: RuntimeContext, system_prompt: String) -> Self { ... }
    pub fn register(mut self, s: Specialist) -> Self { ... }

    /// Build the supervisor's Toolbox. Adds one synthetic tool —
    /// `dispatch_to` — whose JSON schema mirrors `DispatchArgs`.
    /// Returns a runtime context the caller plugs into the standard
    /// `run_to_completion` / `run_streamed` / `run_to_sink` entry
    /// points.
    pub fn into_runtime(self) -> RuntimeContext { ... }
}

#[derive(Deserialize)]
struct DispatchArgs {
    /// Name of a registered specialist.
    specialist: String,
    /// Sub-task description in plain text — the specialist's
    /// ReactAgent runs with this as the user turn.
    subtask: String,
    /// Optional context items pulled from the supervisor's session
    /// (e.g. relevant prior tool outputs) to seed the specialist's
    /// view. Bounded by `max_context_tokens` to keep dispatch costs
    /// predictable.
    context: Option<Vec<ContextItem>>,
}
```

### How dispatch works

The supervisor's system prompt explicitly directs the model:

> You coordinate a team of specialists. When a subtask falls into one
> specialist's domain, emit a `dispatch_to(<name>, <subtask>)` tool
> call. The result of that call is the specialist's answer. Do not
> try to do specialist work yourself. Stitch their answers into your
> final reply.

When the supervisor's ReactAgent emits a `dispatch_to` tool call, the
`McpClient` impl backing that tool:

1. Looks up the specialist by name in the registry (returns
   `invalid_params` if unknown).
2. Constructs a *child* `RuntimeContext` cloned from the specialist's
   registered ctx, with the supervisor's `tenant_id` and a fresh
   `session_id` propagated so the audit chain can correlate.
3. Calls `runtime::run_to_completion` on the child context with
   `subtask` as the user message.
4. Returns the final assistant text as the tool result.

Every dispatch is a complete agent run. The supervisor's transcript
sees only the tool call and the tool result — it does *not* see the
specialist's intermediate reasoning. This is a deliberate context
budget choice: the supervisor handles 5 specialists × 10 turns = 50
internal reasoning steps without exploding its own context window.

### Lifecycle + cancel

`Supervisor::into_runtime()` registers a `CancellationToken` linked to
every active dispatch. The supervisor's own cancel token (already
piped through `xiaoguai-api::CancelRegistry`) becomes the parent. When
the front door cancels the supervisor, the registry walks active
dispatches and cancels each child — solving the v1.1.5a limit #2 for
the supervised case.

### Audit chain

Three options, none free:

| Option | Pro | Con |
|--------|-----|-----|
| A. One audit row per dispatch (in supervisor's chain). | Simple; cross-peer correlation already there. | Specialist's internal turns are invisible to the parent chain. |
| B. One audit row per supervisor turn + child rows linked by parent_id. | Full visibility. | Requires `audit_log.parent_id` migration; chain HMAC needs to thread parent hash. |
| C. Append cross-chain "JOIN row" to the supervisor chain that includes the specialist's chain HMAC at dispatch time. | Cryptographic stitch. | Most complex; requires the specialist to expose chain head via MCP. |

**Recommendation:** start with A in v1.1.5b. Schema-extend to B in a
follow-up if operators ask for it.

### Open questions to resolve before coding

1. **Shared session across dispatches?** Two specialists asked
   sequentially to "summarise" different pages — should the second
   dispatch see the first's summary? Today (peer pattern) no. Plan
   default: no — operators wanting that pass it via `context`.
2. **Loop detection.** What if the supervisor's tool catalogue
   accidentally includes an HTTP MCP tool that points back at the
   supervisor itself? Plan: hop counter on the child context, default
   limit 4, surfaces as a `RuntimeError::TooManyDispatchHops`.
3. **Parallel dispatches.** Should the supervisor be allowed to fire
   `dispatch_to(A, ...)` and `dispatch_to(B, ...)` in parallel? Most
   ReAct-style models emit one tool call per turn; parallel dispatch
   is a multi-tool-call feature. Plan: support it (the
   `ToolCallSpec.id` is already unique per call), but document that
   the supervisor's prompt has to opt-in by emitting both calls in
   one assistant turn.
4. **Cost accounting.** Each dispatch is a child agent run with its
   own token spend. The `TokenUsage` events from the child must
   bubble up to the supervisor's session for `/v1/usage`. Plan: child
   ctx writes usage rows tagged with `parent_session_id`; the
   supervisor's usage view sums both.
5. **Streaming.** Should `dispatch_to` stream the specialist's
   intermediate output back to the supervisor's SSE stream? Default
   plan: no — the supervisor stream emits `tool_call` → `tool_result`
   the same way today's tool calls do. Verbose mode (`?verbose=1`)
   can opt-in to forwarding child `text_delta` events.

## Wiring on top of v1.1.5a

The peer pattern keeps working. The supervisor crate is **additive**:

- Operators who want fire-and-forget delegation use the peer pattern
  unchanged.
- Operators who want orchestration construct `Supervisor` in their
  own binary (or in `xiaoguai-core` behind a config flag) and let it
  produce the `RuntimeContext` they plug into the API surface.

A supervisor *can* dispatch to a peer specialist running on a remote
host — the specialist just needs to be reachable as an `McpClient`
(stdio, http, whatever). That gives operators a free choice: keep
specialists local (low latency, simpler audit), or spread them
horizontally (independent scaling, polyglot impl).

## What this tag delivers

This tag (`v1.1.5b-doc`) ships **only this design doc**. No crate, no
test, no Cargo changes. The actual `xiaoguai-orchestrator` crate is a
future tag (`v1.1.6` or similar) that picks this design up after the
v1.1.5a peer pattern collects real usage feedback.

The reason for splitting them: the peer pattern is already feature-
complete via existing v0.9.0 + v0.9.1 code, so v1.1.5a is a one-day
ship (example + doc + test). The supervisor needs a fresh crate, the
audit-chain decision, and integration with `xiaoguai-api::CancelRegistry`
— at minimum a multi-day session and a real RFC pass on the open
questions above. Bundling them would gate the cheap shippable thing
behind the expensive design exercise.
