# v1.1.6 — bare-metal tarball + systemd unit + install scripts

**Status:** shipped (2026-05-24).

## Why this slice

`deploy/README.md` has listed "Bare-metal tarball + systemd unit +
uninstall" as a v1.0 deferral since the docker-compose pipeline
landed. v0.9–v0.12 closed the container path (multi-arch image +
cosign + SBOM in `release.yml`); now we owe operators who don't run
Docker — single-node deployments, edge boxes, air-gapped lab
environments — a parallel install target.

Constraints kept this small:

- Two Linux targets only (`x86_64-unknown-linux-gnu` +
  `aarch64-unknown-linux-gnu`). musl, Windows, macOS deferred.
- `cross` for the build — no per-target runners, one workflow.
- One systemd unit (`xiaoguai-core.service`). The CLI binary ships in
  the same tarball but isn't service-managed — operators invoke it
  directly.
- Bash install/uninstall scripts. No deb/rpm packaging — that needs
  `fpm` or `cargo-deb`, and the install steps barely warrant it for
  the v1.1 audience.

## Pieces

### 1. `scripts/release/build-tarball.sh`

`set -euo pipefail`. For each target:

1. `cross build --release --target $target --bin xiaoguai-core --bin xiaoguai`.
2. Lay out the release tree:

   ```
   xiaoguai-vX.Y.Z-<target>/
   ├── bin/{xiaoguai-core,xiaoguai}
   ├── share/
   │   ├── config.example.yaml
   │   ├── migrations/    ← copy of crates/xiaoguai-storage/migrations/
   │   └── catalog/       ← copy of crates/xiaoguai-api/catalog/
   ├── systemd/xiaoguai-core.service
   ├── scripts/{install.sh,uninstall.sh}
   ├── README.txt
   └── LICENSE
   ```

3. `tar --owner=0 --group=0 --sort=name -czf` for reproducible-ish
   output.

CI overrides the default `TARGETS` array with a single value via env
so each matrix shard builds one tarball — keeps cache locality + log
volume sane. Local invocation builds both back-to-back.

`SHA256SUMS` is written alongside the tarballs at the end. CI
regenerates it across all shards at publish time (see below).

### 2. `deploy/systemd/xiaoguai-core.service`

`Type=simple` — axum doesn't speak `sd_notify` in v1.1.6, so we don't
claim notify semantics. (If `xiaoguai-core` ever wires the
[`sd-notify`](https://crates.io/crates/sd-notify) crate, flip to
`Type=notify`.)

`User=xiaoguai` static account provisioned by `install.sh`;
`DynamicUser=no` so on-disk state owned by it survives upgrades.

Hardening lifted from the systemd "Strict" template + trimmed for our
needs:

- `NoNewPrivileges=true`
- `ProtectSystem=strict` + explicit `ReadWritePaths=/var/lib/xiaoguai
  /var/log/xiaoguai` + `ReadOnlyPaths=/etc/xiaoguai`
- `ProtectHome=true`, `PrivateTmp`, `PrivateDevices`, full kernel
  protections (`ProtectKernel{Tunables,Modules,Logs}`,
  `ProtectControlGroups`, `ProtectHostname`, `ProtectClock`,
  `ProtectProc=invisible`)
- `RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX` (drops raw
  sockets, packet, netlink)
- `RestrictNamespaces`, `RestrictRealtime`, `RestrictSUIDSGID`,
  `LockPersonality`, `MemoryDenyWriteExecute`
- `SystemCallArchitectures=native`,
  `SystemCallFilter=@system-service` + denials for
  `@privileged @resources @mount @reboot @swap @cpu-emulation
  @obsolete`
- `CapabilityBoundingSet=` (empty) + `AmbientCapabilities=` (empty) —
  we bind a high port so `CAP_NET_BIND_SERVICE` isn't needed.

`StartLimitBurst=5 / IntervalSec=60` caps restart storms on a broken
config. `TimeoutStopSec=30` plus `KillSignal=SIGTERM` so axum's
graceful shutdown finishes in-flight requests.

### 3. `scripts/release/install.sh`

`set -euo pipefail`. Idempotent — every file copy compares with `cmp`
first and skips when unchanged; differing files get a `.bak` of the
prior copy before replacement.

Steps:

1. Create `xiaoguai:xiaoguai` system user/group (skip when present).
2. `install -d -o xiaoguai -g xiaoguai -m 0750 /var/lib/xiaoguai
   /var/log/xiaoguai`.
3. `install -m 0755 bin/* /usr/local/bin/`.
4. `install -m 0644 share/config.example.yaml /etc/xiaoguai/`,
   plus the `migrations/` and `catalog/` trees.
5. `install -m 0644 systemd/*.service /etc/systemd/system/`.
6. `systemctl daemon-reload && systemctl enable xiaoguai-core`.
7. Print next-steps banner pointing at
   `/etc/xiaoguai/config.example.yaml` → `config.yaml` and the
   required `database.url` / `cache.url` / `audit.hmac_key` edits.

Env overrides for non-default paths
(`BIN_DIR` / `CONF_DIR` / `STATE_DIR` / `LOG_DIR` / `UNIT_DIR` /
`XIAOGUAI_USER` / `XIAOGUAI_GROUP`) — covers the "I install under
`/opt/xiaoguai/`" case.

### 4. `scripts/release/uninstall.sh`

Reverse of `install.sh`. Asks `[y/N]` before deleting
`/var/lib/xiaoguai/` (runtime state / audit cache) so an
upgrade-via-reinstall doesn't lose data. Non-interactive use:
`ASSUME_NO=1` skips the prompt and preserves state; `PURGE_STATE=1`
deletes without prompting.

Safety nets:

- Saves `/etc/xiaoguai/config.yaml` to a timestamped sibling path
  before removing the config tree, so a hand-edited config isn't
  vaporised.
- Refuses to delete the `xiaoguai` system user while state/log dirs
  still own files (preserved state owned by a recycled UID is a
  documented security hazard).

### 5. `deploy/config.example.yaml`

The tarball includes a fully-populated example config matching the
shape of `xiaoguai-config::Settings`. Every block documented inline
with the corresponding `XIAOGUAI_*` env override. The example's
`audit.hmac_key` is the same dev string the binary defaults to so
operators get a "MUST be rotated" prompt rather than a 500 on first
boot from a missing field.

### 6. `.github/workflows/release-tarball.yml`

NEW file. Lives separate from `release.yml` so a tarball-side failure
doesn't gate the container pipeline (and vice versa). Fires on `v*`
tags.

Two jobs:

1. **`build`** (matrix over the two targets) — checks out, installs
   the rust toolchain + `cross`, runs `build-tarball.sh` with
   `TARGETS=<one>`, uploads the tarball as an artifact.
2. **`publish`** — downloads all matrix artifacts, regenerates a
   combined `SHA256SUMS`, attaches everything to the GitHub Release
   via `softprops/action-gh-release@v2` (`fail_on_unmatched_files:
   true` so a silently-empty release fails the job).

`release.yml` is untouched — C8 (pip wheel) and C9 (CI security) work
in parallel.

## Tests

- `bash -n scripts/release/*.sh` — all three pass.
- `python3 -c "import yaml; yaml.safe_load(...)"` on the workflow +
  example config — both parse.
- `configparser.ConfigParser` round-trip on the systemd unit — three
  sections (`Unit` / `Service` / `Install`) detected, `Type=simple`,
  `WantedBy=multi-user.target`.

**Not run locally:** the actual `cross build` — would need Docker
running in the agent environment. The user must run
`bash scripts/release/build-tarball.sh` once locally to verify the
two artifacts land in `dist/` and `tar -tzf` matches the documented
layout. The CI matrix exercises both targets on every tag push.

## Deferred (post-v1.1.6)

- **Debian packages (.deb).** Would need `cargo-deb` or `fpm`.
  `dpkg-deb -I` + dependency declarations + postinst/postrm scripts
  would replace install.sh/uninstall.sh on Debian-family distros.
  Useful once a downstream packager asks.
- **RPM packages.** Same as deb, via `cargo-generate-rpm` or `fpm`.
- **musl statically-linked tarball.** Removes the glibc version
  pinning so the tarball runs on Alpine + older RHEL. Adds a third
  CI matrix entry; worth it once an operator hits a glibc symbol
  mismatch.
- **Windows MSI.** `wix-toolset` + a service wrapper. Not yet
  justified — the v1.x audience is overwhelmingly Linux.
- **macOS pkg.** `pkgbuild` + launchd plist. Useful for dev laptops;
  `cargo run` is the supported path today.
- **`Type=notify` systemd integration.** Requires wiring the
  `sd-notify` crate into `xiaoguai-core`. Pure ergonomics — gives
  systemd "ready" semantics so dependent units only start after the
  HTTP listener accepts. Not load-bearing for v1.1.
- **Build provenance / SLSA attestations on the tarballs.** The
  container image is cosign-signed already; mirroring that for the
  tarballs needs `cosign sign-blob` + an attestation workflow.
  Reasonable v1.2 hardening.
- **CAP_NET_BIND_SERVICE drop-in.** Operators binding to :80 / :443
  need a documented drop-in that adds the cap back. Currently
  undocumented; one paragraph in `deploy/README.md` plus an example
  in `systemd/` would close it.
- **`/var/log/xiaoguai` rotation.** Today logs go to journald. If
  an operator opts into on-disk logging there's no `logrotate.d`
  snippet. One file fix if/when needed.

## Design decisions worth flagging

- **install.sh is bash, not Ansible / Salt / etc.** A 200-line bash
  script is auditable in 5 minutes; pulling in a config-mgmt
  framework would be the wrong scale for the v1.1 audience (single
  operator, single box). Anyone running fleets is already speaking
  their preferred IaC and will template our paths in.
- **No `Type=notify`.** Lying to systemd about readiness is worse
  than telling it nothing — `Type=simple` means the unit reports
  "active" the moment the binary spawns, which matches what we can
  truthfully claim today.
- **`CapabilityBoundingSet=` empty by default.** Operators wanting
  privileged ports add a drop-in; the default path doesn't need any
  capabilities and giving them out for free is unnecessary attack
  surface.
- **`ReadOnlyPaths=/etc/xiaoguai`.** Defence-in-depth — a compromised
  service can't edit its own config to weaken `auth.required` or
  redirect `audit.signing_key_env`.
- **`ReadWritePaths=/var/lib/xiaoguai /var/log/xiaoguai`.** Explicit
  allow-list pairs with `ProtectSystem=strict`. Anything outside
  those two paths plus `/tmp` (via `PrivateTmp`) is read-only to the
  service.
- **No state-dir delete by default in uninstall.** Audit cache + any
  embedded state is precious; deleting it silently during a reinstall
  would surprise operators. Explicit prompt or `PURGE_STATE=1` env
  honours the principle of least astonishment.
- **install.sh creates `/var/lib/xiaoguai` with 0750 mode.** Group
  read access lets an admin in the `xiaoguai` group inspect cache
  contents during incident response without escalating to root.
- **build-tarball.sh's `TARGETS` env override.** Same pattern as
  `cargo build --target` — CI passes one at a time so each matrix
  shard's cache hits the right target; local users get both by
  default.
- **CI publish job regenerates SHA256SUMS.** The per-target script
  output is correct but only covers that shard's tarball; the
  release-attached SHA256SUMS spans both. One source of truth for
  operators verifying downloads.
- **`fail_on_unmatched_files: true` on the release upload.** A silent
  "release published with zero assets" is worse than a noisy CI
  failure that someone can investigate.
