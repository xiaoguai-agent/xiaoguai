## Goal

Make the multi-tenant claim a real defense-in-depth boundary, not a
trust-only one. v0.5/v0.6 declared RLS policies on every tenant-scoped
table but never set `app.current_tenant_id`, so the policies were inert
under the test superuser and would have produced an empty result set
against any non-superuser production role.

v0.6.1 closes that gap: every repository method that touches an
RLS-enabled table now opens its own transaction, runs
`SELECT set_config('app.current_tenant_id', $1, true)` when the caller
supplies a tenant, and executes the actual query against `&mut *tx`.

## Why this slice and not the bigger v0.6.1 plan

The v0.6 plan doc carved v0.6.1 as a bundle of *RLS + Casbin per-route +
LlmRouter wire-up*. We picked just the RLS strand for this milestone
because:

- It is the only one of the three that breaks every repo call signature.
  Shipping it on its own keeps the diff one-thing-at-a-time and avoids
  conflating "thread tenant through the stack" with "swap MockBackend
  for a real router".
- Casbin per-route and LlmRouter wire-up are pure-additive changes that
  don't move the security boundary. They can ship as v0.6.2 / v0.6.3
  without touching the now-stabilised repo trait surface.

## What landed

### `xiaoguai-storage`

- New module `repositories::tenant_ctx` exposing
  `begin_tenant_tx(&PgPool, Option<&str>) -> Transaction`. Centralises
  the per-request GUC setting so every repo uses the exact same
  invocation.
- `SessionRepository`, `MessageRepository`, `LlmProviderRepository`,
  `McpServerRepository`, `TokenUsageRepository` traits gained a
  `tenant: Option<&str>` argument on every method whose underlying SQL
  touches an RLS-policy-protected row. Implementations open a
  transaction via `begin_tenant_tx`, run the SQL against `&mut *tx`,
  and commit.
- `UserRepository` was already RLS-aware via its own
  `set_tenant_guc` helper. Left unchanged in this slice â€” its public
  signatures don't take an `Option<&str>` because every method already
  carries an explicit `tenant_id` parameter or derives it via
  `discover_tenant_of_user`. Conversion to the shared helper is a
  follow-up cleanup, not a correctness gap.
- New ignored integration test
  `tests/rls_isolation.rs` provisions a `xiaoguai_app` PG role
  (`NOSUPERUSER`) with the minimum CRUD grants and re-runs cross-tenant
  scenarios through it â€” without `set_config` calls the role sees an
  empty result set, with them it sees only its own tenant's rows.

### `xiaoguai-api`

- `sessions::*` handlers extract `Claims` (when present) and thread
  `tenant.as_deref()` into every repo call. `send_message` switches to
  the **session's own `tenant_id`** once the session is loaded â€” the
  loaded session is authoritative; the claims tenant only gates the
  initial `find_by_id`.
- `mcp::list_servers` is unchanged at the call site â€” its existing
  `list_global` / `list_for_tenant` split already encodes the tenant
  decision.

### `xiaoguai-cli`

- `provider register` and `mcp register` pass the user-supplied
  `--tenant` (or `None`) so the GUC matches what gets stored.
- `provider remove` and `mcp remove` pass `None`; admin CLI relies on
  superuser/owner bypass. A `--tenant` flag for scoped deletes is
  deferred to v0.6.2.

### In-memory test stubs

- `xiaoguai-api/tests/common/`,
  `xiaoguai-im-gateway/tests/common/`,
  `xiaoguai-cli/tests/common/`, `xiaoguai-cli/tests/{provider,mcp}_cli.rs`,
  `xiaoguai-api/tests/mcp.rs` â€” every in-memory repo impl now takes
  `_tenant: Option<&str>` (ignored â€” RLS is a Postgres concern). One
  observability helper (`InMemorySessionRepo::last_tenant()`) records
  what was passed for tests that want to assert the wiring.

### testcontainers test updates

- The existing 42 ignored PG tests in `xiaoguai-storage/tests/*.rs`
  pass `None` everywhere â€” they continue to run as superuser and
  exercise behaviour, not the security boundary. The new
  `rls_isolation.rs` suite is where the boundary itself gets tested.
