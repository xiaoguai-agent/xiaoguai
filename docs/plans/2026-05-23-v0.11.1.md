# v0.11.1 — audit-first console (Today view + `/v1/admin/today`)

**Status:** done.

## What shipped

First P1 console slice. Inverts the chat-first default of every
competitor (OpenWebUI / LobeChat / LibreChat / AnythingLLM) by making
`Today` the admin-ui landing page — a unified, auto-refreshing timeline
of chat sessions, IM sessions, and scheduled job runs, with the audit
metadata (tenant, run id, attempt, proactive reason) up front.

This is the v0.11.1 line in roadmap §1 + §3 made real: "audit view is the
*default* home page, not chat." Existing console panes (Tenants /
Providers / MCP Servers / Marketplace / Audit Log) keep their behaviour
and demote to the sidebar.

## Pieces

### Backend — `GET /v1/admin/today`

* New module `xiaoguai-api/src/today.rs`:
  - `TodayItem` enum with `#[serde(tag = "kind", rename_all = "snake_case")]`
    — three variants: `Chat`, `Im`, `Scheduled`. Mirrors the TS
    discriminated-union pattern used by `ContentBlock` and `AgentEvent`.
  - `TodayQuery { limit, since, kind }` plus `TodayKind` enum (deserialises
    from `?kind=chat|im|scheduled`).
  - `TodayReader` async trait with one method (`list(query)`); the route
    handler does not re-sort, the reader does.
  - `StaticTodayReader` in-memory impl for route + adapter tests — same
    role `StaticAuditReader` plays for `/v1/admin/audit`.
  - `TodayError::{Backend, InvalidArgument}` mirrors `AuditError`.
* `AppState.today: Option<Arc<dyn TodayReader>>` — `None` → endpoint returns
  503 (same trust model as `audit` / `audit_verifier`).
* `routes::admin::list_today` handler — default `limit=50`, max 500;
  forwards `since` (RFC 3339) + `kind` directly to the reader.
* Route mounted at `/v1/admin/today` behind the existing bearer / Casbin /
  rate-limit stack.
* `xiaoguai-api::lib` re-exports `StaticTodayReader, TodayError, TodayItem,
  TodayKind, TodayQuery, TodayReader`.

### Backend — `PgTodayReader` adapter

* New `xiaoguai-core/src/today_bridge.rs`. Same layering pattern as
  `audit_bridge.rs`: api crate stays sqlx-free; the bridge in the core
  binary owns the SQL and implements the api trait.
* Strategy: three independent SELECTs against the existing schema (no
  migration), then merge + sort + cap in Rust. Worst-case merged set is
  `3 * limit` rows before truncation.
  - **Chat** (`fetch_chat`): `sessions` LEFT JOIN `im_conversations`
    WHERE `c.session_id IS NULL` — chat sessions are sessions that the
    IM gateway hasn't claimed. LATERAL subquery aggregates message
    count, tool count (`role = 'tool'`), and the last text preview
    (`jsonb_path_query_first` on the message `content` JSONB).
  - **IM** (`fetch_im`): `sessions` INNER JOIN `im_conversations` — carries
    `provider` + `conversation_id` straight through. Same LATERAL
    aggregate for count + preview (no tool count — IM rendering doesn't
    use it yet).
  - **Scheduled** (`fetch_scheduled`): `scheduled_job_runs` standalone.
    Proactive `reason` comes from a LEFT-JOIN subselect against
    `audit_log`: `actor = 'scheduler:<job_id>'` AND
    `(details->>'run_id')::BIGINT = r.id` AND
    `details->'trigger'->>'type' = 'proactive'` — fishes the field the
    runner merged into the audit row's `trigger.reason` per v0.10.2. Non-
    proactive runs simply produce NULL.
* `clean_preview` helper strips the surrounding `"` that
  `jsonb_path_query_first(...).text` returns; collapses empty strings to
  `None`.
* Wired into `xiaoguai-core/src/main.rs::run_serve` —
  `today: Some(PgTodayReader::arc(pool.clone()))`. Production now
  exposes the endpoint without further opt-in.

### Frontend — `Today` becomes the default route

* New pane `frontend/admin-ui/src/panes/Today.tsx`:
  - `useEffect` initial fetch + 30 s `setInterval` auto-refresh with a
    pause toggle.
  - Three kind pills (multi-select, can't drop all three to zero).
  - Date-range select: Last 24h (default) / Last 7d / All time. The
    "since" timestamp is computed client-side and forwarded as
    `?since=`.
  - Free-text search filters the loaded set client-side on
    `preview + tenant_id`. Server-side search defers to v0.11.2 per
    roadmap §3.
  - Server-side `?kind=` filter is applied only when exactly one pill is
    active — the endpoint accepts a single kind, by design. Two-of-three
    just filters the merged result client-side.
  - Timeline cards have a coloured left border per kind (indigo / emerald
    / amber) and a subtle hover lift; click opens a detail drawer
    (`role="dialog" aria-modal="true"`) showing the full field set —
    timestamp, tenant, started-at, message/tool counts (chat), provider +
    chat id (IM), attempt + status + error + reason (scheduled).
  - "refreshed Xs/m/h ago" relative time in the header.
* `App.tsx` rewritten: default route now `Navigate to /today`; nav order
  is **Today** (first), then a section header "Manage", then Tenants /
  Providers / MCP Servers / Marketplace / Audit Log (demoted).
* `styles.css` adds `.today-header`, `.today-filters`, `.timeline-card`,
  `.kind-tag-*`, `.drawer*`. Reuses existing `--bg / --fg / --muted /
  --border / --row-hover` tokens — same palette as the rest of admin-ui.

### Shared types

`@xiaoguai/shared` adds:

* `TodayItem` discriminated union mirroring the backend's `serde(tag =
  "kind")` shape exactly (chat / im / scheduled). `kind` is the literal
  discriminator on each variant.
* `TodayKind` literal type, `ListTodayQuery { limit, since, kind }`.
* `XiaoguaiClient.listToday(q?)` — builds the query string and returns
  `TodayItem[]`.
* `PACKAGE_VERSION` bumped to `0.3.0`.

## Deferred

* **Server-side text search.** The endpoint doesn't grep messages /
  audit details. Client-side filter on already-loaded items keeps the
  v0.11.1 surface tight; full-text search lands with v0.11.2's eval pane
  work (which needs an ts-vector index anyway).
* **Multi-kind server-side filter.** `?kind=chat,scheduled` would let the
  console push two pills' worth of filter into SQL. Today the route
  takes a single `kind` and falls back to "all + client filter" for
  multi-select. Easy add when v0.11.2 expands the query knobs.
* **Drill-in full transcript.** The detail drawer shows summary fields;
  rendering the full message history for a chat / IM session, or the
  full audit transcript for a scheduled run, reuses
  `XiaoguaiClient.listMessages` and is a v0.11.2 follow-up.
* **`PgTodayReader` testcontainer e2e.** The static reader + adapter
  unit cover the merge contract; a real-PG run-through pairs naturally
  with v0.11.2 when the eval pane will want it.
* **Live update via SSE / websocket.** 30 s polling is the right
  primitive for "I'm eyeballing audit data, not driving the cursor."
  Push-based updates can layer in once `xiaoguai-api` grows an SSE
  channel for admin events (no current ticket).
* **Per-tenant scoping of the Today endpoint.** Today the reader is
  cross-tenant (the admin path bypasses RLS). Per-tenant scope reuses
  the same query with a `WHERE tenant_id = $tenant` predicate and a
  separate `/v1/admin/today/tenant/:id` route — deferred until a real
  multi-tenant operator asks.
* **Read-only browser verification.** Per the v0.8.x convention, visual
  pass is a human task. Plan + final report explicitly flag this.

## Design decisions worth flagging

* **`TodayReader` trait in api crate, PG SQL in core crate** — same
  layering as `audit_bridge.rs`. Keeps `xiaoguai-api` free of sqlx and
  preserves the rule that the api layer is concrete-storage-agnostic.
* **Three SELECTs + Rust merge over one UNION query.** RLS-scoping of
  `sessions` makes a cross-tenant UNION require either `SET LOCAL` per
  tenant or running outside RLS; three SELECTs against indexed columns
  is both faster (each can use its own `LIMIT` + index) and simpler.
  The Rust-side merge is `O((3*limit) log (3*limit))` — trivial.
* **Chat vs IM = absence-vs-presence in `im_conversations`.** No `origin`
  column added to `sessions` — the v0.7.3 mapping table already encodes
  the distinction. The LEFT JOIN / INNER JOIN pair is the cheapest read.
* **Proactive `reason` joined from `audit_log`, not stored on the
  `scheduled_job_runs` row.** Per v0.10.2, the runner already writes
  `trigger.reason` into the audit row's `details` JSONB — re-storing it
  on the run row would denormalise. The LATERAL subselect adds one
  index hit per run.
* **`TodayItem.ts` is `updated_at` for sessions / `created_at` for
  runs.** Chat/IM activity is "most recently touched"; scheduled
  activity is "most recently fired". This matches what an operator
  scanning the timeline expects to see at the top — both sources sort
  by "most recent event".
* **`limit=50` default, `500` max.** Matches the
  "agents read JSON lists token-by-token" guidance from the CLAUDE.md
  context-engineering rules: list endpoints default to a tight cap, with
  filter knobs (`since`, `kind`) doing the real narrowing.
* **Auto-refresh 30 s, pausable.** Audit data is "I'm eyeballing this",
  not "I'm hunting a live bug" — 30 s is slow enough that the UI doesn't
  flicker but fast enough that recent runs surface in one pause. Pause
  toggle exists because a user reading a card shouldn't fight the cursor
  reflowing under them.
* **Today as default route, sidebar reorder.** The hardest part of this
  tag — every prior admin-ui change defaulted to whatever was newest.
  Inverting the default sends the product positioning signal (audit
  first, chat second). The sidebar still has every prior pane; nothing
  was deleted.
* **Detail drawer over modal-in-page.** Drawer slides over the timeline,
  letting users compare cards visually after closing. A centred modal
  would force a "start over" mental model. Click-outside + Esc close
  both work via the backdrop pattern.
