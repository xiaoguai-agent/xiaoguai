# v1.1.3 — DingTalk + WeCom IM adapters

**Status:** done (2026-05-24).

## Why this slice

v1 shipped Feishu as the only fully-wired IM adapter. The
`xiaoguai-im-dingtalk` and `xiaoguai-im-wecom` crates existed only as
scaffolds (`lib.rs = "//! Skeleton (v1.1 target)."`). For users on
DingTalk or WeCom — a substantial fraction of the China-market install
base — there was no path to drop the agent into their existing IM
without rolling their own webhook adapter.

v1.1.3 closes that gap: both adapters now mirror Feishu's design
(`ImProvider` trait, `TokenCache` single-flight, `ApiSink` reply path,
operator-time env-var wiring), are mounted alongside Feishu by the
core binary, and are covered by the same shape of integration tests
(signature happy + tamper, payload parse, mockito send round-trip,
axum route end-to-end through `MockBackend`).

## Pieces

### Crate `xiaoguai-im-dingtalk`

```
crates/xiaoguai-im-dingtalk/
├── Cargo.toml
└── src/
    ├── lib.rs    — DingTalkProvider + verify() + parse_event() + impl ImProvider
    └── api.rs    — DingTalkClient trait + HttpDingTalkClient + TokenCache
```

Signature (DingTalk Stream / OpenAPI webhook):
```
string_to_sign = timestamp + "\n" + app_secret
sig            = base64(hmac_sha256(app_secret, string_to_sign))
```
Adapter accepts the `(timestamp, sign)` pair via either explicit
headers `X-Dingtalk-Timestamp` + `X-Dingtalk-Sign` (preferred — what
the gateway router passes through), or the `x-dingtalk-query` header
holding a URL-encoded query string (fallback for transports that
don't expose individual query keys).

Payload parser handles both single-chat (`conversationType="1"`) and
group (`conversationType="2"`). To preserve the conversation kind
through the gateway's provider-neutral plumbing, the parser encodes
the kind into the `IncomingMessage::conversation_id` as a prefix:
`dingtalk:single:<cid>|<user_id>` or `dingtalk:group:<cid>|<user_id>`.
The reply path decodes that and picks between the two OpenAPI
endpoints (`oToMessages/batchSend` for single, `groupMessages/send`
for group). Non-text msgtypes (image, audio, etc.) are rejected as
`Malformed` — out of scope for v1.1.3.

OpenAPI reply via `https://api.dingtalk.com`:
- `POST /v1.0/oauth2/accessToken` body `{"appKey","appSecret"}`
- `POST /v1.0/robot/oToMessages/batchSend` with
  `x-acs-dingtalk-access-token: <token>` (single)
- `POST /v1.0/robot/groupMessages/send` with same header (group)

Token cache: same shape as `xiaoguai_im_feishu::TokenCache` — one
token per process, refresh 60 s before expiry, `tokio::sync::Mutex`
makes the refresh single-flight.

### Crate `xiaoguai-im-wecom`

```
crates/xiaoguai-im-wecom/
├── Cargo.toml
└── src/
    ├── lib.rs    — WeComProvider + verify() + parse_event() + echo_challenge() + impl ImProvider
    └── api.rs    — WeComClient trait + HttpWeComClient + TokenCache
```

Signature:
```
sig = sha1(sort([token, timestamp, nonce, body_text]).join(""))
```
Lower-case hex digest. The four query fields arrive via explicit
headers `X-WeCom-Timestamp` + `X-WeCom-Nonce` + `X-WeCom-Msg-Signature`
(preferred) or the `x-wecom-query` fallback.

Payload parser uses `quick-xml`:
```xml
<xml>
  <ToUserName>corp_id</ToUserName>
  <FromUserName>user_id</FromUserName>
  <CreateTime>1716355200</CreateTime>
  <MsgType>text</MsgType>
  <Content>hello</Content>
  <MsgId>123456</MsgId>
  <AgentID>1000002</AgentID>
</xml>
```

The conversation-id encoding is `wecom:<corp_id>|<user_id>` so the
reply path can route back through `message/send` with the correct
`touser`. WeCom inbound has no notion of group chat for this endpoint
— each text is a 1:1 user → app conversation.

OpenAPI reply via `https://qyapi.weixin.qq.com`:
- `GET /cgi-bin/gettoken?corpid={cid}&corpsecret={secret}`
- `POST /cgi-bin/message/send?access_token={token}` body
  `{"touser","msgtype":"text","agentid","text":{"content"},"safe":0}`

`echo_challenge(webhook, token, echostr)` covers the URL-verification
ping (plain-text mode): signature is computed with `echostr` as the
`body_text`, success returns `echostr` verbatim. Caller (the gateway
router) is responsible for binding it to a GET handler — v1.1.3 ships
the helper but does not yet auto-mount a GET route (deferred).

### Gateway router additions

`crates/xiaoguai-im-gateway/src/router.rs` grew:
- `mount_dingtalk(state, provider)` + `mount_dingtalk_with_history(state, provider, history)`
- `mount_wecom(state, provider)` + `mount_wecom_with_history(state, provider, history)`
- a shared `mount_with_history(path, ...)` helper so the three IM
  mounts pick the same `handle_webhook` + `spawn_agent_reply` plumbing
- `GatewayState::feishu` renamed to `GatewayState::provider` (the
  field hosts whichever provider this mount was built with — the name
  was a v0.7-era artifact when Feishu was the only adapter)

Tracing log keys updated to include the provider name so multi-channel
deployments can grep by `provider=dingtalk` etc.

### Operator wiring

`crates/xiaoguai-core/src/main.rs::run_serve` now:
- factors the IM history-store construction into `build_im_history`
  (PG-backed default + in-process opt-in — same logic as v0.7.4, just
  shared across all three mounts so per-conversation isolation is
  guaranteed regardless of which provider delivered the message)
- adds `build_dingtalk_gateway` + `build_wecom_gateway` returning
  `Option<Router>` (None when the env vars for the provider are unset)
- merges all three via a new `merge_routers(Vec<Option<Router>>)`
  helper so any subset can be enabled

Env vars consumed:

| Var | Required for | Notes |
|---|---|---|
| `XIAOGUAI_IM_FEISHU__VERIFICATION_TOKEN` | Feishu inbound | mount the route |
| `XIAOGUAI_IM_FEISHU__APP_ID` / `__APP_SECRET` | Feishu outbound | falls back to stub when missing |
| `XIAOGUAI_IM_DINGTALK__APP_SECRET` | DingTalk inbound | mount the route |
| `XIAOGUAI_IM_DINGTALK__APP_KEY` | DingTalk outbound | required for token fetch |
| `XIAOGUAI_IM_DINGTALK__API_SECRET` | DingTalk outbound | defaults to webhook secret |
| `XIAOGUAI_IM_DINGTALK__ROBOT_CODE` | DingTalk outbound | required by both reply endpoints |
| `XIAOGUAI_IM_WECOM__TOKEN` | WeCom inbound | mount the route |
| `XIAOGUAI_IM_WECOM__CORP_ID` | WeCom outbound | corp id |
| `XIAOGUAI_IM_WECOM__SECRET` | WeCom outbound | corp secret |
| `XIAOGUAI_IM_WECOM__AGENT_ID` | WeCom outbound | numeric agent id |
| `XIAOGUAI_IM_WECOM__AES_KEY` | (reserved) | logs a warning that v1.1.3 plain-text only |
| `XIAOGUAI_IM__USE_IN_PROCESS_HISTORY` | (override) | force the in-memory store |

## What this slice deliberately defers

- **Encrypted-payload Feishu variant migration.** v0.7+ Feishu uses
  the plain SHA-256 signature shape; the older `EncodingAESKey` mode
  is not in our adapter and not in scope here either.
- **WeCom AES-encrypted payloads.** When `EncodingAESKey` is enabled
  on the WeCom side, `<Encrypt>` envelopes arrive AES-CBC-256
  ciphertext. We detect that shape and reject as `Malformed` with a
  directive to disable encryption in the WeCom admin console. Full
  decrypt + re-encrypt for replies needs an `aes` + `cbc` crate
  dependency and a `EncodingAESKey` handling layer — defer to v1.1.4
  if a real user surfaces a hard requirement.
- **Interactive cards.** Both DingTalk and WeCom support rich card
  message types (`actionCard`, `markdown`, `news`). v1.1.3 ships text
  only; cards land alongside the Feishu interactive-card work the
  v0.7.2 plan deferred.
- **Long-message handoff.** WeCom caps text payloads at 2048 bytes;
  DingTalk at ~25 000 chars. Neither adapter currently chunks. When
  the agent produces an overlong reply we'll send the head and log a
  warning until v1.1.4 adds a "post a link to the full transcript"
  fallback.
- **WeCom group `@mention` parsing.** WeCom delivers group messages
  via a different inbound webhook path (the "external contact" /
  "customer interaction" suite) which is its own API surface. v1.1.3
  covers the 1:1 user → app path only; group chat support is
  v1.1.4+.
- **DingTalk Stream client.** DingTalk also offers a long-poll Stream
  API for organisations that can't expose an inbound HTTPS endpoint.
  v1.1.3 covers webhook only; Stream is its own crate-shaped
  add-on.
- **GET route binding for WeCom `echo_challenge`.** The helper is
  exposed but the gateway only registers `POST` today. URL
  verification can be exercised manually by a caller wiring the GET
  route; a built-in mount lands when interactive cards do (same code
  path needs split for GET vs POST anyway).
