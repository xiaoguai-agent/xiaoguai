## Why this slice

Roadmap Â§3 v1.1 ("usage endpoint") asks for a single screen the
operator can point a finance / governance reviewer at: "show me
what each tenant is spending across providers and models, for any
date range I pick." The `token_usage` ledger (migration 0004) has
been writing rows since v0.6.4 and the LLM router's background
flusher amortises them â€” but there is no read surface yet, so all
of that data is invisible above the SQL line.

This slice adds the read surface (`GET /v1/usage`), a dedicated
admin-ui pane that drives it, and a one-card summary on the
existing Today pane so the operator does not need to switch panes
to know "did anything explode in the last 24 hours?"

## Pieces

### Backend

* `xiaoguai-api/src/usage.rs` â€” new module mirroring the
  `today.rs` shape:
  * `UsageReader` trait (`async fn aggregate(query) -> UsageReport`).
  * `UsageQuery { tenant_id, since, until, group_by }`.
  * `UsageGroupBy = Day | Provider | Model`.
  * `UsageReport { rows, total_input_tokens, total_output_tokens, cost_cents }`.

* `xiaoguai-api/src/routes/usage.rs` â€” `GET /v1/usage` handler.
  * Admin-bearer-gated (same `route_layer` stack as the rest of
    `/v1/*`).

    400 when an unknown `group_by` enum value is supplied (axum's

* `xiaoguai-core/src/usage_bridge.rs` â€” `PgUsageReader`. Single
  `GROUP BY` over `token_usage`; `tenant_id`/`since`/`until` use
  the `($1::TYPE IS NULL OR col = $1)` idiom so the SQL string
  stays constant per `group_by`. Bucket SQL fragment:
  * `Day` â†’ `to_char(ts AT TIME ZONE 'UTC', 'YYYY-MM-DD')`
  * `Provider` â†’ `provider_id`
  * `Model` â†’ `model`
  * Tenant-scoped reads run inside `begin_tenant_tx` so the RLS
    policy `tenant_isolation_token_usage` filters correctly;
    cross-tenant reads use the superuser pool (same pattern as
    `PgTodayReader`).
* `AppState` gains `usage_reader: Option<Arc<dyn UsageReader>>`.
  Production wires `Some(PgUsageReader::arc(pool.clone()))`.

## Deferrals

* **Per-provider cost rates.** `llm_providers` (migration 0003)
  has no `cost_per_1k_input_tokens` / `cost_per_1k_output_tokens`
  columns today. Until that schema lands, `cost_cents` is `None`
  on every row and on the report total. The UI surfaces "cost: â€”
  (rates not yet configured)" so the operator sees the gap. Adding
  the columns is a single migration + a JOIN inside the
  `PgUsageReader` â€” left for v1.1.1.2 (after the schema work).
* **Recharts bar chart.** v1.1.1.1 will layer a stacked bar chart
  on the rows table once Recharts is in `package.json`. The
  current pane already does enough: see the daily token counts and
  spot anomalies.
* **Per-tenant usage budget alerts.** A budget engine would fire
  push-sink notifications when daily spend crosses a threshold;
  that is v1.1.1.3 work alongside the cost-rates schema.
* **Pagination.** Day buckets are inherently small (one row per
  day per filter); provider/model rarely exceeds 10â€“20 rows in
  practice. We will revisit if a tenant ever produces > 200 rows.
