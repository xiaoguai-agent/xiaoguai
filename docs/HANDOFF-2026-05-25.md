## 4. Key technical notes for resumer

### 4a. Cargo target convoy (root cause of 13 stalls)

33 worktrees sharing the parent `target/` (or each cargo grabbing the workspace lock individually) serialized 33 cargo invocations behind one another → 600s watchdog timeouts. **Fix for future parallel waves**:
- Either `export CARGO_TARGET_DIR=/tmp/xg-wt-<name>/target` per worktree (independent caches, more disk)
- Or batch parallelism by ≤5 cargo-using agents at once (queue the rest)
- Doc-only / config-only / new-crate-only agents were unaffected; rust-modifying agents got hit

### 4b. AWS SDK Rust version

`feat/audit-s3` workspace pinned `aws-sdk-s3 ~1.122`. Transitive `aws-smithy-*` 1.91 declares `rust-version = "1.91.1"` even though it builds on 1.88. Options on integration:
- `cargo build --ignore-rust-version` everywhere
- Or downgrade `aws-sdk-s3` to 1.95 (last with 1.85 rust-version)
- Or bump workspace `rust-version` to 1.91 (Cargo.toml `[workspace.package]`)

Recommend last option: bump to 1.91. CI runners support it.

### 4c. AppState field convoy

3 of the 4 HIGH-risk branches all add new fields to `xiaoguai-api::state::AppState`:
- `feat/hotl-policy` → `hotl_enforcer: Option<Arc<dyn HotlEnforcer>>`
- `feat/outcome-telemetry` → `outcome_recorder: Option<Arc<dyn OutcomeRecorder>>`
- `feat/skill-marketplace-ui` → `skill_catalog: Option<Arc<dyn SkillCatalog>>`
- `feat/rate-limit` → `rate_limit_state: Option<Arc<RateLimitState>>` (already shipped, 15 test files updated)

When merging, each test file constructs `AppState { ... }` literally; merge-driver `ort` should handle additive fields but if conflicts arise, "keep both" each field. Same pattern as v0.12.x.1 / v1.1.1 / v1.1.2 from earlier today.

### 4d. xiaoguai-rag/lib.rs additive register

3 RAG agents (C11 backends, C12 loaders, C13 reranker) all add `pub mod X` lines. Order matters not, all additive.
