  ships in `xiaoguai-core/scheduler_bridge.rs` and is unit-tested but
  the operator binary still hardcodes `RuntimeJobExecutor`. v0.12.2.1.
- **Admin-ui Scheduler pane.** Backend NL-job + upsert + webhook
  routes ship; admin-ui has no UI yet (the v0.12.1 sub-agent shipped
  backend only). v0.12.1.1.
- **`RagClient` binary file re-index path.** `R2RClient::reindex_path`
  handles text only; binary file → R2R multipart is deferred.
- **`notify-debouncer-full`.** File-watch source emits one event per
  `notify` event; a `git checkout` over a 1k-file repo will saturate
  the runner until we layer the debouncer.
- **Write-capable Obsidian connector.** v0.12.2 catalog entry points
  at the community read/write server (`StevenStavrakis/obsidian-mcp`,
  710 stars MIT). A first-party Rust connector with finer-grained
  permissions is v1.1+ work.

## 4. Working conventions to keep

- Commit per logical task with `feat/fix/test/docs(vX.Y.Z): ...`.
- Every milestone closes with a doc update + `cargo test --workspace`
  number recorded + `clippy -D warnings` + `fmt --check` clean.
- Each milestone tagged + pushed: `git tag vX.Y && git push origin
  main && git push origin vX.Y`.
- Always provide a deferral list in the milestone plan doc — leave
  the tree shippable at every tag.
- Tests live next to the code they cover (`tests/` for crate
  integration; `#[cfg(test)] mod tests` for unit). Use scripted
  `MockBackend` + in-memory repos in `tests/common/`.
- Frontend changes: keep `@xiaoguai/shared` as the single source of
  truth for wire types — both chat-ui and admin-ui import from it.
- v0.12.0 introduced `xiaoguai-runtime` as the shared agent-loop
  substrate. Any future caller (a new IM provider, a CLI agent
  command, etc.) should call `run_to_completion` / `run_streamed` /
  `run_to_sink` rather than constructing a `ReactAgent` directly.
- v0.12.x added cross-crate trait pairs to avoid cycles
  (`WebhookPusher`, `NlJobCompiler`, `ScheduledJobUpserter`,
  `ScheduledSessionWriter`). The pattern: trait in `xiaoguai-api` or
  `xiaoguai-scheduler`, production impl in `xiaoguai-core`
  (`scheduler_bridge.rs` is the central hub). Follow the same shape
  for any new bridge.
