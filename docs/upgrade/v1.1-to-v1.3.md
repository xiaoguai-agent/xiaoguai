# Upgrade guide — v1.1.x → v1.3.x (wave-3 features)

This guide covers every operator-facing change introduced between the v1.1 baseline
and the v1.3 wave-3 feature set. Read it top-to-bottom before touching any migration
or config file.

**Upgrade path:** v1.1.x → v1.2.x → v1.3.x (no skip supported).

---

## 1. Compatibility matrix

| From | To | Breaking API changes | Schema changes | Required action |
|------|----|----------------------|----------------|-----------------|
| v1.1.x | v1.2.x | None | Migrations 0011–0015 (additive) | Apply migrations; no config required |
| v1.2.x | v1.3.x | None | None additional | Enable PG bridges for wave-3 endpoints (see §4) |

**v1.1.x → v1.2.x is fully backward-compatible.** All five new migrations add
tables or columns; no existing table is altered destructively. Existing API clients
continue to work unchanged.

**v1.2.x → v1.3.x is additive only.** No existing endpoints are removed or
changed. v1.3.0 is the target release that wires the Postgres bridge crates
(`PgHotlPolicyStore`, `PgOutcomeRecorder`, `PgSkillPackRepository`) into the binary.
Until those bridges land, the new `/v1/hotl/*`, `/v1/outcomes/*`, and `/v1/skills/*`
endpoints return `503 Service Unavailable` — this is expected and documented.

> **Planning note for v1.3.0 operators:** the moment the PG bridges are wired,
> all three endpoint groups become live simultaneously. Schedule an activation-testing
> window (see §8, action item 5) before your first production v1.3.0 deployment.

## 2. Schema migrations

Run migrations in the order listed. Each migration is independently
backward-compatible with the running binary from the previous minor version, so you
can apply them while the v1.1/v1.2 binary is running (zero-downtime path).

**Before any migration:** follow the backup playbook at
`docs/runbooks/backup-wave3.md`.

### 0011 — HOTL policies (v1.2.3)

**File:** `crates/xiaoguai-storage/migrations/0011_hotl_policies.sql`

Creates two new tables: `hotl_policies` and `hotl_usage_log`.

| Detail | Value |
|--------|-------|
| Tables created | `hotl_policies`, `hotl_usage_log` |
| Indexes | `hotl_policies_tenant_scope`, `hotl_usage_tenant_scope_time` |
| Existing tables touched | None |
| Forward cost (100 M-row tenant DB) | ~2 s (DDL only; no data rewrite) |
| Back-out | `DROP TABLE hotl_usage_log; DROP TABLE hotl_policies;` — instant, no data loss |

### 0012 — Outcome telemetry (v1.2.4)

**File:** `crates/xiaoguai-storage/migrations/0012_outcomes.sql`

Creates `agent_outcomes` — the append-only ledger for business-outcome attributions.

| Detail | Value |
|--------|-------|
| Tables created | `agent_outcomes` |
| Indexes | `outcomes_tenant_kind_time` |
| Existing tables touched | None |
| Forward cost (100 M-row tenant DB) | ~2 s (DDL only) |
| Back-out | `DROP TABLE agent_outcomes;` — instant |

### 0013 — Audit export watermarks (v1.2.19)

**File:** `crates/xiaoguai-storage/migrations/0013_audit_export_state.sql`

Creates `audit_export_state` — one row per named S3/MinIO audit sink, tracking the
highest exported `audit_log.id`.

| Detail | Value |
|--------|-------|
| Tables created | `audit_export_state` |
| Existing tables touched | None |
| Forward cost (100 M-row tenant DB) | ~1 s (DDL only) |
| Back-out | `DROP TABLE audit_export_state;` — instant; already-exported audit rows are unaffected |

> **Data risk on back-out:** if you roll back after the S3 exporter has begun
> writing watermarks, re-applying the migration later will restart exports from id=0
> (safe but may re-upload rows already in S3). The exporter is idempotent on the S3
> side.

### 0014 — Rate-limit class on tenants (v1.2.20)

**File:** `crates/xiaoguai-storage/migrations/0014_tenant_rate_limit.sql`

Adds `rate_limit_class TEXT NOT NULL DEFAULT 'standard'` to the `tenants` table.

| Detail | Value |
|--------|-------|
| Tables created | None |
| Columns added | `tenants.rate_limit_class` (default `'standard'`) |
| Index | `ix_tenants_rate_limit_class` |
| Forward cost (100 M-row tenant DB) | ~5–15 s (ALTER TABLE on large `tenants`; Postgres rewrites the heap) |
| Back-out | `ALTER TABLE tenants DROP COLUMN rate_limit_class;` — instant drop, no rewrite |

> **Runtime note:** this is the only migration with a non-trivial forward cost if
> the `tenants` table has millions of rows. Consider running it during a low-traffic
> window. All existing tenants receive `'standard'` by default; no behaviour change
> until the rate-limit middleware is enabled (see §7).

### 0015 — Skill packs (v1.2.28)

**File:** `crates/xiaoguai-storage/migrations/0015_skill_packs.sql`

Creates `installed_skill_packs` — per-tenant pack installation records.

| Detail | Value |
|--------|-------|
| Tables created | `installed_skill_packs` |
| Indexes | `installed_skill_packs_tenant_idx` |
| Unique constraint | `(tenant_id, pack_slug)` |
| Existing tables touched | None |
| Forward cost (100 M-row tenant DB) | ~2 s (DDL only) |
| Back-out | `DROP TABLE installed_skill_packs;` — instant; does not affect running pack behavior |

### Migration order summary

```
0011 → 0012 → 0013 → 0014 → 0015
```

Each migration can be applied independently but must be applied in ascending numeric
order to satisfy sqlx's migration tracker. Verify applied state after each:

```bash
psql "$DATABASE_URL" -c \
  "SELECT version, installed_on FROM _sqlx_migrations WHERE version IN (11,12,13,14,15);"
```

Or via the smoke command:

```bash
xiaoguai-core smoke   # exits non-zero if any migration is missing
```

## 3. Config file deltas

### New environment variables (v1.2.x wave-3)

All wave-3 features are **off by default**. No new env var is required to upgrade;
the binary boots normally without any of them set.

| Env var | Feature | Default | Required? |
|---------|---------|---------|-----------|
| `XIAOGUAI_RATE_LIMIT__ENABLED` | Per-tenant rate limiting (v1.2.20) | `false` | No |
| `XIAOGUAI_HOTL__ENABLED` | Human-on-the-Loop enforcement (v1.2.3) | `false` | No |
| `XIAOGUAI_OUTCOMES__BACKEND` | Outcome telemetry backend: `in-mem` or `postgres` | unset (disabled) | No |
| `XIAOGUAI_AUDIT__S3_BUCKET` | S3/MinIO bucket for audit export sink (v1.2.19) | unset (disabled) | No |
| `XIAOGUAI_AUDIT__S3_PREFIX` | S3 key prefix for audit export | `xiaoguai/audit/` | No |
| `AWS_ACCESS_KEY_ID` | Bedrock / S3 audit sink credentials | unset | Only if using Bedrock or S3 audit |
| `AWS_SECRET_ACCESS_KEY` | Bedrock / S3 audit sink credentials | unset | Only if using Bedrock or S3 audit |
| `AWS_SESSION_TOKEN` | Bedrock temporary credentials (STS) | unset | No |
| `FILE_WATCH_DEBOUNCE_MS` | File-watch event coalescing window (v1.1.10) | `250` | No |
| `XIAOGUAI_IM__DISCORD_TOKEN` | Discord adapter (v1.2.8) | unset | Only if using Discord |
| `XIAOGUAI_IM__TELEGRAM_BOT_TOKEN` | Telegram adapter (v1.2.9) | unset | Only if using Telegram |
| `XIAOGUAI_IM__MATTERMOST_TOKEN` | Mattermost adapter (v1.2.x) | unset | Only if using Mattermost |
| `XIAOGUAI_IM__SLACK_SIGNING_SECRET` | Slack adapter (v1.2.x) | unset | Only if using Slack |

**Cloud LLM v2 providers (Bedrock, Azure OpenAI, Mistral, Groq)** are registered via
the admin API (`POST /v1/admin/providers`), not env vars — the SDK installation is
detected at runtime. The API key for each provider is stored as the `api_key_env`
field on the provider row, which names the env var to read at call time. None of
these providers are auto-enabled even if the SDK is installed.

### Helm values additions (v1.2.x)

No new _required_ Helm values. Optional additions for wave-3 features:

```yaml
# values.yaml — wave-3 optional knobs (all default off)
config:
  XIAOGUAI_RATE_LIMIT__ENABLED: "false"
  XIAOGUAI_HOTL__ENABLED: "false"
  # XIAOGUAI_OUTCOMES__BACKEND: "postgres"  # uncomment to enable
  # XIAOGUAI_AUDIT__S3_BUCKET: "my-audit-bucket"

# Optional: LLM provider keys for cloud providers
# secrets:
#   llm: xiaoguai-llm-keys  # Secret containing ANTHROPIC_API_KEY, AWS_ACCESS_KEY_ID, etc.
```

### Kustomize overlays

No structural changes to the base overlay. Wave-3 knobs are injected as additional
env entries in `deploy/kustomize/overlays/<env>/configmap.yaml`:

```yaml
# configmap.yaml patch (optional wave-3 activation)
apiVersion: v1
kind: ConfigMap
metadata:
  name: xiaoguai-config
data:
  XIAOGUAI_RATE_LIMIT__ENABLED: "false"
  XIAOGUAI_HOTL__ENABLED: "false"
```

### Terraform variables

No new required Terraform variables. The AWS Fargate module
(`deploy/terraform/aws/`) passes config as ECS task environment variables; add
wave-3 vars to `variables.tf` only when enabling a feature. See
`deploy/terraform/aws/README.md` for the variable pattern.

## 4. API additions

All endpoints below are registered in v1.2.x but return `503 Service Unavailable`
until the PG bridge crates are wired in v1.3.0. This is by design: the routes exist,
their request/response schemas are stable, but the backing store is not connected.

### `/v1/hotl/*` — Human-on-the-Loop boundary policies (v1.2.3)

| Method | Path | Description | Status in v1.2.x |
|--------|------|-------------|-----------------|
| `GET` | `/v1/hotl/policies?tenant_id=<uuid>[&scope=<str>]` | List policies | 503 |
| `POST` | `/v1/hotl/policies` | Create a policy | 503 |
| `DELETE` | `/v1/hotl/policies/:id` | Remove a policy | 503 |

Policies declare per-tenant action budgets (count or USD cost) over a rolling time
window. The enforcer checks budgets at LLM call sites; email-send and webhook-invoke
enforcement sites are planned for a follow-up.

**503 until:** `PgHotlPolicyStore` is wired into `AppState.hotl_policy_store` in v1.3.0.

### `/v1/outcomes/*` — Outcome telemetry (v1.2.4)

| Method | Path | Description | Status in v1.2.x |
|--------|------|-------------|-----------------|
| `POST` | `/v1/outcomes` | Record a business outcome | 503 |
| `GET` | `/v1/outcomes/summary?tenant_id=&range=` | Aggregated ROI summary | 503 |
| `GET` | `/v1/outcomes/timeseries?tenant_id=&range=[&kind=]` | Daily breakdown | 503 |

Agents call `POST /v1/outcomes` after completing tasks with measurable business
value (`kind`: `revenue_usd`, `cost_saved_usd`, `hours_saved`, `deals_closed`,
`tickets_resolved`, `custom`). The admin-ui Outcomes pane reads the two GET
endpoints.

`range` accepts `"24h"`, `"7d"`, or `"30d"` (default `"30d"`).

**503 until:** `PgOutcomeRecorder` / `PgOutcomesReader` wired into
`AppState.outcome_writer` / `AppState.outcomes_reader` in v1.3.0.

### `/v1/skills/*` — Skill pack marketplace (v1.2.28)

| Method | Path | Description | Status in v1.2.x |
|--------|------|-------------|-----------------|
| `GET` | `/v1/skills/catalog` | List available packs (no auth) | **Live in v1.2.x** |
| `GET` | `/v1/skills/installed?tenant=<id>` | List installed packs for tenant | 503 |
| `POST` | `/v1/skills/install` | Install a pack | 503 |
| `DELETE` | `/v1/skills/install/:id` | Uninstall a pack | 503 |

Note: `GET /v1/skills/catalog` is live immediately — it reads a static JSON blob
baked into the binary (`catalog/skill_packs.json`) and requires no store. The
installed-pack endpoints return 503 until `PgSkillPackRepository` is wired in v1.3.0.

Installing a pack records a row in `installed_skill_packs` but does **not** hot-reload
the pack runtime. Full pack activation is planned post-v1.3.

## 5. Deprecated APIs

There are no deprecated APIs in v1.2.x or v1.3.x.

**Deprecation convention:** when an endpoint is deprecated, the server will add an
`X-Deprecated: <reason>; use <replacement>` response header for one full minor
version before removal. Operators monitoring for this header can automate
deprecation-window alerts.

---

## 6. CLI changes

No wave-3 subcommands have been added to the `xiaoguai` CLI in v1.2.x or v1.3.x.
The following CLI subcommands are planned for v1.4 (tracked in task #68):

- `xiaoguai hotl list / create / delete`
- `xiaoguai outcomes summary`
- `xiaoguai skills install / list / uninstall`

Do not expect CLI coverage of wave-3 features before v1.4. In the interim, use the
REST API directly or the admin-ui.

## 7. Behavioral changes

### Outcome telemetry — off by default

`POST /v1/outcomes` and the read endpoints return `503` until both:

1. The PG bridge is wired (v1.3.0), **and**
2. The operator sets `XIAOGUAI_OUTCOMES__BACKEND=in-mem` (development) or
   `XIAOGUAI_OUTCOMES__BACKEND=postgres` (production).

Setting the env var without the PG bridge has no effect — the route handler checks
`AppState.outcome_writer` first.

### Human-on-the-Loop — off by default

HOTL enforcement is not active unless `AppState.hotl_enforcer` is wired (v1.3.0
PG bridge). The LLM call site checks the enforcer at runtime; if `None`, calls
proceed without budget checking. No operator action required to preserve existing
behavior.

### Rate-limit middleware — off by default

The per-tenant rate-limit middleware (v1.2.20) only activates when
`XIAOGUAI_RATE_LIMIT__ENABLED=true`. Migration 0014 adds the `rate_limit_class`
column with default `'standard'` for all tenants, but the middleware is not in the
request path until explicitly enabled. Tier values: `free` (10 req/s, burst 20),
`standard` (100 req/s, burst 200), `enterprise` (1,000 req/s, burst 2,000).

### Cloud LLM v2 providers — explicit configuration required

Bedrock, Azure OpenAI, Mistral, and Groq are not auto-detected or auto-enabled.
Each must be registered via `POST /v1/admin/providers` with the appropriate
`ProviderKind` and the name of the env var holding the API key. The env var must be
set in the process environment at call time. The build router emits a warning log
(not an error) when the named env var is absent — the binary continues booting.

### Discord / Telegram / Mattermost / Slack IM adapters — explicit env config required

Each new IM adapter (v1.2.8–v1.2.x) is inactive unless its credential env var is
set. Update adapter configs before upgrading if you plan to use these channels.

### Grafana wave-3 dashboard (v1.2.23)

The wave-3 Grafana dashboard JSON ships in v1.2.23. Most panels show "No data"
until the corresponding metric emission lands (outcome counters, HotL check counts,
rate-limit rejection counts). This is a known limitation — the dashboard schema is
forward-compatible and panels will populate as features are activated.

### Skill pack runtime activation

`POST /v1/skills/install` records the installation row but the pack is not activated
at runtime. Pack loader integration is tracked post-v1.3. Installed packs listed in
`GET /v1/skills/installed` do not affect agent behavior until the runtime loader
ships.

## 8. Operator action checklist

Complete these in order:

- [ ] **Backup** — follow `docs/runbooks/backup-wave3.md` before any migration.
- [ ] **Apply migration 0011** (`0011_hotl_policies.sql`) and verify with `smoke`.
- [ ] **Apply migration 0012** (`0012_outcomes.sql`) and verify.
- [ ] **Apply migration 0013** (`0013_audit_export_state.sql`) and verify.
- [ ] **Apply migration 0014** (`0014_tenant_rate_limit.sql`) — schedule during
      low-traffic window if `tenants` table exceeds 1 M rows; verify.
- [ ] **Apply migration 0015** (`0015_skill_packs.sql`) and verify.
- [ ] **Update Helm values / Kustomize / Terraform** — no required changes; add
      optional wave-3 env vars only if enabling a feature (see §3).
- [ ] **If enabling HotL:** design your per-tenant policies (scope, window, budget)
      before wiring the bridge. Reference the HotL policy design chapter in
      `docs/book/src/operator/day2.md`.
- [ ] **If enabling outcomes:** define a retention strategy for `agent_outcomes`
      before data volume grows. Reference compliance gaps documented in
      `docs/compliance/gdpr/dpia-template.md`.
- [ ] **If using Discord / Telegram / Mattermost / Slack:** set the appropriate
      credential env vars in your deployment secrets before the first v1.2 pod comes up.
- [ ] **Verify Grafana wave-3 dashboard renders** — expect "No data" on most panels
      until features are activated; confirm dashboard loads without errors.
- [ ] **Schedule first DR drill** using `docs/runbooks/disaster-recovery-wave3.md`
      after the migration window closes.

## 9. Rollback plan

> **Recommendation: forward-fix rather than roll back.** All five migrations are
> additive; the v1.1 binary is unaffected by the presence of the new tables.
> Rolling back is safe from a schema perspective but risks data loss depending on how
> far the new features have progressed.

### Rolling back v1.2.x → v1.1.x

1. Stop the v1.2 binary.
2. Run down-scripts in reverse order (0015 → 0014 → 0013 → 0012 → 0011):

   ```sql
   -- 0015 down
   DROP TABLE installed_skill_packs;
   -- 0014 down
   ALTER TABLE tenants DROP COLUMN rate_limit_class;
   -- 0013 down
   DROP TABLE audit_export_state;
   -- 0012 down
   DROP TABLE agent_outcomes;
   -- 0011 down
   DROP TABLE hotl_usage_log;
   DROP TABLE hotl_policies;
   ```

3. Start the v1.1 binary.

**Data loss risk:**
- `hotl_policies` and `hotl_usage_log`: all policy definitions and usage history lost.
- `agent_outcomes`: all outcome attribution records lost.
- `audit_export_state`: **watermarks lost** — re-applying migration 0013 later restarts
  S3 export from id=0. Already-uploaded audit files in S3 are unaffected; duplicates
  may appear on the next export cycle.
- `installed_skill_packs`: all pack installation records lost. No runtime behavior
  impact (packs are not yet activated by the runtime).
- `tenants.rate_limit_class`: column dropped; all tier customizations lost.

### Rolling back v1.3.x → v1.2.x

No schema changes in v1.3.0. Stop the v1.3 binary, start the v1.2 binary. The PG
bridge wiring is in the binary, not the schema; reverting is a binary swap only.

Use this template when notifying internal stakeholders of an upgrade window:

---

**Subject:** Xiaoguai upgrade to v1.3 — maintenance window [DATE], [TIME] [TZ]

We will be upgrading Xiaoguai from v1.1.x to v1.3.x during the following window:
**[DATE], [START TIME] – [END TIME] [TZ]**

**What is changing:**
- Five new database migrations (additive — no existing data modified).
- New API endpoints for Human-on-the-Loop policies, outcome telemetry, and skill
  packs. These endpoints are inactive by default; no behavior changes until explicitly
  enabled by the operations team.
- New cloud LLM providers (Bedrock, Azure OpenAI, Mistral, Groq) are available but
  must be configured before use.

**What is not changing:**
- All existing API endpoints remain unchanged.
- No new required configuration.
- No user-facing changes during the maintenance window.

**Expected downtime:** [X minutes for migration 0014 if tenants table is large;
otherwise near-zero for a rolling deployment.]

**Rollback plan:** All migrations are reversible with down-scripts. Estimated rollback
time: under 5 minutes.

Questions or concerns: [contact / channel].
